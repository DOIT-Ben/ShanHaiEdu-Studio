## 2. 一句话总口径

山海智教智能体不是聊天机器人，也不是单点 PPT/视频生成器，而是：

```text
一个以教师项目为中心，以 Agent 为调度者，以节点契约为规则，以工具/Provider 为执行器，以 Artifact 为事实，以 Memory 为个性化，以 Evidence 为可信度，以 Quality Gate 为交付保障的教育内容生产操作系统。
```

所有后续设计都必须回到这句话判断是否跑偏。

## 4. 统一词汇

| 词汇 | 统一含义 | 禁止误用 |
|---|---|---|
| Project 教师项目 | 一次可持续推进的备课/材料生产项目 | 不等同于一条聊天消息 |
| Conversation Log | 完整原始对话记录 | 不因压缩删除 |
| ContextPackage | 每轮模型实际读取的上下文包 | 不等于完整历史消息 |
| SessionContextSnapshot | 旧对话压缩摘要，可版本化、可重建 | 不是唯一事实源 |
| Node Contract | 节点工作规则、输入输出、禁止项、质量门禁 | 不是普通 prompt，也不是用户偏好 |
| Capability | 产品级能力，例如生成教案、生成 PPT 设计稿 | 不等于底层 provider |
| Provider Adapter | 外部服务或本地工具适配器 | 不决定业务流程 |
| Artifact | 产物事实对象，包含草稿、设计稿、真实文件、最终包 | 不只是聊天附件 |
| Evidence | 教材、课标、页码、引用、来源和置信度 | 不允许模型空口伪造 |
| Memory | 个性化和连续性信息 | 不覆盖项目事实和节点契约 |
| Quality Gate | 判断能否继续、下载、交付、写记忆、调用 provider 的门禁 | 不由模型自评替代 |
| HumanGate | 真实执行或高风险动作的人工确认 | 不靠自然语言短句模糊确认 |
| PlanGuard | 模型计划执行前的安全与可行性检查 | 不允许绕过直接跑工具 |
| Tool | 可被模型选择、由后端执行的稳定工具契约 | 不等同于 prompt 文案，也不等同于 provider |
| Tool Registry | 工具定义注册表，声明工具 ID、说明、schema、adapter、门禁、产物和失败策略 | 不在主链路里散落 if/else 注册工具 |
| ToolRouter | 把模型 tool_call / 内部 toolPlan 路由到 Internal、Provider 或 MCP adapter 的执行层 | 不让模型直接访问文件、数据库、provider 或 MCP server |
| GptProtocolAdapter | GPT/LLM 模型协议适配层，使用 OpenAI SDK / Responses API | 不是工具执行层，也不是 MCP client |
| MCP Client Adapter | MCP 工具协议适配层，用于调用 MCP server 暴露的外部工具 | 不是 Agent OS 状态机，也不是 OpenAI SDK 的替代品 |

## 5. 统一架构思想

### 5.1 业务项目是中心，不是消息流

后续功能必须围绕教师项目、节点状态、产物和交付包组织。聊天只是交互方式，不是系统事实源。

### 5.2 Agent 是调度者，不是事实源

Agent 可以理解、计划、追问、建议和总结，但不能单凭自然语言宣称：

```text
文件已生成
任务已完成
证据已验证
交付已通过
长期记忆已写入
```

这些状态必须由数据库、Artifact、Evidence 和 Quality Gate 决定。

### 5.3 Node Contract 是规则源，不是 prompt 堆

节点要求、输入输出、禁止项、质量标准、失败策略，必须逐步从硬编码 prompt 中抽出，进入可版本化的 Node Contract。

### 5.4 ContextPackage 是模型输入边界

模型每轮调用不应直接吞完整长对话，而应读取：

```text
Project State
+ Workflow State
+ active SessionContextSnapshot
+ scoped Project Memory
+ approved Teacher Memory
+ relevant Artifact summaries
+ relevant Evidence
+ recent messages
+ current user input
+ Runtime Guardrails
```

### 5.5 原始对话必须完整保留

压缩只替代模型输入中的旧历史，不删除用户历史。用户回到项目时必须能看见完整对话和产物连续性。

### 5.6 Artifact 是事实对象

真实 PPTX、图片、视频、最终包必须有真实文件、metadata 和校验结果。草稿、设计稿、真实文件必须区分。

### 5.7 Evidence 是可信边界

教育内容必须区分：

```text
教材依据
课标依据
课堂设计建议
模型推断
不确定项
```

无证据不能冒充教材结论。

### 5.8 Quality Gate 是交付保障

模型自称完成不等于完成。只有通过合同校验、真实文件校验、证据校验、隐私安全和最终交付门禁，才能进入下载或交付。

### 5.9 Agent Harness 是受控循环，不是裸 ReAct

山海智教可以吸收 ReAct 的“观察—思考—行动”思想，但不得把它实现成模型无限自由循环。统一口径是：

```text
不确定任务，两阶段；确定流程，工作流。
模型负责判断；门禁负责安全；工具负责执行；数据库负责记忆；上下文负责呈现状态。
```

两阶段适用于高不确定性或高副作用动作：例如选择下一步能力、解释失败原因、决定是否需要教师确认、判断是否调用真实 PPT/图片/视频 provider。第一阶段只做理解、诊断、计划候选；第二阶段在 `PlanGuard`、`HumanGate`、`CapabilityAvailability` 和 `Quality Gate` 通过后才允许执行。

确定流程不交给模型反复“想”。当节点契约、上游产物、教师确认和 provider 可用性已经明确时，应由 workflow / runner / adapter 按契约推进，而不是每一步都重新询问模型。

### 5.10 Harness 不写业务分支，业务规则进入 Contract / Gate / Tool

工业级 Agent 的循环层只负责调度、预算、上下文编译、工具调用协议和 observation 回写，不得在 harness 内部硬编码“如果是 PPT 就怎样、如果是视频就怎样”的业务分支。

业务规则必须沉到以下可审查边界：

```text
Node Contract
Capability Registry
CapabilityAvailability
PlanGuard
HumanGate
Quality Gate
Artifact Validator
Workflow State Machine
Provider Adapter
```

这样既避免“循环层长成屎山”，也避免把业务规则全部交给模型脑补。

### 5.11 不设玩具式硬步骤上限，但必须有资源预算和熔断

山海智教不采用 `max_turns=10` 这类会截断真实任务的玩具式上限。复杂备课、PPT、图片、视频和最终包链路可以跨多轮、多节点、多次确认。

但系统必须设置资源预算和安全熔断：

```text
最大耗时
最大成本或调用次数
单节点最大重试次数
相同动作重复上限
连续失败上限
上下文预算
并发锁
高副作用动作 HumanGate
```

超过预算时，系统应进入可恢复失败或请求教师确认，而不是继续自动循环。

### 5.12 上下文是模型可见状态，不是系统真实状态源

“上下文是唯一状态载体”只能理解为：模型每轮只能依据 `ContextPackage`、`AgentWorldState`、`CapabilityAvailability` 和受控消息输入进行判断。

系统真实状态源必须仍然是：

```text
Database
Conversation Log
Artifact
Generation Job
ToolObservation
Evidence
Quality Gate Result
Audit / ContextBuildLog
```

上下文会被裁剪、压缩、摘要和预算约束影响，因此不得把上下文本身当作唯一真实状态源。正确关系是：

```text
真实状态源 -> Context Compiler -> ContextPackage / AgentWorldState -> 模型判断 -> Tool / Workflow 执行 -> Observation / Artifact / Job 回写真实状态源
```

任何摘要、模型回答或上下文片段都不能单独把未完成产物写成完成、把草稿写成真实文件、把失败 provider 写成可用。

### 5.13 OpenAI SDK 管模型，MCP 管工具，中间必须有 ToolRouter

统一口径：

```text
OpenAI SDK 管模型协议；MCP 管工具协议；ToolRouter / Agent Runtime 负责把模型的 tool_call 变成真实工具执行。
```

更准确的链路是：

```text
业务后端
  -> GptProtocolAdapter
    -> OpenAI SDK / Responses API
      -> GPT 模型

GPT 输出 tool_call / function_call 意图
  -> ToolRouter
    -> Internal Capability Adapter
    -> Provider Adapter
    -> MCP Client Adapter
      -> MCP Server / External Tool
```

模型不能直接调用 MCP server，也不能直接访问文件、数据库、浏览器、Coze、图片服务、视频服务或本地脚本。模型只表达工具调用意图；真正执行工具的是 ShanHaiEdu 后端的 ToolRouter、adapter、provider client 或 MCP client。

因此文档和代码中不得再写成“OpenAI SDK 是 LLM 调用 MCP 的适配层”。正确说法是：

```text
GptProtocolAdapter 使用 OpenAI SDK 连接 GPT/LLM；
MCP Client Adapter 使用 MCP 协议连接工具服务；
ToolRouter 连接两者，并负责门禁、预算、可用性、执行和 observation 回写。
```

### 5.14 工具注册先于 MCP 接入

ShanHaiEdu 当前的关键缺口不是“有没有 MCP”，而是“有没有稳定工具注册和路由层”。后续新增能力必须优先进入 `Tool Registry`，再由 `ToolRouter` 分流：

```text
ToolDefinition
  -> adapterKind: internal_capability | provider | mcp
  -> inputSchema / outputSchema
  -> requiresHumanGate
  -> requiredArtifactKinds / producedArtifactKind
  -> failurePolicy
  -> sideEffectLevel
```

执行分流统一为：

```text
内部文本节点 -> Internal Capability Adapter / CapabilityRunner
真实 PPTX、图片、视频、打包等外部能力 -> Provider Adapter
浏览器、文件、数据库、搜索、资料库等标准工具 -> MCP Client Adapter
```

这意味着 MCP 只是工具来源之一。即使未来接入 MCP，也必须通过 ToolRegistry / ToolRouter / PlanGuard / HumanGate / CapabilityAvailability / AgentHarnessBudget / ToolObservation / Artifact Truth Gate，不得把 MCP 调用直接塞进 `ConversationTurnService`。

## 6. 当前主线：MVP1

后续第一阶段只做一个核心骨架：

```text
MVP1：上下文、契约与门禁骨架
```

优先级：

```text
ConversationContextBuilder
ContextBudgetManager
SessionContextSnapshot
SessionCompactor
SummaryValidator
NodeContractRegistry
PromptCompiler
PlanGuard
HumanGate actionId
```

本阶段不追求大而全，目标是保证未来每个节点、每个 provider、每个产物都能接入同一套边界。

## 7. 不做项统一口径

MVP1 阶段明确不做：

```text
复杂 Admin 可视化后台
完整组织权限系统
全量视频生产链路重写
所有节点一次性契约化
长期用户画像自动写入
删除原始 Conversation Log
把 deterministic draft 伪装为真实生成
把模型自评当作质量门禁
```

如确实需要新增范围，必须先更新阶段规划，再实施。

## 8. 设计和开发流程

任何后续实施按固定顺序：

```text
确认是否属于当前 MVP 范围
  -> 对照统一口径和核心设计串联
  -> 写或更新阶段规划
  -> 先写测试或测试文档
  -> 小步实现
  -> 集中验收
  -> 审查是否破坏架构不变量
  -> 再进入下一任务
```

禁止：

```text
先写代码后补理由
为了临时效果绕过 PlanGuard/HumanGate
为了省事把规则继续堆进 prompt 字符串
为了演示把假文件/文本 fallback 标成真实交付
```

## 9. 代码落地边界

### 9.1 对话相关

`conversation-turn-service.ts` 后续只做编排，不应继续承载全部业务逻辑。新增逻辑优先拆到：

```text
conversation-context-builder.ts
context-budget.ts
session-compactor.ts
summary-validator.ts
```

### 9.2 契约相关

节点规则优先进入：

```text
config\node-contracts\*.json
src\server\contracts\node-contract-registry.ts
src\server\contracts\prompt-compiler.ts
```

### 9.3 门禁相关

计划和确认边界优先进入：

```text
src\server\guards\plan-guard.ts
src\server\guards\human-gate.ts
```

### 9.4 产物相关

真实文件状态必须由 Artifact 和 Provider Adapter 校验，不由对话消息推断。

## 10. 验收统一标准

后续任何人声称“完成”，必须至少回答：

```text
改了哪些文件？
对应哪个设计文档和哪个任务？
跑了哪些测试？
测试输出是否 0 failed？
是否影响真实 PPTX/图片/视频/最终包链路？
是否引入工程词外露？
是否保留完整 Conversation Log？
是否绕过 PlanGuard/HumanGate？
是否把草稿伪装成真实交付？
```

## 11. 架构不变量

以下原则不得被阶段性需求破坏：

1. 教师项目是中心对象。
2. Agent 是调度者，不是事实源。
3. Node Contract 是规则源，不是普通 prompt。
4. Tool/Provider 是执行器，必须可替换、可校验。
5. Artifact 是事实，必须可追溯、可下载、可验证。
6. Memory 是个性化，不可污染项目事实和节点契约。
7. Evidence 是可信度，教育内容不能无依据冒充教材结论。
8. Quality Gate 是交付保障，模型自称完成不等于完成。
9. 完整 Conversation Log 必须保留。
10. ContextPackage 是模型输入边界。
11. 高风险动作必须经过 HumanGate actionId。
12. 用户可见界面不得暴露工程词、本地路径、密钥和内部 schema。
13. 不确定任务可用两阶段判断，确定流程必须交给工作流和契约执行。
14. Harness 层不写业务分支，业务规则必须进入 Contract / Gate / Tool / Workflow。
15. 不采用玩具式硬步骤截断；必须设置耗时、成本、重试、重复和失败熔断。
16. 上下文是模型可见状态，不是系统真实状态源；真实状态必须持久化、可审计、可恢复。
