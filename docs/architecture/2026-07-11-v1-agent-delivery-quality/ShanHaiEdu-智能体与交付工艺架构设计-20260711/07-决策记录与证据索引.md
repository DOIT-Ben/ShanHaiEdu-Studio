## 3. 关键 ADR

### ADR-001：采用三层架构

**状态：Proposed / 推荐接受。**

决策：智能体架构、内容生产工艺架构、交付质量架构分别建模。拒绝把所有内容塞进 system prompt 或一个线性 workflow 文件。

后果：规则归属清楚，可单独版本化；需要 PromptCompiler 和 provenance 把三层在运行时组合。

### ADR-002：Director/Critic 以 Agent-as-Tool 暴露

**状态：Proposed。**

决策：Main Agent 不背完整手册；PPT Director、Video Director、DeliveryCritic 通过 ToolRouter 或等价受控入口调用。当前 `internal_capability` 只支持固定 `CapabilityId → AgentRuntimeTask → Markdown Artifact`，不能直接充当通用 Agent-as-Tool。实施前先建立 `AgentToolExecutor + runtime profile + input/output contract + report artifact`；推荐使用独立 `agent` adapterKind 或保持同等语义隔离的专用协议。

后果：Director/Critic 可被审计、版本化并受预算控制，但其报告不能自动推进产品节点、修改批准状态或替代 QualityDecision。

### ADR-003：只建立两个可路由 Skills

**状态：Proposed。**

决策：`edu-ppt-production`、`edu-intro-video-production`；共享不变量为 reference。拒绝每个节点建立一个 Skill。

### ADR-004：PPT 采用 Fast/Quality 双路径

**状态：Proposed。**

决策：保留 Coze 文本生成作为 Fast；Quality 路径必须消费批准的样张/资产并用结构化 Composer 组装、渲染和审查。

原因：Coze 当前只消费文本设计稿，不能证明使用批准资产；直接废弃 Coze 又会破坏已有可用路径。

### ADR-005：v8 立体生图是 Profile，不是全局合同

**状态：Proposed。**

决策：`ai_illustrated_3d` Profile 承载全 AI 生图、立体真实、clean family 等策略。全局合同只保留教材正确、投影可读、精确信息可编辑、真实文件和质量门。

### ADR-006：视频必须一等建模 shot/unit

**状态：P0 Proposed。**

决策：多个镜头不是同一 node 的互斥 Artifact 版本。批准唯一性必须是 `(project, node, unitId)`；时间线顺序来自 `shot.ordinal`。

原因：当前批准新 clip 会取消旧 clip，而 concat 需要多个 approved clip，真实数据模型自相矛盾。

### ADR-007：MP4 合成只允许媒体级工具

**状态：P0 Proposed。**

决策：`ffprobe → normalize → FFmpeg concat → ffprobe`。禁止 `Buffer.concat(MP4[])` 作为正式成片合成。

### ADR-008：Contract 必须进入运行主链

**状态：P0 Proposed。**

决策：PreValidator 在 Tool 执行前，PostValidator 在 Artifact 保存/状态推进前。只发布 JSON 但不消费不算合同系统。

### ADR-009：生成者与审查者分离

**状态：Proposed。**

决策：报告链固定为 `ValidationReport + CriticReport → deterministic QualityDecisionEngine → QualityDecision`，之后才进入教师决定。Critic 不重判文件结构、页数、ffprobe、hash 或血缘。生成 Agent 的自检只能作为草稿信息。

### ADR-010：暂不迁移 Vercel AI SDK

**状态：Proposed。**

决策：先完成领域质量架构。将来以 ADR 比较 runtime/streaming/durable execution，不把 SDK 迁移与交付质量绑定。

### ADR-011：ConversationControlResolver 不是第二规划器

**状态：P0 Proposed。**

决策：Resolver 只管理 `ActionOffer` 的确认、取消、修改、替换、过期和副作用政策。领域能力选择、缺少前置时的最小上游动作和质量返修路由归 Main Agent。复杂或否定表达不得依赖 capability 正则命中顺序。

原因：当前未提交 resolver 会在 Main Agent 之后用正则覆盖 capability，容易把“不要做分镜，先写脚本”误路由，也造成两个规划权威源。

### ADR-012：所有状态推进动作使用 ProjectExecutionLease

**状态：P0 Proposed。**

决策：TurnJob 的局部锁经验上升为统一 `projectId` mutation lease，覆盖 Artifact/Job 写入、Provider、文件和打包动作。租约使用 idempotencyKey、inputHash、heartbeat、expiry 和单调 fencingToken；只读 Main Agent/Director/Critic 不占长租约。

原因：产品基线明确同一项目同一时间只能推进一个生成任务，现有 TurnJob 锁不能覆盖其他执行入口。

### ADR-013：最终材料包必须经过 FinalDeliveryGate

**状态：P0 Proposed。**

决策：现有 package Tool 只负责组装。系统在打包前核对角色、路径资格、报告、人工决定和 lineage；打包后核对 ZIP、manifest 和 hash。Fast PPT、Short Preview 或陈旧来源不得进入最终包。

原因：文件存在和 ZIP 可打开不能证明输入具有最终交付资格。
