# V1-10G 原子容器切换计划

更新时间：2026-07-13

状态：`accepted / in progress`

## 1. 背景

V1-9G 目标服务器切换证明应用、数据和回退镜像可用，同时暴露现有发布控制面的缺口：仓库只有 compose、生产预检、健康 API、备份恢复和人工 runbook，没有可执行的常驻容器切换工具。人工流程曾把 HTTP 就绪与 Docker Health 分开处理，并在纠正时与原回退计时器重叠；手工重建还一度遗漏 compose 已声明的 `cap_drop: ALL` 与 `no-new-privileges`。

该问题不属于 Agent、媒体工作流或产品需求，而是 V1-10 发布可靠性缺陷。目标服务器已先恢复上述安全参数，本阶段将这套合同固化为可测试、可复用、单进程互斥的发布脚本。

## 2. 目标

- 新增 Linux 目标主机可执行的原子容器切换脚本。
- 使用 `flock` 保证同一常驻容器同一时间只有一个切换事务。
- 停止当前容器前，先用候选镜像、正式 env 和共享卷运行 `production-preflight`。
- 新容器固定使用非 root、`cap-drop ALL`、`no-new-privileges`、loopback 端口、共享 SQLite/Artifact、单实例和与 compose 一致的 Docker Health。
- 只有 Docker Health 与 HTTP `/api/health` 同时通过才提交切换；失败时移除失败容器、恢复旧容器名称并启动旧镜像。
- 成功后保留旧容器为停止态回退点；代码回退继续挂载当前共享数据，不自动恢复旧数据库或 Artifact。

## 3. 复用与适配

- 复用 `deploy\v1-compose.yml` 的用户、安全、端口、挂载和 Health 合同。
- 复用 `scripts\production-preflight.mjs` 作为候选镜像发布前门。
- 复用 `/api/health` 作为应用就绪门，不在切换脚本中重复实现数据库检查。
- 复用 V1-10D 的“代码回退不等于数据恢复”原则。
- 目标主机 Node 版本不是发布工具依赖；脚本只依赖 Bash、Docker、curl 与 util-linux `flock`。

## 4. 执行合同

1. 校验参数、依赖、候选镜像和当前容器。
2. 获取不可阻塞的独占锁；锁被占用立即失败，不等待、不启动第二个计时器。
3. 若当前容器已运行目标镜像且 Docker/HTTP 均健康，幂等返回 `already_current`。
4. 用一次性候选容器执行生产预检；失败时当前容器保持不动。
5. 停止当前容器并改名为唯一回退名。
6. 按固定安全合同创建并启动新容器。
7. 在单一超时循环内同时等待 Docker Health 与 HTTP 200。
8. 成功则保留旧容器为停止态；失败则删除失败容器、恢复旧容器原名并验证其重新健康。
9. 输出不含 env 内容、密钥、数据库路径、Artifact 路径或容器日志的最小 JSON 结果。

## 5. 风险与回退

- 风险：预检和新容器不能同时占用正式端口。处理：预检容器不映射端口，只执行脚本后退出。
- 风险：两个操作者并发切换。处理：`flock -n` fail-closed，第二个事务不做任何 Docker 变更。
- 风险：新容器应用 HTTP 200 但没有 Docker Health。处理：两门必须同时通过，缺一即回退。
- 风险：回退镜像启动失败。处理：返回明确的 `rollback_failed`，保留失败现场，不触碰共享数据，按 runbook 进入人工恢复。
- 回退：本阶段只新增发布脚本、测试与文档；可独立 revert。目标服务器验证失败时恢复当前 `v1-10f-098e651` 镜像，继续挂载当前共享数据。

## 6. 退出标准

- 静态合同测试证明互斥、预检顺序、安全参数、双健康门、失败回退和敏感信息边界存在。
- Bash 语法检查通过。
- 隔离端口与隔离 SQLite/Artifact 上完成一次成功切换和一次强制失败回退演练。
- 目标 localhost staging 继续保持单容器、非 root、安全参数、Docker healthy、200/404/403、重启和数据摘要不变。
- 专项、全量测试、构建与 `git diff --check` 通过。
