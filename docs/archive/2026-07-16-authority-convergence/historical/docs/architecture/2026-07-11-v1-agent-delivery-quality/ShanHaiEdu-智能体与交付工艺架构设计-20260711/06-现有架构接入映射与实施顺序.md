## 1. 接入结论

不重做现有 Main Agent，也不把 PPT、视频改成固定流水线播放器。正确接法是：

```text
教师可见层：线性里程碑、可按需切入、明确确认和交付状态
智能体控制层：Main Agent 规划 + 受控 ReAct + 局部返修
专业能力层：PPT/Video Director 作为 Agent-as-Tool
执行层：现有 ToolRouter + Provider/Package Tools
质量层：ValidationReport + CriticReport + QualityDecision
状态层：ProjectExecutionLease + Artifact/Job + Checkpoint
交付层：FinalDeliveryGate + 真实最终材料包
```

这既不否定当前产品基线中的“线性 AI 备课媒体工作台”，也不否定 M72 的按需能力入口。线性描述教师看到的里程碑；非线性描述智能体内部如何选择、改道、返修和恢复。

## 3. M72 的适配边界

M72 是“非线性入口”在途实现，不等于完整 ReAct，且当前有四个必须先收口的问题：

1. `ConversationControlResolver` 当前可用正则重新选择 capability 并覆盖 Main Agent 结果，事实上形成第二个 planner。
2. 计划声明支持确认、取消、修改、消歧和改道，但当前实际 return 主要只有 confirm、switch 和 ordinary。
3. 安全 internal tool 已开始免确认直执行，但 Node Contract 仍未进入运行主链。
4. 当前成功后仍按固定 delivery plan 计算下一能力，observation 没有真正回到 Main Agent 做 Replan。

因此 `ConversationControlResolver` 的最终职责必须收窄为：

- 识别是否在确认、取消、修改或替换一个已披露 `ActionOffer`。
- 校验收到的 actionId 是否仍指向当前活动动作。
- 对 Main Agent 已给出的结构化意图做控制语义裁决。
- 应用服务端 `SideEffectPolicy`，但不自行决定教学领域能力。
- 模糊或复合表达返回 clarification observation，由 Main Agent 规划最小下一步。

它不得：

- 用关键词正则成为第二个 capability planner。
- 决定缺少前置时应该补哪个上游工件。
- 生成 PPT/视频业务规则。
- 代替 AgentHarness 的 Observe、Evaluate 或 Replan。

建议由 Main Agent 输出 `AgentIntent { intentKind, candidateCapabilityId, confidence, ambiguity, requestedChange }`，Resolver 只消费它与当前 `ActionOffer` 状态。明确的显式能力命令可以作为高置信控制信号，但否定句、复合句和上下文改道必须回到 Main Agent，而不是依赖正则命中顺序。

## 5. Agent-as-Tool 的具体接法

当前 `internal_capability` 不能直接承担 Director/Critic，因为它强绑定 Capability、固定 RuntimeTask 和 Markdown Artifact。不能只给 `ToolDefinition` 多加几个字段就宣称完成。

第一阶段应建立通用 `AgentToolExecutor` 协议：

```text
AgentToolDefinition
  toolId
  agentProfileId
  inputContractId/version
  outputContractId/version
  allowedToolClasses: read_only only (v1)
  executionSideEffectLevel: none
  resultPersistencePolicy: report_artifact
  persistenceSideEffectLevel: artifact_write
  persistenceLeaseMode: short
  requiresHumanGate: false

AgentToolInvocation
  projectId / runId / sourceMessageId
  worldStateVersion
  stage / targetLocators
  trustedArtifactRefs + reportRefs
  skillSliceRef + profileRef + contractRef
  inputHash

AgentToolResult
  status: succeeded | needs_input | failed | inconclusive
  structuredOutput
  reportArtifactDraft
  observation
  runtime/model/prompt/contract/skill provenance
```

实现边界建议是新增独立 `agent` adapter kind 或等价的专用 executor 分支，而不是扩张现有 `internal_capability` 直到两种语义混在一起。AgentToolExecutor 只返回结构化结果；Harness 在短租约和 inputHash 复核后持久化报告。结果报告可以作为内部 Artifact，但不能走当前“Tool 成功即保存产品 Artifact”的默认路径，也不能自动把教师产品节点标记为完成或批准。

三个首批 Agent Tools：

| Tool | 输入 | 输出 | 权限 |
|---|---|---|---|
| `ppt_director.plan_or_repair` | 当前 stage、教材证据、目标 pageIds、历史问题、PPT Skill slice | PlanDraft / RepairPlan | 只读，不调用 Provider |
| `video_director.plan_or_repair` | 课程锚点、目标 shotIds、Provider 能力摘要、历史问题、Video Skill slice | Creative/Beat/Shot/Repair Draft | 只读，不调用 Provider |
| `delivery_critic.review` | ValidationReport、真实 render/media、rubric、target locators | CriticReport | 只读，不修改产物、不重判硬门 |

## 6. ExecutionControl：统一项目执行租约

现有 `ConversationTurnJob` 已有 `lockedBy/lockedUntil` 和过期恢复语义，可复用算法经验，但它只锁对话任务，不能覆盖 ToolRouter、直接生成路由、Provider Job 和最终包。

新增 `ProjectExecutionLease` 的适用边界：

- 需要租约：`artifact_write`、`external_call/provider_call`、`file_write`、`package_write`。
- 不需要长租约：Main Agent 思考、普通聊天、Director/Critic 只读推理、只读 Validator。
- 若要保存只读推理结果，保存前获取短租约并重新核对 `inputHash/worldStateVersion`。
- HumanGate 和 PreValidator 通过后、Act 前才获取，等待教师时不能占锁。

最小字段：

```text
leaseId, projectId, scope=project_mutation
ownerRunId, ownerJobId?, actionInvocationId, actionKey
actionOfferId?, humanDecisionId?
idempotencyKey, inputHash
status: active | released | expired
fencingToken
acquiredAt, heartbeatAt, expiresAt
releasedAt?, releaseReason?
```

`actionInvocationId` 是每个计划动作的通用身份，无论是否需要人工确认都必须存在；`actionOfferId/humanDecisionId` 只在 HumanGate 场景出现。每个项目只能有一个 active slot；过期回收必须递增 `fencingToken`。leased action 导致的每一次状态写都在事务中校验 `leaseId + fencingToken`，包括 providerTaskId、poll state、attempt、Artifact、Node、Job、report 和 observation，阻止旧 worker 迟到覆盖。相同 `idempotencyKey + inputHash` 复用原 Job/结果；相同 key 不同 hash 返回冲突。不同项目仍可并发。

Provider submit 已发出但 taskId 尚未可靠写入时是特殊不确定窗口：记录 `PROVIDER_SUBMISSION_UNKNOWN`，先使用 Provider idempotency key 或查询接口对账；无法确认时暂停并请求人工处理，不得自动再次提交付费请求。

物理结果与业务发布必须分开：Tool 先把原始文件或输出放进 immutable staging/quarantine；PostValidator 通过后，才在带 fencingToken 的事务中创建/提升可见 Artifact 并推进 WorkflowNode。失败时只保留 rejected evidence、ValidationReport 和 observation，不更新 selected/approved 指针。

## 7. ContractControl 与 QualityControl

建议逻辑模块如下，名称不是强制文件名：

```text
MethodRegistry
  -> SkillDescriptor / StageSlice / ProfileRegistry

PromptCompiler
  -> role + world state + contract + skill slice + profile + output schema

ContractControl
  -> ContractRegistry
  -> ContractPreValidator
  -> ContractPostValidator
  -> ValidatorBindingRegistry

QualityControl
  -> DeterministicValidator -> ValidationReport
  -> DeliveryCritic Agent Tool -> CriticReport
  -> QualityDecisionEngine -> QualityDecision

ExecutionControl
  -> ProjectExecutionLease
  -> IdempotencyLedger
  -> Fencing / Queue / Recovery

FinalDeliveryControl
  -> DeliveryProfileRegistry
  -> pre-package FinalDeliveryGate
  -> package Tool
  -> post-package FinalDeliveryGate
```

Critic 不得输出或改写页数、文件结构、ffprobe、hash、参考图是否真实发送等硬事实。最终结果只能由确定性 `QualityDecisionEngine` 聚合：

```text
ValidationReport + CriticReport + QualityDecisionPolicy
  -> QualityDecision(pass | repair | block)
```

### 9.1 P0 事实模型

```text

每个 unit 可有多个 clip version
每个 unit 只能选择一个 approved clip
多个 unit 可以同时 selected/approved

```

### 9.2 新能力

| 能力 | 推荐形式 | 关键要求 |
|---|---|---|
| Video Director | Agent-as-Tool | VideoIntent、BeatSheet、ShotSpec、ReferencePlan、局部 RepairPlan |
| video reference assets | Provider Tool | `assetDomain=video`；禁止 PPT/contact sheet/候选视频抽帧血缘 |
| shot resolver | deterministic service | ProviderProfile 与 ShotSpec 匹配 |
| submit/poll/download | Provider Tool + durable Job | 真实传参考资产；保存 providerTaskId/inputHash |

## 10. FinalDeliveryGate 的接入

当前 `create_final_package` 已能检查 PPTX、图片、视频存在并生成带 manifest 的 ZIP，但这只证明“能打包”，不能证明“所有输入都有最终交付资格”。

正确协议：

```text
WorldState 编译 sourceRefs
  -> pre-package FinalDeliveryGate
  -> exact idempotency lookup(projectId + gateInputHash)
  -> HumanGate(ActionOffer v2)
  -> acquire ProjectExecutionLease
  -> recheck gateInputHash/worldStateVersion/preGateReport digest
  -> package Tool
  -> stage package candidate
  -> post-package FinalDeliveryGate
  -> fenced promote/publish 或 quarantine
  -> release lease
```

`DeliveryProfile` 决定本次必需角色。当前完整课堂交付 Profile 可以要求 lesson、quality PPT、课堂图和 full intro video；其他未来 Profile 可调整角色，但不能把 `preview_only` 产物提升为最终产物。

FinalDeliveryGate 必须核对：

- required roles 齐全且是当前 selected version。
- 每个 source 都是 `final_eligible`；Fast PPT 和 Short Preview 必须阻断。
- ValidationReport、CriticReport、QualityDecision、HumanDecision 全部绑定同一 artifact version/hash/path。
- sourceRefs 使用 `{id,digest}` 报告引用并包含 inputHash/lineageHash；intro video 还包含指向当前 lesson 的 courseAnchorRef。
- 当前上游选择与 lineageHash 一致，任何 stale 节点或报告都阻断。
- 视频课程锚点指向当前 lesson，但不要求视频依赖 PPT。
- post-package ZIP 条目、manifest、content hash 完全一致。

### Phase 0.5：先建立状态推进安全底座

- 将 TurnJob 租约经验抽成统一 `ProjectExecutionLease`，覆盖 ToolRouter 和直接生成路由。
- 加入 fencing、心跳、过期恢复、FIFO/复用策略与幂等 ledger。
- 给 GenerationJob 预留 `idempotencyKey/inputHash/providerTaskId/leaseId/fencingToken`。
- 冻结 FinalDeliveryGate 的 pre/post 合同和失败码。

验收：同项目并发只允许一个状态推进动作；同幂等请求不重复调用；过期 worker 无法写回。

### Phase 1：让 Contract、Agent Tool 和质量报告进入主链

- 所有 direct internal/provider/package 路径统一经过 ContractPre/PostValidator。
- 建立通用 `AgentToolExecutor + runtime profile + output contract + report artifact`。
- 执行结果采用 `immutable staging -> PostValidator -> fenced promote/quarantine`，不能先污染产品 WorldState 再校验。
- 实现 ValidationReport、CriticReport、QualityDecisionEngine 与类型化 locator。
- 实现 FinalDeliveryGate preflight/postvalidate 骨架，当前旧 final package 只可作为待验证候选。
- 未被合同覆盖的 M72 direct internal capability 必须 beta flag/allowlist，不能默认裸放行。

验收：缺输入、错血缘、假文件、错输出、报告错版本和 preview path 都能产生可定位 observation；Critic 无权覆盖硬门。

### Phase 1.5：最小受控 ReAct

- 每次 Tool/Validator/Critic observation 回到 Main Agent。
- 支持 `continue | repair_unit | repair_upstream | ask_teacher | pause | finish`。
- 缺硬前置时由 Main Agent 选择最小上游动作，Resolver 不负责规划。
- 建立 run 级 `maxSteps/maxToolCalls/maxCost/maxWallClock/maxContext` 与 action/tool/provider 级预算。

验收：至少完成“工具失败后换路”“质量失败后定点返修”“缺输入时提出最小问题”三种 Replan；不存在无限循环或绕预算调用。

### Phase 2：闭环 PPT Quality 路径

- 保留并清晰标记 Fast Coze。

### Phase 3：闭环视频逐镜头路径

- 迁移重复旧视频节点。

### Phase 4：Durable checkpoint 与更高自主度

验收：中断后先恢复 providerTaskId/poll，不重复付费提交；等待人审时不占 lease；恢复后从新 WorldState Replan。

### Phase 5：框架与 SDK ADR

比较当前 OpenAI SDK + 自有 Harness、Vercel AI SDK、OpenAI Agents SDK、LangGraph、Vercel Workflows。只有量化证明迁移能降低维护成本或补足流式/恢复能力时才切换；框架迁移不作为交付质量前置。

## 13. 风险与回退

| 风险 | 控制 | 回退 |
|---|---|---|
| M72 覆盖 Main Agent 规划 | Resolver 收窄为控制语义；复合/模糊表达回 Agent | beta flag 关闭 direct switch |
| 影响当前 Coze | Fast 与 Quality 并行，路径身份不可变 | 关闭 Quality feature flag，保留 preview |
| Lease 卡死或旧 worker 写回 | TTL、heartbeat、fencing、recovery | pause 项目动作并人工恢复，不放松 fence |
| Critic 成本/漂移 | 先 Validator，证据裁剪，Rubric 版本化 | 转人工语义审查，不绕过硬门 |
| Final gate 过度固定流程 | required roles 由 DeliveryProfile 决定 | 换 Profile，不改写 preview ceiling |
| Skills 规则漂移 | MethodRegistry 单一版本源 | 回退上一 Skill/Profile 版本 |
| 自动循环失控 | run 总预算 + action 预算 + checkpoint | pause，转单步/人工模式 |

## 14. 实施前仍需冻结的参数

1. ActionOffer 自然语言确认对高副作用动作的最终产品口径。
2. ProjectExecutionLease TTL、心跳频率、队列公平性和批次范围。
3. DeliveryProfile v1 的 required roles。
4. AgentToolExecutor 采用新 `agent` adapterKind 还是等价专用内部协议。
5. Rubric 初始阈值的教师样本校准方案。

这些参数会影响实现细节，但不改变本设计的职责边界和实施顺序。
