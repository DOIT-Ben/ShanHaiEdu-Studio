## 4. 当前智能体并非纯线性工作台，但也还不是完整自主 Agent

### 已经是 Agent 的部分

- 模型主对话 Agent 可以自然理解输入、追问、聊天、选择 capability，而不是只靠关键词分类。
- Main Agent 输出 `toolPlan`，代码负责 CapabilityAvailability、PlanGuard、HumanGate、预算和真实执行。
- 当前未提交的 Control Resolver 会在模型之后处理 actionId 确认、显式改道和直接 capability 请求。
- Tool 结果会写成 observation，失败不会直接伪装成功。
- ContextPackage、AgentWorldState、Artifact/Job 提供了“模型可见状态”和“系统真实状态”的分离基础。

### 仍偏线性的部分

- 完整交付顺序被分别硬编码在 `capability-planner.ts:16-32` 和 `model-main-conversation-agent.ts:31-47`。
- 每个产物成功后生成下一个 pending plan，等待“继续下一步”，见 `conversation-turn-service.ts:449-493`。
- 当前不是一次 run 内持续 Observe→Replan，而是跨教师对话逐节点推进。
- 原生工具循环只有在 `SHANHAI_OPENAI_NATIVE_TOOL_LOOP=1` 时启用，且默认 `maxToolRounds=1`，见 `runtime-factory.ts:34-43`、`openai-tool-loop-runner.ts:45-84`。
- 多工具并行调用被直接阻断；工具失败会结束该工具循环，不会自动让模型基于失败 observation 重规划。
- 当前预算只有相同 action、连续失败、capability retry 和 context token 等局部限制，没有 run 级 maxSteps/maxToolCalls/maxCost/maxWallClock 总熔断。

结论：当前是“模型主导规划 + 受控单步执行 + 对话级继续”的 Agent 雏形。它吸收了 ReAct 思想，但还没有完整、可恢复的受控 ReAct Harness。

### ConversationControlResolver 的边界

当前在途 Resolver 有必要解决自然语言确认、旧计划失效和直接能力切换，但必须限定为**对话控制裁决器**：

- 可以识别确认/取消/改道，并把服务端已有 action/Tool policy 绑定到当前 turn。
- 可以把教师明确点名的 capability 转为候选 ActionOffer。
- 不得决定 PPT/视频专业路径、补造前置输入、绕过 CapabilityAvailability，或成为第二套 capability planner。
- 正则只能作为确定性控制协议的窄层，不得重新把 Main Agent 退化成关键词分类器。

### “线性工作台”与“非线性智能体”如何兼容

`current-requirements-baseline.md:9` 仍把产品定义为线性 AI 备课媒体工作台；最新未提交 `requirements-backlog.md` 的 RQ-015 又要求教师可从任意可用能力切入。实施前必须统一口径：

```text
教师可见体验：线性里程碑、清楚的产物状态和确认点
智能体内部执行：受控 ReAct、按需选路、局部返修和失败改道
```

在产品基线正式更新前，M72 仍是未提交设计/实现观察，不能以阶段计划覆盖最高权威基线。

## 5. Vercel AI SDK 核验

当前项目没有 Vercel AI SDK：

- `package.json:62-66` 有 Next.js、`openai@^6.45.0`、PptxGenJS、React。
- `package.json`、lockfile 中没有 `ai`、`@ai-sdk/*`、LangGraph 或 OpenAI Agents SDK 依赖。
- 当前模型协议由官方 `openai` 包和自定义 OpenAI Responses adapter/tool loop 实现。

纠偏：Vercel AI SDK 能提供 ToolLoopAgent、工具调用、流式消息与 UI 生态，但“持久化业务工作流”不是只装 AI SDK 就自动得到；Vercel Workflows 是独立产品。对当前项目而言，换 SDK 不会自动补上 PPT 样张、逐页渲染、视频 shot 模型、ffprobe、Critic 和质量量表。

建议：短期不迁移。先在现有 `AgentRuntime + ToolRouter` 上完成质量架构；将来若流式 UI 和 agent loop 维护成本成为主要问题，再做独立 ADR 比较 Vercel AI SDK、OpenAI Agents SDK、LangGraph 与当前自研边界。

4. 拼接使用 `Buffer.concat(MP4[])`，不是容器级合成。证据：`tools\package-tool-adapter.ts:47-64`。
5. 拼接顺序按 Artifact version/updatedAt 猜测，不按 `shot.ordinal`。证据：同文件 `:209-211`。
6. 质量门只有 `ftyp/moov` 等基础文件标记，没有 ffprobe、时长、流参数、连续性、音字和内容审查。

## 9. 最重要的分析纠偏

- 不是“项目没有 Tool 层”，而是 Tool 层已经较成熟，质量工艺和合同执行不足。
- 不是“把手册封装成一个 Tool 就结束”，而是 Main Agent 调用专家 Agent-as-Tool，专家内部按阶段加载 Skill；真实执行仍走普通 Tool。
- 不是“加 Vercel AI SDK 就有持久化多 Agent”，而是 SDK、业务状态、checkpoint 和交付质量是四个不同问题。
- 不是“线性节点越多质量越高”，而是节点必须有真实工件、硬门禁、质量证据和正确回退。
- 不是“把所有规则写死就可靠”，而是只把教学事实、安全、产物真实性和血缘写成硬合同；风格、镜头数、模型、样张数等进入可覆盖 Profile。
