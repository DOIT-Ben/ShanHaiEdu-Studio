## 3. 未完成或需要重构的事项

### 3.1 上下文记忆不完整

当前模型上下文缺少完整 `AgentWorldState`，不能稳定知道：

- 当前目标。
- 当前节点。
- 最新产物状态。
- 用户刚刚确认了什么。
- 哪些工具真实可用。
- 最近失败的根因。

### 3.2 任意节点回退能力不足

当前虽然有版本和 artifact，但还没有产品化的任意节点回退机制。

需要补齐：

- 节点 checkpoint。
- 产物版本依赖图。
- 回退后下游失效标记。
- 回退后的模型上下文重建。

### 3.3 工具调用没有形成 Agent Loop

目前工具失败后大多由服务端直接返回失败文案。

需要改为：

```text
tool call -> observation -> model reflection -> retry/fallback/ask/complete
```

### 3.4 PPTX 生成工具不稳定且无可靠 fallback

当前真实 PPTX 强依赖外部 Coze 路径。

未完成：

- 本地 PPTX fallback。
- HTML/SVG 转 PPTX fallback。
- 失败类型诊断。
- 重试策略。
- 让模型知道 fallback 选项。

### 3.5 门禁口径需要调整

旧口径过于强调前置硬门禁。新口径应改为：

- 真实交付硬门禁。
- 理解、规划和节点推进软约束。
- 产品范围默认引导，不用关键词硬锁。

### 3.6 动态工具健康状态缺失

工具注册表应记录运行时健康状态，而不仅是静态能力定义。

至少需要：

- `available`
- `lastFailure`
- `retryable`
- `fallbacks`
- `inputReadiness`
- `qualityGate`

### 4.1 已完成阶段如何归档

### 4.2 未完成阶段如何迁移

未完成事项统一迁移到 M61 及后续 Agent OS 主线，不继续在 M59/M60 文档里追加补丁。

建议映射：

| 旧问题 | 新主线归属 |
| --- | --- |
| PPTX 找不到设计稿 | AgentWorldState + 隐式确认语义 |
| 模型忘记上下文 | ContextCompiler + 工作记忆 |

## 5. 下一阶段建议拆分

### M61-A：AgentWorldState 与 ContextCompiler

目标：让模型每轮知道完整当前进度和可行动作。

### M61-B：动态节点状态与回退

目标：节点版本化、可回退、下游失效可解释。

### M61-C：工具 observation loop 与 fallback

目标：工具失败后模型可重试、降级、换工具或询问教师。

### M61-D：PPTX 多路径交付

目标：Coze 失败后可走本地 PPTX 或 HTML/SVG 转 PPTX fallback。

### M61-E：渐进门禁重定义

目标：保留真实交付硬门禁，放松理解和规划阶段硬锁。

## 6. 当前不要继续扩大修改的区域

在新主线确认前，不建议继续大规模修改：

- 节点列表顺序。
- Coze PPTX 业务口径。
- 视频全链路工具门禁。
- 小学范围硬编码扩展。
- 更多 if-else 形式的异常补丁。

应先完成 M61 底层方案确认。
