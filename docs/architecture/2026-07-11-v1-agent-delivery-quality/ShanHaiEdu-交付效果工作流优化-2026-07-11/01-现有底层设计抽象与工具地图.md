# 现有底层设计抽象与工具地图

## 1. 系统真实定位

现有系统不是单一聊天机器人，也不是“一条 Prompt 生成所有材料”。它已经接近一个受控的教育内容生产操作系统：

```text
教师项目与对话
  -> ContextPackage / AgentWorldState
  -> 主 Agent 理解与计划
  -> PlanGuard / HumanGate
  -> ToolRegistry / ToolRouter
  -> Internal / Provider / Package Adapter
  -> Artifact / ToolObservation / GenerationJob
  -> Quality Gate
  -> 下载与最终交付
```

### 核心不变量

1. Agent 是调度者，不是真实状态源。
2. ToolRouter 是唯一工具执行入口，模型不能直接触碰 Provider、文件或数据库。
3. Provider 工具只信任服务端解析出的、同项目、已批准的 Artifact。
4. 草稿、设计稿、真实文件和最终包必须分层。
5. 外部调用必须经过 HumanGate；失败必须形成 Observation。
6. 文件真实不等于教学效果好；这一层正是本优化包要补齐的部分。

## 2. 现有智能体职责

### 主对话 Agent

负责理解教师意图、判断下一步能力、提出工具调用和解释状态。它应读取项目状态、最近消息、已批准产物、可用能力和运行时约束，不应直接输出“文件已生成”等事实结论。

### Capability Runtime

负责生成需求规格、教案、PPT 大纲、PPT 设计稿、视频知识锚点、创意主题、脚本、分镜、资产说明和片段计划等文本 Artifact。目前主要通过统一的任务指导字段和 Markdown 自检清单约束模型。

### ToolRouter

负责解析工具、检查实现状态、核验上游 Artifact、选择 Adapter、把成功或失败转换为统一 ToolExecutionResult。

### Provider Adapter

负责调用 Coze PPT、图片和视频 Provider，并把返回文件写入 Artifact Storage。目前强项是文件真实性校验，弱项是内容和视觉效果评价。

### Package Adapter

负责视频片段组装和最终材料包。当前只检查容器、文件存在和清单，尚未形成真正的媒体时间轴与跨产物一致性审查。

## 4. 现有节点契约覆盖

当前文件化 Contract 只有：

- `requirement_spec`
- `lesson_plan`
- `ppt_design`
- `coze_ppt`
- `final_package`

缺失 Contract 的高风险节点包括：PPT 叙事、逐页视觉决策、PPT 渲染审查、视频创意选择集、节拍表、分镜、资产 Bible、片段提示、片段审查、成片审查和跨产物一致性审查。

## 6. 应保留与应升级的边界

### 应保留

- ToolRegistry / ToolRouter / Adapter 分层。
- resolved Artifact 真值校验。
- HumanGate 与 Provider 调用隔离。
- Artifact Truth Gate 和失败 Observation。
- 视频与 PPT 分支解耦。

### 应升级

- 从“字段出现”升级为“结构化 schema + rubric + 证据”。
- 从“文件可打开”升级为“效果可验收”。
- 从“整份重做”升级为“失败页/失败镜头定向返工”。
- 从“单模型生成后自检”升级为“生成者与审查者分离”。
- 从“上游 Markdown 摘要”升级为“版本化 DesignSpec / ShotSpec”。
