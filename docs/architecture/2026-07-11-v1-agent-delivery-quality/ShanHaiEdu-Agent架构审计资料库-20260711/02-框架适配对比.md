# Agent 框架适配对比

## 1. 评估口径

框架不是按“名气”排序，而是按 ShanHaiEdu 的真实需求评估：

- 是否支持 `Observe -> Plan -> Guard -> Act -> Observe -> Replan`。
- 是否支持教师自然语言打断、审批、回跳和恢复。
- 是否支持 PPT Director、Video Director、Critic 等专业智能体。
- 是否能在长时间 Provider 任务中保存运行状态。
- 是否能复用当前 ToolRouter、Artifact、Job、Lease 和质量门。
- 是否适配 Next.js、流式 UI 和多 Provider。
- 是否会形成两个互相竞争的业务事实源。

## 2. 总体对比

| 判断项 | 继续全自研 | LangGraph OSS Core | OpenAI Agents SDK TS | Vercel AI SDK / Workflow SDK |
|---|---|---|---|---|
| ReAct Tool Loop | 当前较弱 | 强 | 强 | `ToolLoopAgent` 强 |
| 条件边、循环、动态 Replan | 需继续自研 | 原生 StateGraph | Runner Loop 强，显式业务图较弱 | Loop 控制强，任意图状态编辑较弱 |
| Human Interrupt / Resume | 需继续自研 | 原生 `interrupt` / `Command(resume)` | 原生审批和序列化 RunState | Tool Approval 与 Workflow Hook |
| Durable Checkpoint | 当前不完整 | 原生 Checkpointer | 需要应用自行持久化 RunState | ToolLoopAgent 内存态；Workflow SDK 可持久执行 |
| 子智能体 | 需继续自研 | Subgraph | 原生 Agent-as-Tool / Handoff | 用 Tool 包装 Subagent |
| 状态回放和分支 | 需继续自研 | Time travel / fork | 不是核心强项 | Event log 可恢复，但不是 Artifact 图编辑器 |
| Next.js 流式 UI | 自研 | 支持事件流，需要适配 | 支持流式运行，需要适配 | 最强项 |
| Provider 中立性 | 当前 OpenAI-compatible + 自有 Adapter | 较高 | 中等，OpenAI 体系最顺 | 高，Provider 生态丰富 |
| Artifact DAG / 版本 / stale | 已有业务基础 | 不提供 | 不提供 | 不提供 |
| 质量 Contract / Rubric | 已有业务基础 | 不提供 | Guardrail 不能替代 | Approval/Schema 不能替代 |
| 迁移复杂度 | 持续增长 | 中 | 中 | UI 低，Durable Workflow 中高 |
| 当前成熟度风险 | 自研维护风险高 | OSS Core v1 相对稳定 | JS SDK 仍为 0.x | AI SDK 与 Workflow SDK 版本变化较快 |

## 3. LangGraph OSS Core

### 适配优势

- StateGraph 直接表达观察、规划、门禁、执行、评价和返修条件边。
- Checkpointer 在 super-step 边界保存图状态。
- `interrupt()` 和相同 `thread_id` 恢复适合样张确认、教师审批和等待外部事件。
- Subgraph 适合 PPT Director、Video Director 和 Delivery Critic。
- Streaming 可输出消息、状态更新、工具事件和自定义进度。
- OSS Core 可嵌入当前 Next.js 服务端，不要求先采用 LangSmith Agent Server。

### 不能替代

- Artifact 版本、血缘、stale 和真实文件。
- Provider taskId、inputHash、幂等、取消和结果隔离。
- ProjectExecutionLease。
- PPT/视频 Contract、Validator、Rubric 和 FinalDeliveryGate。

### 风险

- Graph State 与业务数据库形成双事实源。
- Interrupt 恢复时节点从头执行，副作用必须幂等。
- 图代码升级可能破坏历史 Checkpoint 恢复。
- 错误地把大型工件放进 Graph State 会造成状态膨胀。

### 当前定位

首选候选：接管 AgentHarness 和 WorkflowEngine，不接管业务事实数据库。

## 4. OpenAI Agents SDK TypeScript

### 适配优势

- Runner 原生管理模型、工具结果、继续推理和停止条件。
- `agent.asTool()` 很适合 Main Agent 调用 PPT Director、Video Director 和 Critic。
- 原生 Handoff、Guardrail、Tracing、Sessions 和 Human-in-the-loop。
- RunState 可序列化，适合长时间审批后恢复。

### 不能替代

- 自动业务节点 Checkpoint。
- Artifact DAG、版本和下游失效传播。
- Provider exactly-once、跨进程 Lease 和任务队列。
- 教育交付质量 Validator。

### 风险

- JavaScript SDK 当前仍为 0.x，升级兼容风险高于 LangGraph v1 Core。
- Guardrail 覆盖边界不能替代现有 PlanGuard 和确定性质量门。
- Tracing 可能记录模型与工具输入输出，必须制定教材和教师数据策略。
- 若与 LangGraph 同时承担循环权威，会形成嵌套运行时和调试困难。

### 当前定位

第二候选：如果最终更看重 Manager + Agents-as-Tools，而不需要复杂可编辑状态图，可选它作为唯一 Agent Runtime Kernel。第一阶段不建议与 LangGraph 同时引入。

## 5. Vercel AI SDK 与 Workflow SDK

### 适配优势

- 与 Next.js、React 的流式消息和 Tool 状态 UI 结合紧密。
- ToolLoopAgent 支持多步工具循环、停止条件、动态工具开放和结构化输出。
- `prepareStep` 适合按当前阶段压缩 ContextPackage、切换模型和限制 Tool。
- Workflow SDK 可通过持久步骤、事件和 Hook 支持长任务。

### 不能替代

- ToolLoopAgent 自身是内存态，不能单独承担进程崩溃后的 durable resume。
- Subagent 主要通过 Tool 包装，不具备 OpenAI Agents SDK 同等级的原生 Agent-as-Tool 语义。
- Workflow 事件日志不是教师项目、Artifact 或质量决策的事实数据库。
- Tool Approval 不理解 ShanHaiEdu 的 actionId、inputHash、Artifact version 和 gateScope。

### 风险

- AI SDK 主版本和 Workflow SDK 仍在快速变化，旧教程中的 Durable Agent 或审批字段可能已经迁移或弃用。
- 将 UI SDK、Agent Loop 和 Workflow SDK 一次性引入会扩大迁移面。
- Workflow 的取消不能保证已经提交的 Provider 动作停止，仍需结果隔离。

### 当前定位

优先作为后续 Next.js 流式 UI 和 Tool 事件协议层。是否采用 Workflow SDK，需要在版本锁定和部署兼容 Spike 后决定。

## 6. 当前候选排序

| 优先级 | 候选 | 用途 |
|---:|---|---|
| 1 | LangGraph OSS Core | 状态图、Replan、Interrupt、Checkpoint、专业子图 |
| 2 | OpenAI Agents SDK TS | 单一 Agent Runtime Kernel 的替代候选，Agent-as-Tool 更直接 |
| 3 | Vercel AI SDK | Next.js 流式交互、Tool 状态和 Provider/UI 协议 |
| 暂不采用 | LangSmith Agent Server | 当前阶段运维、许可证和平台耦合收益不明确 |

排序是审计建议，不是已接受决策。正式选型必须以 Spike 结果更新。
