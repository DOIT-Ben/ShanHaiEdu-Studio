# PPT 工作流优化

## 1. 工作流目标

把“有 12 页、每页有四层字段”升级为“有完整学习叙事、每页只有一个教学动作、主视觉真正解释内容、真实投影可用”。

## 2. Agent 可选黄金路径

下列是效果较稳定的参考路线，不是顶层固定 DAG。Main Agent 应根据已批准 Artifact、教师输入和 Tool Observation 决定从哪里进入、是否复用、是否并行以及是否回退。已有合格 Deck Narrative、Visual Strategy 或页面资产时，不得为了“走完流程”重复生成。

```text
EvidencePackage + InstructionIntent
  -> Deck Communication Job
  -> Deck Narrative
  -> PageSpec Set
  -> Visual Strategy
  -> Per-page Asset Plan
  -> Asset Generation / Reconstruction
  -> PPTX Generation
  -> Slide Rendering
  -> Deterministic QA + Multimodal Critic
  -> Page-level Repair
  -> Teacher Rehearsal Gate
```

不可绕过的只有：教材/数学真值、每个最终页面的可执行 PageSpec、真实 PPTX、逐页渲染审查和最终交付审计。其余节点是 Agent 可调用的能力或 Tool 内部 Capsule。

## 3. 新增关键能力与产物契约

### 3.1 Deck Communication Job

一句话格式：

```text
到课程结束时，[学生群体] 应能 [可观察结果]，因为他们经历了 [核心认知变化]。
```

同时定义 PPT 的角色：讲授、设问、演示、操作、反馈或总结。PPT 不承担完整教案文本展示。

### 3.2 Deck Narrative

建议小学公开课默认节奏：

```text
情境张力 -> 观察与猜测 -> 操作/表征 -> 形成概念 -> 变式辨析 -> 迁移应用 -> 出口评价
```

输出每一段的学生认知状态、关键问题、证据和转折，不先定页数再填内容。

### 3.3 PageSpec

每页必须包含：

- `narrative_job`：本页在整套叙事中的唯一职责。
- `teaching_action`：教师在这一页做什么。
- `student_action`：学生可观察的反应或操作。
- `takeaway_title`：表达结论或任务，而非栏目名。
- `primary_visual_type`：实物情境、数量表征、对比、过程、关系图、操作台等。
- `primary_visual_brief`：主视觉必须解释什么。
- `visible_text_budget`：标题、问题、标签的数量和长度上限。
- `local_math_layers`：必须用本地矢量/文字保证准确的数字、公式、数量和符号。
- `layout_constraints`：安全区、最小字号、对比度和不可遮挡区域。
- `acceptance_checks`：本页独有的数学和教学检查。

### 3.4 Visual Strategy

全套只定义 3～5 种页面轮廓，而不是每页一个模板：

1. 沉浸情境页：主视觉全幅，问题覆盖在安全区。
2. 聚焦观察页：单一对象或数量组，局部高亮。
3. 操作与验证页：大操作区 + 极少指令。
4. 对比辨析页：两种表征并置，但不做卡片墙。
5. 总结迁移页：关系结构或学生作品成为主视觉。

同一轮廓不得连续使用超过 2 页；不得把五六个小卡片平均铺满画面。

## 4. 逐页主视觉策略

### 生成图承担

- 真实感或绘本式场景。
- 难以本地绘制的主体、人物、空间或质感。
- 提供情绪与注意焦点。

### 本地确定性图层承担

- 数字、公式、文字、字幕。
- 精确数量对象、点阵、计数器、几何关系。
- 需要课堂逐步出现或可编辑的标注。
- 教材引用、页码和证据标签。

### 资产生成要求

- 每个生成请求必须绑定 `page_id`。
- Prompt 明确构图留白位置、主体朝向、画幅和裁切方式。
- 不在图像中烘焙中文、公式和精确数量。
- 生成后记录尺寸、模型、Prompt、哈希和许可/来源。

## 5. PPT 生成 Prompt 的关键升级

Provider 不应只收到 Markdown 四层描述，而应收到：

1. 全套 Deck Narrative 和视觉系统。
2. 有顺序的 PageSpec JSON。
3. 每页已批准资产文件引用。
4. 明确的本地数学层和可编辑性要求。
5. 页面级验收条件。
6. “不得增加说明页、不得合并页、不得重写学习顺序”的硬约束。

## 6. 渲染后质量门

### 确定性检查

- PPTX ZIP、`presentation.xml`、真实页数。
- 逐页渲染成功，无空白页。
- 文本框越界、元素重叠、图片低分辨率和裁切异常。
- 标题最小 35pt、关键文本建议 24pt 以上、正文不低于 18pt；低年级投影页优先更大。
- 普通文字对比度至少 4.5:1，大文字至少 3:1。
- 颜色不是唯一信息编码。
- 可编辑文字没有被烘焙进图片。

### 多模态 Critic

逐页回答：

1. 3 秒内能否看出本页让学生做什么？
2. 主视觉是否承担解释任务，还是纯装饰？
3. 是否存在“教案搬屏”、小卡片墙或视觉重点竞争？
4. 数学对象和数量是否正确？
5. 页面是否与上一页形成推进而非重复？
6. 教师是否能据此组织真实互动？

## 7. 页级返工

Critic 输出 `failed_page_ids` 和精确原因。返工路由只重做：

- 文字问题：修改 PageSpec 文案和排版，不重生图片。
- 主视觉问题：重生或重裁对应资产，不重做整套 PPT。
- 数学错误：修正本地数学层，禁止让生成图承担修复。
- 叙事重复：调整相邻 PageSpec，最多影响前后各一页。
- Provider 忽略设计稿：重发单页硬约束或改用本地确定性重建。

## 8. 教师演练门

最终批准前，用 5 个问题做人工或模拟演练：

- 教师在每页停留多久？
- 学生何时观察、说、做、写或讨论？
- 哪一页可能提前泄露答案？
- 哪一页需要黑屏、停顿或离开 PPT？
- 没有教案文本时，教师是否仍能理解本页组织动作？
