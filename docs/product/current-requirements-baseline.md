# ShanHaiEdu 当前需求与质量门禁基线

更新时间：2026-07-13

> 本文是当前 ShanHaiEdu 备课工作台开发与验收的唯一权威产品口径。旧阶段计划、历史报告、台账快照、MVP 过渡方案仅作为历史证据；若与本文冲突，以本文为准。

## 1. 当前产品目标

- ShanHaiEdu 是面向教师的智能公开课制作助手，不是固定线性流水线或 mock 展示页。教师可通过自然语言从当前可用能力切入、暂停、改道和局部返修；Main Agent 根据当前事实执行受控的 Observe、Plan、Guard、Act、Observe、Replan。
- 教师可见的需求、教案、PPT、视频和最终交付仍有明确里程碑与质量依赖，但里程碑顺序不能被实现成强制从头执行的固定 DAG；已有合格产物应复用，缺少硬前置时只补最小必要输入。
- 系统必须围绕教师真实备课交付：结构化教案、逐页 PPT 设计稿、真实 PPTX、课堂视觉图、导入视频和最终材料包。
- 可以保留草稿、测试运行时和中间状态，但不得把 mock、placeholder、deterministic draft、文本 fallback 或未达标产物包装成真实完成。

## 2. 当前已确认问题

- 断点任务需要能继续执行，失败节点不能让整个项目卡死。
- 同一项目同一时间只能有一个有效写者和一条有效生成主线；两个受邀教师在不同项目中的对话和生成任务必须允许受控并发，且状态、强度、费用任务和产物完全隔离。
- 教案必须结构化，不能用普通教案或弱草稿搪塞真实处理需求。
- PPT 设计稿必须逐页完整，不能用“第 4-8 页”“第 9-12 页”这种范围合并描述代替逐页设计。
- PPTX 必须真实图文并茂，不能用文本大纲 fallback 冒充。
- 视频生成前必须先产出完整视频主题、脚本、资产图和分镜提示词，再进入视频生成。
- 图片、视频、PPTX、最终包必须真实生成、真实下载、真实校验。
- 教师可见界面不能暴露 `schema`、`provider`、`node_id`、`storage`、`debug`、`local path`、`capabilityId`、`runtimeKind`、`providerStatus`、`placeholder` 等工程词。
- 下载按钮和状态文案必须基于真实校验结果，不能根据目标页数或任务成功状态写死。

## 3. 教案质量门禁

教案必须是结构化产物，至少包含：课程目标、教材依据、学情分析、重点与难点、教学流程、教师活动与学生活动、板书设计、评价方式、课堂风险与处理策略、可被 PPT/图片/视频继续复用的结构字段。

不合格情形：只有泛泛文本、无结构字段、无教材依据，或直接把草稿标成真实教案完成。

## 4. PPT 设计稿质量门禁

PPT 设计稿必须为逐页设计。每一页都必须独立写明：底图、元素、文字、排版、教学动作、视觉重点。

不合格情形：使用范围合并描述，例如“第 4-8 页”“第 9-12 页”；只写页面清单；只有文字大纲，没有视觉与教学动作。

## 5. PPTX 真实交付门禁

- PPTX 下载文件必须是真实 `.pptx` zip。
- 必须包含 `ppt/presentation.xml`。
- 必须统计 `ppt/slides/slide*.xml` 得到真实页数。
- 真实页数必须等于目标页数，才能标记为可验收 PPTX。
- 页数不符时必须标记失败或未达标，不得保存为“真实 N 页 PPTX”。
- 目标页数、设计稿页数、文件名中的页数都不能代替真实 slideCount。

当前已修正的典型问题：曾出现“页面写真实 12 页，但下载文件只有 2 页”的错误。后续必须以真实 slideCount 为准。

## 6. 视频交付门禁

视频生成前必须先形成结构化前置产物：视频主题、视频脚本、资产图、分镜提示词、每镜头时长、画面动作、旁白或字幕、课堂边界约束。

课程锚点是独立创意短片与课程任务之间的最小回接，不是全片世界观、受众年龄、儿童角色、教室场景或教材内容复述。视频必须先在脱离教材、PPT、教师讲解和课堂角色时仍然成立，再以唯一的触发事件、课堂第一问或交接瞬间回到课程；“面向小学生观看”不得被推导为“必须讲小学生在课堂中的活动”。

课程锚点、独立创意和镜头叙事的首轮审查、阻塞与返修决策必须由产品内部 Main Agent 调用独立的 `delivery_critic.review(domain="video", stage="course_anchor")` Agent Tool 完成，并留下计划、Tool调用、CriticReport、QualityDecision、Observation和Replan证据。Video Director或Main Agent自己的自评不能解锁真实媒体Tool；有效CriticReport必须绑定projectId、IntentEpoch、创意版本digest、rubricVersion和独立Critic Executor来源。产品外部Codex不得替智能体选择创意、批准锚点或决定返修镜头后，再把结果记为产品能力。

课程锚点Critic至少执行六个硬门：脱离教材仍可理解、去掉回接仍值得观看、不是教材/PPT复刻、全片只有一个最小回接、受众年龄没有被扩张成故事世界限制、不提前泄露答案。儿童角色本身不禁止，只有“因为面向小学生所以必须儿童主角”才阻塞。教室仅在最终回接是一个明确可通过的正例，但不是唯一允许情形；只要教室服务于脱离教材仍成立的独立叙事，而不是课堂教学活动或教材复刻，也可以通过。任一硬门失败或证据不足时，真实图片、视频、拼接和最终包Tool调用次数必须为0。

不合格情形：直接拿一段笼统描述调用视频生成；没有分镜和资产约束；把本地占位视频、测试 smoke 或非目标视频冒充为最终导入视频。

真实图片、视频和整包生成不作为日常编排调试手段。V1 前段优先使用确定性夹具、Provider adapter测试和可审计状态验证 Main Agent 的协调能力；只有产品内编排、HumanGate、Quality Gate、双用户隔离和恢复门通过后，才在阶段收尾执行一次受控真实交付。该真实交付包由产品外部验收者进行黑盒审核，用审核发现反推 Agent、Tool、Prompt、Rubric或门禁缺陷，但外部验收者不介入包内编排。

## 7. 断点续跑与并发门禁

- 后端必须能识别当前项目的可继续节点。
- 失败节点必须保留可重试状态和失败原因。
- 同一项目同一时间只能推进一个有效写生成任务；不同项目允许并发或按Provider容量进入可恢复队列。
- 跨标签页、快速重复点击或多项目并发不能导致状态错乱。
- 前端按钮禁用只能作为体验优化，不能代替后端锁或队列。

## 8. 教师界面门禁

- 教师界面只展示可理解的教学语言和交付状态。
- 禁止展示工程字段、调试字段、存储路径、provider 名称、token、签名 URL 或本地绝对路径。
- 如果产物未达标，界面必须明确说明“未达标原因”和“下一步建议”，不能显示成功态。
- 助手不得承诺系统实际无法执行的交互，例如声称“回复一句即可继续”但后端只承认按钮附带的隐藏确认字段。
- 推荐按钮只能是快捷入口，不能成为唯一有效入口；教师使用自然语言继续、改道、补充或取消时，系统必须结合当前活动计划理解。
- 多个候选分支无法唯一判断时，必须提出一个具体消歧问题；不得把语义歧义错误显示为“没有有效确认”。
- 教师实质修改推荐文本或改换任务时，旧确认动作必须失效；不得用旧 actionId 授权新意图。
- 改道后旧计划必须退出活动状态，历史失败保留审计但不得污染新分支。

## 9. 内测反馈上线门禁

- 正式邀请内测用户使用前，必须提供一个真实可保存的反馈入口，不能继续显示“反馈入口暂未开放”。
- 反馈入口必须采用引导式分类，而不是只给一个空白文本框让用户自行组织问题。
- 第一版至少支持：页面/视觉问题、功能异常、内容结果问题、操作不清楚、功能建议、性能问题、其他。
- 每个分类提供可点击的预制反馈提示，帮助用户说明“发生了什么、期待怎样、影响程度”。
- 必须支持图片附件，至少包括文件选择上传和从剪贴板直接粘贴图片；粘贴后必须显示预览并允许删除。
- 反馈提交后必须保存到服务端可持久化位置，并记录分类、描述、附件、页面上下文、项目上下文、提交时间和应用版本；不得只保存在浏览器内存或当前页面状态。
- 项目维护者必须有受控的查看和导出方式，能够按时间、分类和影响程度整理内测反馈；反馈内容不得公开暴露给普通用户。
- 公网内测必须使用密码认证和邀请制账号；不得因为认证配置缺失回退到 local 模式，也不得向任意公网访问者开放自助注册。
- 反馈图片、用户描述和上下文不得放入公开静态目录，不得在教师界面回显服务端路径、账号、token 或调试信息。
- 提交成功必须给出明确回执；提交失败必须保留用户已经填写的内容和图片，允许重试。

### 9.1 多用户与用户管理门禁

- M67 后的下一阶段必须把现有密码认证、邀请脚本和管理员角色扩展为真实多用户能力，不能只依赖单个 local 用户或手工改库。
- 项目、对话、产物和反馈必须按 owner / membership 做服务端授权；前端隐藏不能代替权限检查。
- 管理员可以受控邀请、停用、启用、调整允许范围内的角色和重置凭据；普通教师不能读取用户列表、管理他人账号或提升自身权限。
- 账号停用、角色变更和凭据重置必须撤销既有会话并保留脱敏审计，不回显明文密码、会话 token 或内部路径。
- 公开注册继续保持关闭；组织架构、学校多租户、SSO 和复杂 RBAC 不自动进入下一阶段范围。

## 10. 旧文档处理规则

- 旧阶段计划、历史报告、API 台账快照、MVP 早期方案只保留为历史证据。
- 旧文档若写有“可降级产物”“deterministic fallback 跑完整链路”“页数不校验直接交付”“第一版文本大纲转 PPTX”等口径，不能作为当前开发依据。
- 后续开发、测试、验收和需求讨论必须先对照本文。

## 11. 推荐修复顺序

分档关系：

```text
接管门槛：关闭 M72 按需视频脚本和 M77/M78 owner 写路径浏览器差距
第一档：V1 执行安全、节点合同、质量决策与受控 ReAct
核心交付：PPT Quality、视频 Full Intro、最终包真伪与局部返修
上线门槛：产品内一次真实交付E2E、两用户隔离与并发、目标服务器恢复门、至少一名真实教师签收
第二档：竞品研究衍生增强，现阶段不实现
```

推荐顺序：

1. 先完成 V1 接管对账，关闭 M72 按需视频脚本和 M77/M78 owner 写路径浏览器差距；保持既有 `v1` 标签不可变。
2. 建立执行身份、单项目写租约、意图版本栅栏、幂等和 Provider 任务恢复，再接入可执行 Node Contract、Validator、Critic 与确定性质量决策。
3. 分别闭环 PPT Quality 与视频 Full Intro：优先证明产品内部 Main Agent 能自主调用 Tool、Critic、HumanGate 和 Quality Gate；PPT 必须支持真实 12 页 PPTX，视频必须逐镜头生成、恢复、审查和返修，课程锚点不得把独立创意收缩成小学课堂活动。
4. 完成教案、PPT、视觉图、视频与 `ClassroomRunSpec` 的版本一致性检查，从真实目录反向生成最终包和 manifest。
5. 编排、双用户和恢复门全部通过后，在收尾阶段执行一次产品内真实交付 E2E；由外部验收者审核成包效果并定位链路缺陷，修正后再执行必要的定点复验。目标服务器重启/回滚/备份恢复和至少一名真实教师课堂签收通过后，才开放邀请制 V1。
6. 完成一轮邀请内测后，再根据反馈决定是否启动 Studio 编辑、资源库、共享资产池、模板体系和 LMS 等第二档需求。
