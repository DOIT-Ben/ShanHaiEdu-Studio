# 主线：MVP E2E Verification

## 1. 主线目标

证明 ShanHaiEdu Media Workbench MVP 不是看起来能用，而是真的能在本地跑通完整工作流。

## 2. 为什么单独成主线

前端、后端、runtime 都可能各自通过，但集成后仍可能失败。E2E 主线负责独立验证真实用户路径、刷新恢复、并发隔离和无工程词暴露。

## 3. 可复用方案

- Playwright：浏览器端流程验证。
- API smoke：验证后端合同。
- DeterministicRuntime：稳定端到端。
- 构建检查：`npm run build`。

## 4. 职责边界

负责：

- E2E 测试计划。
- 浏览器流程测试。
- 并发试运行。
- UI 红线扫描。
- 构建和回归验收。

不负责：

- 实现前端功能。
- 实现后端业务。
- 写 prompt。
- 修复非本主线 bug，除非是测试代码问题。

## 5. 长期阶段

### 阶段 1：验收合同与测试计划

交付：

- E2E test plan。
- 核心用户路径定义。
- 红线词扫描清单。
- 并发试运行方案。

验收：

- 每条测试都能对应需求基线。
- 每个失败都能定位到前端、后端或 runtime 主线。

### 阶段 2：Deterministic E2E

交付：

- 新建项目。
- 输入一句话。
- 生成需求规格。
- 节点显示。
- 复制/作为输入/确认。
- 刷新恢复。

验收：

- 无 API key 也能稳定跑通。
- 浏览器截图覆盖桌面和窄屏。

### 阶段 3：多项目与并发隔离

交付：

- 两个项目并行。
- 两个浏览器会话。
- artifact version guard 验证。

验收：

- 项目不串。
- 消息不串。
- 节点状态不串。

### 阶段 4：OpenAI Runtime Smoke

交付：

- 显式配置环境后运行真实生成 smoke。
- 记录耗时、失败、输出质量。

验收：

- 至少真实生成需求规格和一个教学产物。
- 失败时前端恢复态正确。

## 6. 测试策略

- 阶段内可以最小 smoke。
- 阶段结束集中运行完整 E2E。
- 不用碎片化小测代替验收。
- 重型测试限制 worker，避免资源失控。

## 7. 集成输入输出

输入：

- 前端可运行 URL。
- 后端 API。
- deterministic runtime。
- 可选 OpenAI runtime。

输出：

- 测试报告。
- 截图或浏览器证据。
- 失败归因。
- 阶段验收结论。

## 8. 阻塞条件

- 没有可运行 dev server，不做浏览器验收。
- 没有 deterministic runtime，不做稳定 E2E 封板。
- 没有项目隔离 API，不做并发验收结论。
