# ShanHaiEdu 新需求整合：模型解放、上下文记忆与渐进门禁

日期：2026-07-09

状态：需求整合草案，待确认后并入当前需求基线

## 1. 新主线一句话

ShanHaiEdu 应从“线性节点流水线”升级为“模型主控的备课 Agent OS”：先释放大模型的理解、规划和自修复能力，再对真实交付、安全和质量底线逐步建立门禁。

## 2. 新增核心需求

补充故障反推来源：`docs\product\2026-07-09-conversation-failure-derived-requirements.md` 记录了 15:00-15:16 本地验收对话中反复卡在 PPTX 与视频资产图的真实卡点，以下需求应与该复盘一起纳入 M61。

### R1. 模型主控优先

系统应让模型作为主控 Agent 理解教师目标、当前状态和可用工具，而不是只让模型填充固定节点文本。

要求：

- 模型必须看到结构化项目状态，而不是只有最近聊天。
- 模型可以选择追问、继续、回退、重试、换工具或降级。
- 除安全和真实交付底线外，不使用关键词或固定顺序过度限制模型。

### R2. 结构化上下文记忆

每轮模型调用前必须构建结构化工作记忆。

最小字段：

```json
{
  "goal": "当前教师目标",
  "currentStage": "当前工作阶段",
  "lastUserAction": "教师最近动作",
  "completedSteps": [],
  "activeNode": {},
  "artifacts": [],
  "pendingDecisions": [],
  "toolHealth": {},
  "fallbackOptions": [],
  "risks": []
}
```

### R3. 任意节点可回退

教师应能回退到关键节点，例如需求、教案、PPT 大纲、PPT 设计稿、PPTX、视频脚本、分镜和最终包。

要求：

- 产物必须版本化。
- 节点必须记录输入来源和输出产物。
- 回退后下游节点应标记为受影响或需要重算，而不是静默沿用旧结果。

### R4. 动态节点状态

节点状态不能只靠 `not_started / needs_review / approved / failed`。

建议扩展语义：

- `usableAsInput`：是否可作为下游输入。
- `requiresManualReview`：是否仍建议人工复核。
- `confirmationMode`：显式确认、点击继续隐式确认、系统自动确认。
- `qualitySignals`：页数、结构完整性、校验结果。
- `lastError`：失败类型和可重试性。
- `rollbackTarget`：可回退版本。

### R5. 动态工具能力

工具应从静态能力注册表升级为动态 runtime。

每个工具至少暴露：

- 是否可用。
- 输入是否就绪。
- 最近失败类型。
- 可重试策略。
- fallback 工具。
- 输出校验方式。

### R6. 工具 observation 回到模型

工具执行失败后，不能只由服务端直接报错结束。应把工具结果作为 observation 交给模型，让模型决定下一步。

示例：

```json
{
  "tool": "coze_ppt",
  "status": "failed",
  "errorType": "pptx_download_failed",
  "retryable": true,
  "fallbacks": ["local_pptxgen", "html_to_pptx", "regenerate_ppt_design"]
}
```

### R7. PPTX 多路径 fallback

PPTX 是核心交付物，不能只有一个外部工具。

建议顺序：

1. Coze 或外部高质量 PPTX 生成。
2. 本地 `pptxgenjs` 基于逐页设计稿生成基础可编辑 PPTX。
3. HTML/SVG 页面转 PPTX。
4. 导出 Markdown/图片包作为临时可用材料，但不得标成真实 PPTX。
5. 重新生成更适合 PPTX 的设计稿。

### R8. 小学范围改成默认引导，不做硬锁

产品默认面向小学公开课，但不应使用关键词硬编码锁死模型。

推荐口径：

- 默认按小学一至六年级优化。
- 模型不得主动跑到初中或高中例子。
- 若用户明确提出非小学内容，系统提示“超出默认优化范围”，可继续低保障规划或请用户改回小学。

### R9. 门禁后置且分级

门禁分三类：

| 类型 | 示例 | 策略 |
| --- | --- | --- |
| 硬门禁 | 真实 PPTX、slideCount、文件有效性、安全隐私 | 必须强制 |
| 软门禁 | 节点顺序、用户确认、默认小学范围 | 允许模型解释、追问、继续或降级 |
| 诊断门禁 | 工具失败、配置异常、外部服务不可用 | 记录内部错误，教师界面给可理解建议 |

### R10. pending plan 失败释放与卡点检测

挂起计划不能在工具失败后继续无条件吸收后续输入。若同一 pending plan 或同一工具连续失败，系统必须触发卡点检测，转入重规划、fallback 或询问教师，而不是重复执行原工具。

### R11. 上下文必须基于事实而非计划

模型上下文中的 `upstreamAvailable`、已完成节点和可用产物必须来自真实 artifact 与任务结果，不得仅由 deliveryPlan 顺序推导。失败、pending 或不存在的产物不能被写成可用上游。

### R12. 工具失败必须产生 observation

工具失败后必须生成结构化 observation，并交回模型进行二次决策。服务端不应直接用固定失败文案终止任务，除非涉及安全或不可恢复错误。

## 3. 对旧口径的修订建议

旧口径强调“线性工作台”和“节点门禁”。新口径应调整为：

```text
ShanHaiEdu 是模型主控的 AI 备课媒体工作台。系统用结构化记忆和动态工具状态帮助模型持续理解当前进度、可用产物、失败原因和下一步选项。真实交付、安全与质量底线保持硬门禁；理解、规划、节点推进和失败恢复采用软约束与模型自修复。
```

## 4. 不再推荐的实现方式

- 只传最近 8 条消息给模型。
- 只传 artifact kind，不传 artifact 状态、版本、可用性和失败记录。
- 用关键词列表粗暴拦截模型理解。
- 工具失败后直接报错给用户，不让模型重新规划。
- 单一路径依赖外部 PPTX 服务。
- 固定节点顺序卡死模型跳步、回退和降级。

## 5. 下一步验收标准

- 模型能准确说明当前做到哪里。
- 模型能引用最新产物状态和失败原因。
- “继续下一步”能结合上下文解释为继续、确认、重试或询问。
- PPTX 工具失败后至少给出一个真实 fallback 选项。
- 用户能回退到任意关键节点。
- 真实交付依然不允许占位、伪文件或错误页数冒充成功。
