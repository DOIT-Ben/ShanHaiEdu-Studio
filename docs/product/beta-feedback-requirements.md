# 内测反馈中心产品需求

更新时间：2026-07-10

状态：已接受；上线内测前必须完成。

## 1. 决策

在邀请内测用户使用前，先交付一个简单但真实可用的反馈中心。第一版不追求复杂工单系统和运营后台，重点是让用户在任何页面都能快速、被引导地提交问题，并把文字与截图可靠保存下来。

## 2. 用户问题

当前消息点赞/点踩只显示“已在本页记下，反馈入口暂未开放”，没有真实保存。用户遇到页面不美观、流程卡住、结果不对或缺少功能时，无法把现场信息传回项目团队。

如果先上线而没有反馈闭环，将出现：

- 用户发现问题但不知道从哪里反馈。
- 用户只说“这里不好用”，缺少可定位的信息。
- 截图需要通过外部聊天工具发送，反馈与页面上下文分离。
- 团队无法按问题类型汇总内测结果。

## 3. 入口

第一版统一复用一个 `FeedbackDialog` / `FeedbackCenter`，入口可来自：

1. 全局工作台固定反馈入口，任何页面可见但不抢主视觉。
2. 左下角头像菜单中的“反馈”。
3. Assistant 消息点赞/点踩后的“补充反馈”，自动带入消息上下文。

所有入口必须进入同一个表单和同一套保存接口，不能各自实现一套反馈逻辑。

项目维护者需要一个受控的内部查看方式：第一版可以是受保护的只读列表/详情页，或可验证的管理端导出接口；不要求建设完整工单后台，但不能让反馈只能写入、无法整理。

## 4. 引导式分类

用户打开反馈后，先看到分类卡片或单选项：

| 分类 ID | 用户可见名称 | 预制引导 |
|---|---|---|
| `visual` | 页面不好看 / 排版不舒服 | 哪个位置看起来不协调？是文字、颜色、间距、布局还是移动端显示？ |
| `bug` | 功能异常 / 点了没反应 | 你刚才做了什么？哪个按钮或步骤没有按预期工作？ |
| `content_quality` | 生成结果不对 | 哪部分内容不准确、不完整或不适合课堂？你期待怎样的结果？ |
| `confusing` | 不知道怎么操作 | 你停在哪一步？你原本以为下一步应该是什么？ |
| `feature_request` | 希望增加功能 | 你希望系统帮你完成什么任务？现在你用什么方式替代？ |
| `performance` | 加载慢 / 生成慢 / 卡顿 | 哪一步最慢？大约等待了多久？是否最终成功？ |
| `other` | 其他反馈 | 请描述你遇到的情况和期待的改进。 |

每个分类还必须提供 2-3 个可点击的预制提示 chip。例如：

- 页面不好看：`文字层级不清楚`、`间距或对齐不舒服`、`手机上显示有问题`。
- 功能异常：`按钮没有反应`、`操作后结果不对`、`页面卡住无法继续`。
- 内容结果：`内容不准确`、`结果不完整`、`不适合真实课堂`。
- 操作不清楚：`不知道下一步做什么`、`提示看不懂`、`找不到需要的入口`。
- 功能建议：`希望减少操作步骤`、`希望增加新的产物`、`希望支持新的材料类型`。
- 性能问题：`打开页面很慢`、`生成等待太久`、`操作时明显卡顿`。

点击 chip 时，将对应短句追加到描述草稿末尾；如果草稿已经包含该短句则不重复添加。chip 只能帮助起稿，不能自动提交，也不能覆盖用户已经输入的内容。分类选择后描述框同时显示对应 placeholder。

## 5. 表单字段

### 用户填写

- 反馈分类：必填。
- 反馈描述：必填，支持多行。
- 图片附件：选填，第一版最多 5 张。
- 影响程度：选填，`一般` / `影响使用` / `完全无法继续`。

### 系统自动附带

- 当前页面路由。
- 当前项目 ID；没有项目时为空。
- 关联消息 ID；从消息反馈入口进入时记录。
- 应用版本或构建版本。
- 浏览器和视口基础信息。
- 提交时间。
- 内测用户标识；只使用系统已有身份，不要求用户重复填写敏感信息。

第一版不自动采集完整对话、密钥、请求头、本地路径或用户未授权的屏幕内容。

## 6. 图片要求

图片能力是上线门槛，不得延期：

1. 支持点击选择图片文件。
2. 支持将截图复制后，在反馈弹窗内直接 `Ctrl+V` / `Cmd+V` 粘贴。
3. 从 `ClipboardEvent.clipboardData.items` 读取 `image/*`，生成待上传附件。
4. 粘贴或选择后立即显示缩略图、文件大小和删除按钮。
5. 提交前校验类型、单张大小、总数量；失败时明确告诉用户原因。
6. 图片不得以 base64 长字符串直接写入反馈 JSON；元数据和文件内容分开保存。
7. 提交失败后保留本地预览和描述，允许重试。

## 7. 保存与存储

第一版固定采用：

```text
Feedback metadata -> 当前生产 SQLite 数据库中的 Feedback 记录
Feedback images   -> ARTIFACT_STORAGE_ROOT\feedback\<feedbackId>\<attachmentId>.<ext>
```

要求：

- 数据库继续使用当前 `DATABASE_URL`，生产值必须指向服务器共享持久数据卷中的 SQLite 文件。
- 图片复用 `ARTIFACT_STORAGE_ROOT`，固定使用其 `feedback` 子目录；生产值必须指向服务器共享持久素材卷。
- 服务器部署约定：release 目录之外保留 `/opt/shanhai-edu-studio/shared/data/` 和 `/opt/shanhai-edu-studio/shared/artifact-storage/`，应用更新不得覆盖这两个目录。
- `DATABASE_URL` 与 `ARTIFACT_STORAGE_ROOT` 只写服务器 `.env`，不提交真实值，不沿用本机或兄弟工作区路径。
- 文件使用服务端生成的不可猜测 ID；数据库只保存受控对象引用。
- 图片不进入公开 `public\` 目录。
- 用户提交后返回反馈编号，便于后续追踪。
- 维护者可以按时间、分类、影响程度查看列表与详情、下载附件并导出反馈，附件访问必须经过受控服务端入口。

通过 `FeedbackStorage` 抽象保留后续迁移到对象存储的边界，但本轮不再等待对象存储选型。

备份与复验：

- 每次发布前备份生产 SQLite 文件和 `feedback` 附件目录。
- 部署重启后读取一条测试反馈和其附件，证明数据仍存在。
- 回滚 release 时继续挂载同一 shared 数据目录，不回滚或覆盖反馈数据。

## 8. 安全与权限

- 提交、读取和导出接口复用现有认证与 CSRF 门禁。
- 客户端传入的 `projectId`、`messageId` 必须在服务端验证当前 actor 有权访问，不能直接采信。
- 图片白名单只允许服务端真实解码通过的 PNG、JPEG、WebP；拒绝 SVG、HTML、脚本和仅伪造扩展名的文件。
- 硬限制：最多 5 张；单张不超过 10 MiB；全部图片二进制合计不超过 25 MiB；单张宽高均不超过 8192 px；单张解码像素不超过 40,000,000。
- 提交接口需要 `idempotencyKey`；`createdBy` 必须来自认证 actor 且不可为空，唯一约束作用域为 `createdBy + idempotencyKey`，并保存规范化请求指纹。同一用户同键同内容返回原反馈；同键不同内容返回 409，不得覆盖原反馈。
- 如果同时传入 `projectId` 和 `messageId`，服务端必须验证 message 属于该 project，且 actor 有权访问该 project；不能只分别检查两个 ID 是否存在。
- 文件与 SQLite 无法组成单一事务，采用 staged submission：先将附件写入同一存储根下的 staging 目录并完成真实解码，再创建 `processing` 反馈记录，原子重命名为最终目录后将记录改为 `submitted`。
- 应用启动或维护任务必须对账长期停留的 `processing` 记录、staging 目录和无数据库引用的最终目录；能恢复则恢复，不能恢复则标记失败并清理。只有 `submitted` 能向用户返回成功。
- 只有 `actor.isAdmin === true` 的密码认证管理员可以查看列表、详情、下载附件和导出脱敏 CSV/JSON；普通教师只能提交，不能读取他人反馈。CSV 必须使用结构化 writer 做转义，并中和 `= + - @`、制表符和回车开头的用户单元格，防止表格公式注入。
- local 模式不提供反馈总览；维护者通过密码认证管理员账号进入受控查看入口。
- 管理员账号创建和凭据配置属于部署步骤，凭据不得进入代码、文档或提交。
- 公网内测固定使用密码认证和邀请制账号：生产同时设置服务端/客户端密码认证开关，并关闭公开自助注册。内测账号由维护者通过受控 bootstrap/invite 流程创建。

## 9. 提交状态

- `draft`：正在填写，前端本地状态。
- `submitting`：正在上传和保存，禁止重复提交。
- `submitted`：服务端已确认保存，显示反馈编号。
- `failed`：保存失败，保留输入并允许重试。

成功文案示例：

```text
反馈已收到，编号 FB-20260710-XXXX。谢谢你帮助我们改进。
```

## 10. 第一版不做

- 不做复杂工单流转、客服 SLA、邮件通知和评论线程。
- 不做复杂运营后台；只提供受控的反馈查看/导出能力。
- 不做公开反馈广场。
- 不自动上传整页截图或录屏。
- 不接入竞品研究衍生的大型编辑器、模板市场和 LMS 功能。

## 11. 验收标准

1. 任意主要工作台页面都能打开反馈入口。
2. 必须选择分类后才能提交；每类显示对应 placeholder 和 2-3 个可点击提示 chip，点击只追加草稿、不覆盖、不自动提交。
3. 可以选择图片，也可以把截图直接粘贴到反馈弹窗。
4. 图片可预览、删除，非法类型和超限有教师可理解提示。
5. 提交成功后，服务端数据库和配置化存储中存在对应记录和附件。
6. 刷新页面后反馈仍存在，证明不是前端临时状态。
7. 失败时文字和图片不丢失，可直接重试。
8. 页面、接口和日志不泄露存储路径、token、账号和内部调试字段。
9. 项目维护者可以通过受控入口按时间、分类和影响程度查看列表与详情、下载附件并导出脱敏 CSV/JSON，普通用户不能查看他人反馈。
10. 非管理员读取/导出返回拒绝；跨项目伪造上下文、伪造图片类型、SVG、超限图片均被服务端拒绝。
11. 网络超时后使用相同幂等键重试只产生一条反馈；附件失败不产生孤儿文件。
12. 生产进程重启后，数据库记录和附件仍可由管理员读取。
13. 同时传入跨项目的 `projectId + messageId` 时提交被拒绝。
14. 同一用户同幂等键同内容返回同一反馈；同键不同内容返回 409。
15. 模拟 staging 写入后崩溃、数据库记录后崩溃、重命名后崩溃和清理失败，重启对账后不存在错误成功态和永久孤儿附件。
16. 公网未登录访问工作台 API 返回 401；关闭公开注册后，非邀请用户不能创建账号。
