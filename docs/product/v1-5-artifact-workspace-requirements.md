# V1.5 当前成果工作区需求

日期：2026-07-13

状态：accepted / required for V1.5

关联需求：RQ-015、RQ-016、RQ-028、RQ-032、RQ-034、RQ-036、RQ-037

## 1. 版本目标

V1.5 必须完成 ShanHaiEdu 教师工作台的信息架构升级：从“对话旁常驻一条产物节点链”，转为“对话控制台 + 当前成果工作区 + 全部成果抽屉”。

本需求解决的不是普通视觉美化，而是教师如何理解、查看、修改和批准智能体成果的问题。系统仍保留真实 Artifact、版本、依赖、HumanGate 和 Quality Gate，但不再要求教师通过线性节点隐喻理解智能体能力。

V1.5 完成口径：

```text
对话控制台：交流、指挥、解释、确认和自然语言修改
当前成果工作区：阅读、编辑、审查当前主要成果
全部成果抽屉：查找项目内全部历史成果和版本
当前制作计划：展示智能体准备做什么和正在做什么
```

## 2. 核心概念

以下概念必须分离：

- `Plan`：Main Agent 为达成当前意图准备执行的业务动作。
- `Artifact`：教师需要阅读、修改、确认或下载的业务成果。
- `Quality State`：成果经过确定性检查、独立 Critic 或 QualityDecision 后的质量结论。
- `Artifact Workspace`：当前主要成果的教师可见阅读与编辑投影。

需求分析、教案、PPT 大纲、逐页设计稿、视频脚本和分镜都是 Artifact，不是执行 Plan。Artifact 之间可以存在依赖，但前端不得把依赖强制呈现为只能从头到尾执行的固定链路。

## 3. 产品决策

### 3.1 下线常驻糖葫芦

- V1.5 桌面主界面不再常驻 `ArtifactRail` 或任何按节点/能力纵向排列的糖葫芦式导航。
- RQ-016 已完成的成果分组、状态聚合、权限、空态和抽屉能力继续保留，统一进入“全部成果”入口。
- 本决策只替代 RQ-016 的常驻视觉形态，不删除 Artifact 数据模型、状态、版本、依赖、下载、确认或审计能力。
- 当前制作计划继续按 RQ-028 独立展示，不由成果导航代替。

### 3.2 当前成果工作区

- 桌面端在主要成果生成、重要版本更新、教师明确要求查看或 HumanGate 需要审查时，按策略自动打开右侧成果工作区。
- 默认宽度建议为可用工作区的 48%-55%，教师可以拖动到约 38%-70%；最终数值在视觉阶段按现有布局验证。
- 对话区与成果区独立滚动，输入框始终可用；打开成果区时不得把对话压缩到不可读。
- 工作区标题栏至少提供：成果标题、保存状态、版本、关闭、完整详情；具体成果按权限提供下载、反馈、确认和版本比较。
- 文档型成果支持结构化富文本阅读与编辑；图片、PPT、视频和最终包使用各自适合的查看器，但共享同一个工作区外壳。

### 3.3 对话与成果分工

- Main Agent 在对话中说明：理解了什么、生成或修改了什么、质量状态如何、下一步是什么。
- V1.1由assistant-ui的`artifact-ref` Part打开现有阅读面板；V1.3-V1.5的BlockNote与各类成果查看器继续由本文件约束，不能把assistant-ui消息组件当作成果编辑真源。
- 大段教案、PPT 大纲、逐页设计和视频脚本不再完整重复到聊天气泡；聊天只保留摘要、关键变化和“查看成果”入口。
- 用户要求“只改第 3 页”“补充这一段”“不要重写全文”时，应绑定当前成果及 locator，生成局部 Patch 或新版本，而不是把整份文档重新塞回对话。
- 当前成果不明确时，Main Agent 先根据活动意图、最近操作和显式引用判断；仍有多个候选时只提出一个具体消歧问题。

## 4. Main Agent 与呈现合同

Main Agent 决定业务意图和主要成果；前端不能通过标题关键词或消息长度猜测要打开什么。

建议在 Turn 结果中提供等价于以下语义的呈现指令，正式字段由合同阶段冻结：

```text
assistantMessage
primaryArtifactRef
presentationAction: none | open | focus | update | keep | close
focusLocator
reason
```

- `primaryArtifactRef` 必须绑定真实 artifact/version/digest。
- `focusLocator` 可以定位文档章节、PPT 页、视频镜头、时间段或最终包文件。
- `presentationAction` 只控制教师可见投影，不能批准 HumanGate、改变 QualityDecision 或替代 Tool 调用。
- 刷新、重新进入项目和跨标签页后，应从持久化状态恢复当前成果、版本、面板开关和必要焦点；面板宽度可以按用户与设备保存为偏好。

## 5. 自动打开策略

应自动打开或聚焦：

- 新生成的需求理解稿、教案、PPT 大纲、逐页设计稿、视频脚本和分镜。
- 当前主要成果的重要版本更新。
- 需要教师确认的样张、内容稿、返修结果或最终交付。
- 教师明确说“打开、查看、修改、继续改刚才那份”。

不应抢占打开：

- 普通问答、问候和能力说明。
- 工具执行事件、后台日志和内部 QA 中间过程。
- 与当前意图无关的后台成果更新。
- 教师正在输入或编辑另一成果时的低优先级通知；此时只提示有新成果可查看。

## 6. 编辑、版本与自然语言修改

- 直接编辑必须具备 `dirty -> saving -> saved | failed` 状态，保存失败不得丢失教师输入。
- 已确认或进入下游生产的成果不能静默原地覆盖；实质修改必须生成新版本并传播 stale 影响。
- 自然语言修改与面板直接编辑写入同一版本体系，不形成两个互不一致的事实源。
- 教师选中文档段落、PPT 页或视频镜头后发出的修改，默认绑定该 locator；扩大影响范围时由 Main Agent 解释并在必要时确认。
- 至少支持撤销当前未保存修改、查看最近版本、比较关键变化和恢复到允许恢复的版本。
- 多用户同时编辑同一成果时不得最后写入静默覆盖；V1.5 至少提供版本冲突检测和重新加载/另存版本路径。

## 7. 门禁与质量呈现

产品应表现为“前台自由、后台受控”：

- 普通问答、内容分析、草稿、大纲和本地局部编辑不新增 HumanGate。
- 真实媒体费用、外部写入、不可逆动作、显著提高强度和正式批准仍执行必要门禁。
- Quality Gate 默认由系统内部执行；只有失败、需教师判断或成果可批准时，才在当前成果工作区呈现教师可理解的结论。
- V1.2 阶段 QA 接入当前成果工作区：finding 必须定位到章节、页、镜头或时间段，返修后在原位置显示复验结论。
- 不向教师展示 schema、provider、nodeId、Tool 参数、Prompt、思维链或原始日志。

## 8. 响应式要求

- 桌面使用可调分栏；关闭成果工作区后，对话恢复完整宽度。
- 窄屏和移动端不强制双栏，使用“对话 / 当前成果”单面板切换，并提供明确返回动作。
- 移动端编辑工具栏允许横向安全滚动或收纳进菜单，不得裁切主要操作。
- 打开键盘、编辑长文、查看表格和切换成果时不得遮挡输入、保存状态或返回操作。

## 9. 分阶段落地

| 版本 | 必须完成 |
|---|---|
| V1 收尾 | RQ-036 回复结构、阶段摘要和主动引导；不在此阶段强推完整编辑器 |
| V1.1 | 下线桌面常驻糖葫芦；保留成果抽屉；主要成果可自动打开现有阅读面板；避免聊天重复完整正文 |
| V1.2 | 阶段 QA、finding、返修和复验结果进入当前成果工作区 |
| V1.3-V1.4 | 富文本编辑、保存状态、版本、局部 Patch、自然语言修改当前章节和冲突处理 |
| V1.5 | 完整成果工作区总验收，旧常驻 Rail 不再作为正式产品入口 |

阶段允许重新排序，但不得把 V1.5 完整验收缩减为“隐藏一个 Rail”或“把侧栏加宽”。

## 10. V1.5 验收门

1. 桌面主界面不再常驻糖葫芦 Rail，全部成果仍可在统一入口找到。
2. Main Agent 能通过真实引用指定当前主要成果，前端不靠文本猜测。
3. 需求稿、教案、PPT 大纲/逐页设计、视频脚本/分镜生成后按策略自动打开。
4. 对话只显示可靠摘要，完整内容在成果工作区阅读和编辑。
5. 直接编辑、保存失败恢复、版本、撤销、比较和冲突处理具备真实用户路径。
6. 自然语言可以修改当前成果或选中章节，并只返修受影响范围。
7. QA finding 可以定位到章节、页或镜头，返修与复验状态真实可恢复。
8. 刷新、重进项目和切换标签页后，当前成果和关键面板状态恢复正确。
9. 1366x768、1440x900 和 390x844 完成真实浏览器验收，无溢出、遮挡或不可返回状态。
10. 两名教师在不同项目使用时，当前成果、编辑、版本、QA 和下载完全隔离。
11. 教师界面不出现工程词、思维链、伪进度或模型自评冒充质量通过。
12. Artifact、HumanGate、Quality Gate、下载和最终交付原有业务能力无回归。

## 11. 非目标

- 不删除 Workflow、Artifact、Evidence、PlanGuard、HumanGate 或 Quality Gate。
- 不要求所有成果都使用同一种富文本编辑器。
- 不把全部项目成果永久铺在右侧。
- 不因为取消线性视觉就取消成果依赖、stale 传播和受控返修。
- 不在 V1 收尾阶段一次性重写整个工作台。
