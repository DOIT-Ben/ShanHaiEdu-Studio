# Codex 式对话面板重构规划 v0.3.3

## 1. 第一性原理：当前阶段核心需求

中央工作区的首要任务不是展示系统状态，而是让教师像和助教对话一样推进公开课材料生成。当前问题是对话、进度、产物摘要都被边框和底色包起来，视觉重心被“卡片集合”抢走，缺少 Codex 式纯白、留白、左右对话关系和轻操作反馈。

本轮成功标准：
- 对话画布为纯白，不再有大面积灰底画布或嵌套卡片。
- 教师消息靠右，使用极浅灰圆角消息块。
- AI 回复靠左，正文无底色；回复下方提供复制操作。
- 任务进度和产物摘要降级为对话下的轻量产物块，减少边框和底色交叉。
- 底部输入框更像 Codex：纯白、轻浮起、hover/focus 有克制光感，不使用彩色装饰。

## 2. 可复用方法和成熟实践

参考成熟对话式工作台的通用模式：
- ChatGPT / Codex 类界面：用户消息右对齐浅底，助手内容左对齐无底色，操作按钮隐藏或低调跟随回复。
- Linear / Notion 类工作区：结构靠留白、字体重量和细边界建立，而不是多色块。
- shadcn / Radix 复用：继续使用现有 Button、Textarea、ScrollArea，不重做 UI kit。

本轮不引入新依赖，不做炫技动效，不更改右侧产物 rail 和项目侧栏。

## 3. 复用、适配和必要自研

复用：
- `ScrollArea` 维持中间稳定滚动。
- `Button` 维持统一按钮体系。
- `Textarea` 维持输入框基础能力。
- `lucide-react` 用于复制、发送、附件、重新生成等图标。

适配：
- `ChatTranscript` 拆成更清楚的消息行职责，用户和 AI 分支分开渲染。
- `GenerationPanel` 从多卡片状态区改为单个轻量“任务产物块”，减少视觉噪点。
- `PromptComposer` 调整为 Codex 式浮起输入面板，hover/focus 用阴影和边框变化表达。

必要自研：
- AI 回复下方 `AssistantMessageActions`，提供复制反馈。
- `TeacherMessageBubble` / `AssistantMessageBlock` 用于稳定左右对话布局。

## 4. 落地方案、风险和验证标准

开发方案：
1. 新增或重构中央对话组件边界，让消息行、AI 操作条、任务产物块职责独立。
2. 移除中央大灰底和多层卡片框，保留纯白主画布。
3. 教师消息右对齐，AI 消息左对齐并增加复制按钮。
4. 输入框增加轻浮起视觉、hover/focus-within 状态、底部工具栏低噪声。
5. 保持窄屏不横向溢出，输入框不遮挡核心内容。

风险：
- 过度模仿 Codex 可能削弱 To B 公开课任务状态感；解决方式是保留轻量产物块和阶段条。
- AI 操作按钮若过显会增加噪点；解决方式是默认低对比，hover 后更清晰。
- 输入框阴影过重会变成装饰；解决方式是只使用灰阶和透明阴影。

验证标准：
- `npm run build` 通过。
- 浏览器检查 1374x1004：中间纯白，用户消息靠右，AI 消息靠左无底色，右侧节点仍显示。
- 浏览器检查 hover：输入框有轻微层级变化，AI 回复复制按钮可见并可点击。
- 浏览器检查 390px：无横向溢出，输入框不挡住关键对话。
