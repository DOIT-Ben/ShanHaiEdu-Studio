# V1.1 反馈闭环需求

日期：2026-07-13

状态：accepted for staged planning

关联需求：RQ-028、RQ-029、RQ-032、RQ-039

## 1. 版本目标

V1.1 是 V1 邀请制上线后的首个正式增强版本。它不重做智能体架构和生产链，重点把教师可见体验与产品改进能力补齐：教师知道系统正在做什么、可以随时调整；团队能把每条真实反馈推进到定位、修复、验证和回告。

V1.1对话、消息、活动流、当前计划和HumanGate的统一承载以`docs\archive\2026-07-16-authority-convergence\historical\docs\product\v1-1-assistant-ui-conversation-runtime-requirements.md`为唯一技术产品合同。本文件只定义反馈case、分诊、locator、教师回告、搜索和运营闭环，不重复定义另一套消息Runtime或事件协议。

V1.1 的成功不以“新增一个反馈入口”判断，而以以下结果判断：

1. 教师可以对页面、对话和真实产物的具体位置快速反馈。
2. 团队可以明确问题发生在哪个版本、哪个产物、哪个业务节点和哪个责任层。
3. 反馈可以关联修复版本与回归证据，并把处理结果反馈给原教师。
4. 至少两条真实教师反馈完成从提交到解决的闭环，且没有跨用户泄露。
5. 桌面与 390px 下，工作台、当前计划、成果查看和反馈流程均清晰可用。

## 2. 前提与边界

### 2.1 启动前置

- 进入 V1.1 实施前，V1-1 至 V1-10 必须已完成，产品内真实交付 E2E、两用户隔离、目标服务器恢复和 V1 发布门必须已关闭；当前规划不代表这些门禁已经完成。
- V1.1 继续使用邀请制账号；不因“正式上线”自动开放公开注册。
- V1.1 首轮验收沿用 V1 的单 Node 进程、单 Prisma singleton 和 SQLite 拓扑，并至少验证两个受邀教师并行使用。扩展到多实例或更高并发必须单独规划。
- 已存在的反馈分类、图片选择、剪贴板粘贴、失败重试、回执、持久化、管理员读取和导出全部复用。
- 教师侧反馈入口、当前计划和错误恢复统一挂载在assistant-ui对话Runtime；反馈业务状态仍由反馈服务端合同权威管理，不写入消息组件本地状态。

### 2.2 明确不做

- 不建设公开反馈广场、完整客服系统、复杂 SLA 或 Jira 替代品。
- 不自动录屏，不自动上传整页截图、完整对话、Prompt、密钥、请求头或本地路径。
- 不以粉色、儿童插画或大面积卡通元素迎合目标教师。
- 不重做三栏工作台，不用营销页、大 Hero、渐变、玻璃拟态和卡片套卡片。
- 不把模型思维链、Provider、API、nodeId 或内部调试状态展示给教师。

## 3. 用户与核心任务

核心用户是约 20 岁、需要高频完成公开课备课的年轻小学教师。设计假设是她熟悉手机和常见对话产品，但没有义务理解 Agent、工作流和媒体 Provider。

她需要完成四件事：

1. 快速开始或继续一个备课项目。
2. 看懂当前制作进度、等待确认和失败恢复位置。
3. 检查教案、PPT、视频和最终包，并对具体问题提出修改。
4. 提交反馈后知道团队是否理解、是否修复以及在哪个版本生效。

## 4. 视觉与交互口径

### 4.1 视觉系统

| 角色 | 建议值 | 用途 |
|---|---|---|
| 纸白 | `#FCFDFC` | 页面与阅读背景 |
| 墨绿黑 | `#18312B` | 主文字与高对比标题 |
| 山海绿 | `#367D6D` | 主操作、选中、正常进度 |
| 珊瑚 | `#E98573` | 轻提醒、教师回告，不用于大面积背景 |
| 琥珀 | `#D69A38` | 等待确认、需要补充 |
| 错误红 | `#B84C3F` | 阻塞与真实失败 |
| 中性边框 | `#E7ECE9` | 分区、输入和静态边界 |

- 中文正文优先使用 MiSans、HarmonyOS Sans SC 或可靠系统字体；正文 14-16px，工作台标题不超过 24px。
- 圆角统一为 6-8px；低频按钮 hover/focus 时出现；触控目标不小于 44px。
- 动效仅用于当前状态变化、计划展开和完成反馈，时长 120-200ms，并遵守 `prefers-reduced-motion`。
- 视觉记忆点是输入框上方的“当前制作计划”，不是装饰插画。

### 4.2 关键界面

- 欢迎页：最近项目、开始新备课和真实状态优先，不做营销内容。
- 对话工作台：中间对话保持主视觉；输入框始终可用，不被计划和弹窗遮挡。
- 当前制作计划：按 RQ-028 提供逐行“计划视图”和紧凑“进度视图”。开始与 Replan 时查看完整计划，稳定执行和收尾时用绿色已完成、蓝色进行中、灰色未开始的分段轨持续展示当前活动；等待确认与失败使用独立语义色和文字。
- 执行活动流：按RQ-032持续展示教师可理解的计划、业务动作、工具摘要、Observation、现有门禁状态和最终收尾，不展示思维链或工程日志；通用阶段QA展示延期到V1.2。
- 成果阅读：教案、PPT、视频和最终包统一提供版本、达标状态和“对此处反馈”。
- 失败状态：说明发生位置、影响范围、是否可重试和下一步动作，不只显示“生成失败”。
- 错误现场反馈：错误条只展示安全摘要；“反馈问题”打开已预填面板，教师确认后提交，系统不静默上报。
- 搜索与快捷操作：页面平时只保留一个低噪声入口；打开后按“建议/最近 -> 输入搜索 -> 分类结果”渐进展开。
- 移动端：项目与成果进入抽屉，中央对话保持唯一主区；计划与反馈不造成横向溢出。

### 4.3 当前制作计划双视图

```text
计划视图                           进度视图
计划名称                           计划名称        真实已耗时   [展开]
✓ 理解课程目标                     当前：审查样张和相关模块
✓ 形成叙事大纲                     [绿][绿][绿][蓝][灰][灰]
● 设计关键样张                     ● 进行中
○ 生成整套PPT
○ 审查并局部返修
```

- 计划视图承担“理解和调整”，按业务步骤逐行显示；进度视图承担“持续感知”，在输入框上方低高度常驻。
- 两种视图是同一 `activePlan` 的展示投影，不是两套计划、固定 DAG 或前端 Todo。
- 新计划、实质 Replan、等待确认、失败和阻塞默认展开一次；稳定执行和收尾可自动收起，但教师手动选择在当前 revision 内优先。
- 进度轨的分段对应业务里程碑，不对应底层 Tool 调用；超过可读数量时按父级里程碑聚合。
- 完成绿色、进行中蓝色、未开始浅灰、等待确认琥珀、失败红色，同时必须配文字、图标或 ARIA 状态。
- 只显示真实已耗时；没有可靠统计时不显示剩余时间。完成后全部有效分段转为绿色并提供最终成果入口。

## 5. 反馈闭环模型

### 5.1 双状态分离

现有 `FeedbackRecord.status` 继续表示提交与附件保存生命周期：

```text
processing -> submitted | failed
```

V1.1 新增独立业务处理生命周期，不复用或改写上述字段：

```text
submitted -> triaged -> planned -> in_progress -> verifying -> resolved
                  \-> needs_info
                  \-> declined
```

- `submitted`：教师反馈已可靠保存，等待分诊。
- `triaged`：已确认问题类型、影响和责任层。
- `planned`：已关联需求、缺陷或目标版本。
- `in_progress`：正在修复或调整。
- `verifying`：已实现，正在回归或教师复核。
- `resolved`：回归通过并向教师回告。
- `needs_info`：缺少复现或期望信息；必须提出一个具体补充问题。
- `declined`：明确不处理；必须记录教师可理解原因，不等同于删除反馈。

### 5.2 推荐数据边界

- `FeedbackRecord`：保留教师原始提交、上下文和附件，作为不可静默改写的事实记录。
- `FeedbackCase`：承载处理状态、优先级、责任层、负责人、重复归并、关联需求和目标版本。
- `FeedbackCaseEvent`：记录状态变化、内部归因、验证和回告审计。
- 多个重复 `FeedbackRecord` 可以归并到一个 `FeedbackCase`，但每位教师仍能看到自己提交的回执和处理结果。

最终字段以独立 API/数据合同阶段确定；数据库变更必须加法迁移、幂等升级、可备份恢复，不在需求文档中提前锁死物理表结构。

## 6. 精确反馈定位

每条反馈保留通用上下文，并根据目标增加受控 locator：

| 目标 | 自动绑定 | 教师快捷原因 |
|---|---|---|
| 页面/UI | route、viewport、buildVersion、交互入口 | 看不懂、找不到、布局不舒服、手机显示异常 |
| 对话消息 | projectId、messageId、IntentEpoch | 没理解需求、承诺未执行、追问太多、回答不自然 |
| 教案 | artifactId、versionId、sectionId | 目标不准、学情不符、活动不可执行、评价不合理 |
| PPT | artifactId、versionId、pageId/pageNumber | 内容太多、排版不好、图片不合适、知识不准、课堂动作不清楚 |
| 视频 | artifactId、versionId、shotId、timeStartMs/timeEndMs | 锚点错误、创意受教材束缚、节奏不当、信息太少、画面不连贯、音字问题 |
| 最终包 | packageId、manifestDigest、artifactVersions | 版本不一致、缺文件、不可直接上课、下载或打开失败 |
| 当前计划 | planId、revision、currentStepId、HumanGate/Quality状态 | 卡住、状态不清、无法改道、返修范围错误 |

所有 ID 必须由服务端验证当前 actor 的访问权；普通教师界面只显示页码、镜头、时间段和业务步骤等可理解信息。

### 6.1 错误现场一键反馈

当对话、Agent任务、生成任务、产物处理或下载发生可反馈错误时，系统在受影响位置附近显示错误条：

```text
生成服务连接已中断                  [查看详情] [复制编号] [反馈问题] [关闭]
```

- 错误摘要必须是教师可理解的业务语言；Provider、transport、stdio、stack、workspace path 等只允许进入脱敏诊断包，不直接显示。
- 点击“反馈问题”后复用同一个反馈面板，自动预选分类、填写安全标题，并在知道真实上下文时预填“当时正在做什么”和“期望结果”；无法确定时保留引导占位，不让模型编造。
- 表单中显示“已附带诊断信息”，教师可以展开查看脱敏字段摘要或关闭附带；打开面板、查看详情和复制错误编号均不提交反馈。
- 自动上下文至少覆盖：安全错误码、发生时间、页面、应用版本、项目、消息、计划revision、当前步骤、任务、产物版本、重试次数、最后成功步骤和关联trace；缺失字段保持空，不用默认字符串伪装。
- 诊断包禁止包含完整对话、Prompt、请求/响应正文、token、密钥、Cookie、请求头、签名URL、本机路径、原始堆栈和未获授权的截图。
- 错误现场与普通反馈进入同一 `FeedbackRecord -> FeedbackCase` 闭环，并用 diagnostic fingerprint 支持团队归并重复问题；fingerprint 不向教师暴露。

## 7. 搜索与快捷操作

### 7.1 入口与层级

- 桌面通过顶栏/侧栏搜索图标或 `Ctrl/Cmd+K` 打开；移动端通过搜索图标打开近全屏 Sheet。
- 关闭态不常驻大搜索框；打开且无输入时展示常用操作和最近内容；输入后再显示分类标签和匹配结果。
- 搜索输入提示固定使用教师语言：`搜索操作、项目或成果`。
- 第一版分类标签为：`全部`、`操作`、`项目`、`成果`。

### 7.2 结果内容

| 分组 | 第一版内容 |
|---|---|
| 建议 | 开始新备课、继续当前项目、上传教材、打开当前制作计划 |
| 操作 | 查看成果、提交反馈、我的反馈、调整生成强度、打开个人设置 |
| 项目 | 有权访问的活跃/最近项目；归档项目仅在筛选或明确关键词下出现 |
| 任务 | 当前计划、等待确认、失败可恢复任务、最近完成任务 |
| 成果 | 教案、PPT、课堂视觉图、视频、最终包及其真实版本/达标状态 |
| 管理 | 仅对具备权限的管理员或owner显示用户/成员管理入口 |

- 搜索是导航和安全快捷操作，不是第二个AI聊天框，也不是通用命令终端。
- 低风险操作可以直接进入对应界面；归档、删除、重新生成、提高强度等有副作用动作继续走原确认或HumanGate。
- 结果必须在服务端授权范围内生成；项目、任务、成果和反馈不得先返回后隐藏。
- V1.1 不搜索完整对话、教材正文和附件全文，不显示MCP、Skill、Provider、模型、API、路径或调试项。
- 桌面支持方向键、Enter、Escape和焦点回归；390px 输入固定、结果滚动、标签可横向安全浏览且不遮挡软键盘。

## 8. 教师侧闭环

### 8.1 入口

- 保留全局反馈、个人菜单和消息赞踩入口。
- 成果阅读区新增局部入口：`反馈这一页`、`反馈这一段`、`反馈这个镜头`。
- 视频支持选择镜头或时间段；PPT 默认绑定当前页；教案默认绑定当前段落。
- 完成、下载、连续失败或准备离开未完成任务时，只显示一次非阻塞轻问，不自动打开大弹窗。

### 8.2 我的反馈

教师只能查看自己提交的反馈，最少展示：

- 回执编号、提交时间、目标和简要标题。
- 教师可见状态：已收到、处理中、等待补充、待验证、已解决、暂不处理。
- 处理结果、关联发布版本和“问题仍存在”入口。
- `needs_info` 只提出一个具体问题；教师补充后进入同一 case，不新建无关联反馈。

V1.1 不建设自由评论线程。团队回告采用结构化更新，教师可以补充信息或重新打开，降低复杂度和骚扰风险。

## 9. 团队侧分诊工作台

管理端提供轻量工作队列：

- 按状态、分类、严重度、产物类型、责任层、版本和提交时间筛选。
- 查看原始反馈、问题图/参考图、合法上下文、产物 locator 和相同 case 的重复反馈。
- 设置优先级、责任层、负责人、复现状态、目标版本和关联需求。
- 记录根因、修复摘要、回归证据和教师可见回告。
- 只允许结构化状态推进；`resolved` 必须具备修复版本、验证证据和用户回告摘要。

责任层最少支持：前端体验、Main Agent 编排、Prompt/规则、Tool、Provider、质量门禁、数据/权限、性能/部署、产品范围。

## 10. 反馈驱动产品决策

V1.1 需要最小指标面板或可验证导出，至少回答：

- 反馈提交成功率、上下文完整率和有效反馈率。
- 各类型、各产物、各节点的问题数量与重复率。
- 首次分诊时间、平均解决周期、重新打开率。
- PPT 页级问题率、视频课程锚点问题率和整包首次可授课率。
- 首次任务完成率、失败恢复率和受邀教师继续使用率。

反馈数量不是成功指标。真正指标是问题能否定位、能否关闭、是否在新版本重复出现，以及教师是否更快拿到可用成果。

## 11. 权限、隐私与审计

- 教师只能读取自己的反馈与教师可见更新；不能读取他人反馈、内部根因、负责人或敏感运行字段。
- 管理端继续要求 password auth、管理员权限、CSRF 和受控附件访问。
- locator 中的 project、message、artifact、plan 必须逐项校验 actor 权限和归属一致性。
- 不记录 Prompt、模型思维链、token、请求头、密钥、签名 URL、本地路径和完整对话。
- 所有状态推进、归并、回告和重新打开必须有脱敏审计。
- 导出继续防 CSV 公式注入；用户文本不得直接进入日志或指标标签。

## 12. V1.1 总验收

1. V1 原有反馈提交、附件、幂等、权限和恢复能力无回归。
2. 上传保存状态与业务处理状态明确分离，旧反馈可以安全进入新闭环。
3. 教师可以查看自己的反馈进度，不能读取他人反馈。
4. 团队可以完成分诊、归因、规划、修复、验证、回告和重新打开。
5. PPT、视频、教案和最终包的 locator 均绑定真实版本，局部反馈不会误指向旧产物。
6. 错误条可以一键打开已预填反馈，诊断信息可预览/关闭且不含敏感值；未经提交不会保存反馈。
7. 当前制作计划在刷新、Replan、HumanGate、Quality Gate、失败和局部返修后保持真实一致。
8. 搜索与快捷操作覆盖授权项目、任务、成果和常用操作，权限变化后不泄露旧结果。
9. 长任务活动流真实恢复业务动作、Observation、现有门禁状态和最终收尾，不伪造事件或显示思维链。
10. 1366x768 与 390x844 下无溢出、遮挡、不可达操作或工程词泄露。
11. 两名受邀教师在不同项目并行提交、查看和补充反馈，活动事件和数据完全隔离。
