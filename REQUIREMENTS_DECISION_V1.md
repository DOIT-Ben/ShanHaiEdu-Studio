# ShanHaiEdu 媒体工作台需求决策文档 v1

日期：2026-07-07

## 1. 文档定位

本文档是 ShanHaiEdu 媒体工作台第一版产品需求基线。它基于原始需求做产品经理视角、教师用户视角和工程落地视角的纠偏，但不展开成完整技术架构文档。

本文档用于指导下一阶段开发、验收和范围控制。

关联文档：

- 原始需求记录：`原始需求记录_V1.md`
- 当前前端原型：`ShanHaiEdu-Studio`

## 2. V1 产品定义

### 2.1 一句话定义

ShanHaiEdu 媒体工作台是一套面向小学数学公开课教师的对话式线性 AI 备课工作台，帮助教师从一句话需求开始，逐步确认并生成需求规格、教材证据、教案、PPT 规划、导入视频方案和最终交付清单。

### 2.2 V1 核心目标

V1 先验证“真实可用的线性备课闭环”，而不是一开始做完所有高成本文件生成。

V1 必须做到：

- 教师能用自然语言启动项目。
- 系统能识别需求并追问缺失字段。
- 系统能生成并保存需求规格说明书。
- 系统能生成并保存教材证据包。
- 系统能生成并保存教案。
- 系统能生成并保存 PPT 大纲和逐页脚本。
- 系统能生成并保存导入视频方案、脚本和分镜提示词。
- 系统能生成最终交付清单。
- 每个节点都能查看、复制、作为下一步输入、确认和重做。
- 刷新后项目、对话、节点和产物仍然存在。

### 2.3 V1 不追求

V1 不追求一次性完成全部生产能力：

- 不强求高质量 PPTX 自动生成完整上线。
- 不强求视频成片自动生成完整上线。
- 不强求图片资产真实批量生成完整上线。
- 不强求教材库自动检索覆盖所有版本。
- 不强求企业级多租户、计费、权限审批。

这些能力可以预留入口，但不能伪装成已经完成。

## 3. 核心产品原则

### 3.1 线性可控

公开课生产链路必须按节点推进。每个下游节点只能使用已确认的上游产物作为可靠输入。

节点顺序：

```text
需求澄清
-> 需求规格
-> 教材证据
-> 教案
-> PPT 规划
-> 导入视频方案
-> 图片与分镜提示
-> 最终交付
```

### 3.2 对话是主工作区

教师不应面对复杂表单和工程面板。中间对话区是主要工作空间，负责：

- 接收自然语言。
- 展示系统追问。
- 展示任务卡。
- 展示候选方案。
- 展示确认动作。
- 展示生成结果摘要。

### 3.3 节点是产物导航，不是日志

右侧线性节点不是运行日志，而是教师复用上游产物的工具。

每个节点都必须回答：

- 这个节点是什么。
- 当前状态是什么。
- 产物摘要是什么。
- 这个产物能不能作为下一步输入。
- 如何查看详情。
- 如何复制关键内容。
- 如何重做。

### 3.4 真实状态优先

V1 允许某些高成本产物降级，但不能把演示数据包装成真实完成。

必须真实保存：

- 项目。
- 对话。
- 节点状态。
- 节点产物。
- 用户确认。
- 重做记录。

### 3.5 先把文本链路跑通

第一版最重要的产物是文本和结构化教学规划：

- 需求规格。
- 教材证据。
- 教案。
- PPT 大纲。
- 导入视频方案。
- 分镜和提示词。
- 最终交付清单。

PPTX、图片、视频属于后续增强或外部能力接入，不阻塞 V1 闭环。

## 4. 用户角色与核心场景

| 用户角色 | 核心目的 | V1 关注点 |
| --- | --- | --- |
| 小学数学教师 | 快速完成公开课备课材料 | 简单、可控、能改、能下载 |
| 教研负责人 | 检查公开课内容是否合理 | 教材依据、教学目标、流程完整性 |
| 内部试用人员 | 验证产品闭环 | 状态保存、节点推进、失败恢复 |

## 5. V1 业务流程

### 5.1 新建项目

用户从左侧项目栏新建公开课项目。

项目可以先只有标题，也可以通过一句话自动生成标题。

进入项目后：

- 中间展示对话工作区。
- 右侧节点定位到“需求澄清”。
- 输入框可直接发送需求。

### 5.2 一句话启动

示例输入：

```text
我想要生成一个小学五年级关于百分数这个知识点的公开课 PPT。
```

系统回复不应只是一段聊天，而应形成“已理解 + 待确认 + 下一步动作”。

系统识别：

- 年级：五年级。
- 学科：数学。
- 知识点：百分数。
- 场景：公开课。
- 目标产物：PPT。

系统追问：

- 教材版本。
- 是否已有教材。
- 是否需要教案。
- 是否需要导入视频。
- 是否需要完整交付包。
- PPT 页数。
- 公开课时长。
- PPT 风格。

### 5.3 需求澄清

需求澄清应遵守：

- 一次只追问一组强相关问题。
- 每个问题给出 2-4 个选项。
- 有明确推荐项。
- 支持用户自定义输入。
- 支持用户跳过非必要字段。

推荐字段优先级：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| 年级 | 必填 | 已知时自动填入 |
| 学科 | 必填 | V1 默认小学数学 |
| 课题/知识点 | 必填 | 决定后续内容 |
| 公开课时长 | 必填 | 影响教案和 PPT 节奏 |
| 教材版本 | 建议必填 | 没有时可先用“待补充” |
| 教材内容 | 建议必填 | 没有时降低证据可信度 |
| 是否生成教案 | 必填 | PPT 和视频需要教案约束 |
| 是否生成 PPT | 必填 | 用户可能只要教案 |
| 是否生成导入视频 | 可选 | 默认询问 |
| PPT 页数 | 需要 PPT 时必填 | 给推荐页数 |
| 视频类型 | 需要视频时必填 | 给推荐类型 |

### 5.4 需求配置清单

需求分析完成后，系统生成配置清单。

配置清单需要显示：

- 已确认字段。
- 仍缺失但可后补字段。
- 系统推荐配置。
- 可能影响交付质量的风险。

用户可以：

- 确认。
- 修改。
- 补充。
- 取消某些交付物。

### 5.5 需求规格说明书

用户确认配置清单后，系统生成需求规格说明书。

需求规格说明书是第一个正式节点产物。

它需要保存为 Markdown，并包含：

- 项目概述。
- 用户目标。
- 年级、学科、课题、课型。
- 教材信息。
- 交付范围。
- PPT 要求。
- 视频要求。
- 风格要求。
- 质量约束。
- 后续节点输入说明。

### 5.6 教材证据包

教材证据包根据用户提供的教材内容生成。

V1 决策：

- 优先支持用户粘贴或上传教材内容。
- 教材自动检索作为后续增强，不进入 V1 必做。
- 用户没有教材时，可以继续，但节点必须标记“教材依据待补充”。

教材证据包包含：

- 教材版本。
- 页码或页段。
- 知识点。
- 关键例题或情境。
- 教材依据摘要。
- 与教学目标的关系。

### 5.7 教案生成

教案基于需求规格和教材证据生成。

教案必须包含：

- 教材依据。
- 教学目标。
- 教学重点。
- 教学难点。
- 教学流程。
- 导入设计。
- 学生活动。
- 板书设计。
- 课堂总结。
- 教师讲稿要点。

用户可以：

- 查看 Markdown 预览。
- 复制关键内容。
- 作为 PPT 输入。
- 确认。
- 重做。

### 5.8 PPT 规划

V1 的 PPT 阶段先做“规划与脚本”，不把 PPTX 作为硬性闭环。

PPT 规划包含：

- 建议页数。
- 页面类型配比。
- 封面风格方向。
- 逐页脚本。
- 每页教学目标。
- 每页学生活动。
- 主视觉需求。
- 板书或课堂衔接。

PPTX 文件能力：

- V1 可以预留“生成 PPTX”入口。
- 如果尚未接入真实生成能力，必须标记为“待生成”或“后续接入”。
- 不得显示为已生成。

### 5.9 导入视频方案

导入视频阶段先生成视频方案、脚本、分镜和提示词。

视频方案必须遵守：

- 不提前讲知识点。
- 不把教案改写成视频。
- 不替代教师授课。
- 以吸引、悬念、冲突、现象为主。
- 结尾通过课程锚点接回本课。

视频方案包含：

- 独立主题。
- 开场钩子。
- 吸睛点。
- 课程锚点。
- 课堂落点问题。
- 视频脚本。
- 分镜摘要。
- 图片提示词。
- 旁白建议。

### 5.10 图片与分镜提示

V1 输出图片和视频生成前所需的提示词，不强制真实生成图片。

每条提示词包含：

- 用途。
- 画幅。
- 画面主体。
- 风格约束。
- 角色或场景引用。
- 使用位置。
- 可复制文本。

### 5.11 最终交付

最终交付节点汇总：

- 需求规格说明书。
- 教材证据包。
- 教案。
- PPT 大纲和逐页脚本。
- 导入视频方案。
- 图片和分镜提示词。
- 教师讲稿要点。
- 检查清单。

如果 PPTX、图片文件、视频成片未真实生成，最终交付必须明确标记。

## 6. 智能体职责决策

### 6.1 V1 不直接拆成过多独立智能体

原始需求中的多智能体方向是合理的，但 V1 如果一开始拆成主智能体、需求智能体、教材智能体、PPT 智能体、视频智能体、审查智能体、剪辑智能体，会导致首版闭环变慢。

V1 决策：

- 使用一个主控对话入口。
- 按任务模式组织能力。
- 每个任务模式对应一个节点产物。
- 为后续拆独立智能体保留边界。

### 6.2 V1 任务模式

| 任务模式 | 负责节点 | 输出 |
| --- | --- | --- |
| 需求澄清 | 需求规格 | 配置清单、需求规格说明书 |
| 教材与教案 | 教材证据、教案 | 教材证据包、公开课教案 |
| PPT 规划 | PPT 草稿 | PPT 大纲、逐页脚本、风格方向 |
| 视频导入 | 导入视频、分镜提示 | 视频方案、脚本、分镜、提示词 |
| 交付汇总 | 最终交付 | 交付清单、检查清单 |

### 6.3 后续可拆分的专业智能体

当 V1 跑通后，再逐步拆分：

- 需求分析智能体。
- 教材解析智能体。
- 教案智能体。
- PPT 智能体。
- 视频创意智能体。
- 图片资产智能体。
- 审查智能体。
- 剪辑智能体。

这样可以避免第一版过早复杂化。

## 7. 审查机制决策

### 7.1 V1 必须有审查意识

V1 不能完全没有质量把关。每个主要产物都应有自检清单。

自检内容：

- 是否符合年级认知。
- 是否匹配课题。
- 是否有教材依据。
- 教学流程是否完整。
- PPT 信息密度是否适中。
- 视频是否提前讲知识点。
- 视频是否有课程锚点。
- 最终交付是否齐全。

### 7.2 V1 不做复杂自动重试

V1 不做多轮自动打回重试作为必做项。

原因：

- 自动审查和自动重试容易消耗时间。
- 质量阈值需要真实样本校准。
- 过早做复杂审查会拖慢第一版闭环。

V1 做法：

- 产物生成后给出自检摘要。
- 用户决定确认或重做。
- 重做时保留旧版本。
- 失败时给可理解恢复方案。

### 7.3 后续增强

后续再引入：

- 独立审查智能体。
- 多维度评分。
- 低于阈值自动重试。
- 最多重试 3 次。
- 超过次数返回最高分版本，由用户决定。

## 8. 前端体验需求

### 8.1 桌面布局

桌面端保留当前产品形态：

```text
左侧项目栏
中间对话主工作区
右侧阅读侧栏
最右压缩线性节点
```

其中：

- 中间对话是视觉和操作中心。
- 右侧阅读侧栏用于查看 Markdown、图片、提示词和节点详情。
- 最右节点串用于快速跳转、预览、复制和复用。

### 8.2 左侧项目栏

要求：

- 紧凑项目列表。
- 当前项目浅灰选中态。
- 显示项目标题、状态、更新时间。
- 支持新建项目。
- 支持切换项目。
- 不使用花哨卡片。

### 8.3 中间对话区

要求：

- 纯白背景。
- AI 回复靠左，尽量无重底色。
- 用户消息靠右，浅灰气泡。
- 低频按钮默认隐藏，悬浮时出现。
- AI 回复下方支持复制。
- 输入框固定底部。
- Enter 发送，Shift+Enter 换行。
- 支持插入节点引用。
- 支持任务卡和确认动作。

### 8.4 对话导航条

中间对话区左侧可有细导航条，用于快速跳转关键对话节点。

要求：

- 默认低调。
- 悬浮时显示预览。
- 点击跳到对应消息位置。
- 不抢主视觉。

### 8.5 右侧节点串

要求：

- 默认压缩。
- 线性排列，状态清楚。
- 当前节点略高亮。
- 已完成节点低调。
- 需重审节点用低饱和提醒。
- 悬浮显示小预览。
- 悬浮预览有延迟关闭，保证能点击按钮。
- 点击节点打开详情侧栏。
- 再次点击同一节点关闭详情侧栏。
- 详情侧栏打开时，不显示冲突悬浮层。

### 8.6 阅读侧栏

要求：

- 从右侧平滑打开。
- 支持 Markdown 渲染。
- 支持图片和提示词查看。
- 支持复制、作为输入、确认、重做。
- 支持宽度调整。
- 调整宽度需要响应及时。

### 8.7 窄屏和平板

窄屏降级：

- 左侧项目栏折叠。
- 节点串变成顶部或底部入口。
- 阅读侧栏改成抽屉。
- 中间对话保持第一优先级。

## 9. 视觉规范需求

### 9.1 总体风格

V1 继续保持：

- 纯白。
- 极简。
- 高级。
- 低噪声。
- 类 Codex 对话体验。
- To B 工具气质。

### 9.2 字体

要求：

- 全局字体统一。
- 字号层级控制在 3 种以内。
- 最大字号不超过 35px。
- 不使用负字距。
- 不随视口宽度缩放字号。

建议层级：

- 标题：20-24px。
- 正文：14px。
- 辅助信息：12px。

### 9.3 颜色

要求：

- 主界面以白色和浅灰为主。
- 选中态用浅灰底。
- 成功、警告、错误只在状态必要时出现。
- 不使用多色块交叉堆叠。
- 不使用 emoji。
- 不使用大面积渐变。

### 9.4 组件密度

要求：

- 信息密度适中。
- 对话区留白充足。
- 节点预览只展示关键摘要。
- 详情内容放到阅读侧栏，不挤在节点串里。
- 不做卡片套卡片。

## 10. 产物与节点需求

### 10.1 节点状态

V1 节点状态：

| 状态 | 教师可见含义 |
| --- | --- |
| 未开始 | 还没有生成 |
| 进行中 | 正在生成 |
| 待确认 | 已生成，等待确认 |
| 已确认 | 可作为后续输入 |
| 需重审 | 上游内容变化，需要重新确认 |
| 失败 | 生成未完成，可重试或调整 |

### 10.2 节点通用动作

每个节点支持：

- 查看详情。
- 复制关键内容。
- 作为下一步输入。
- 确认。
- 重做。

### 10.3 上游变更规则

如果上游产物变更：

- 下游节点不删除。
- 下游旧内容保留。
- 下游标记“需重审”。
- 用户可以查看旧内容。
- 用户可以继续使用旧内容，但必须明确提示风险。
- 用户可以重做下游节点。

### 10.4 失败恢复

失败时不显示技术错误。

教师可见内容应包括：

- 哪一步没有完成。
- 已生成内容是否保留。
- 可以怎么继续。
- 是否建议重试或修改输入。

| 模块 | 功能 | 优先级 | 说明 |
| --- | --- | --- | --- |
| 项目管理 | 新建项目 | 必须 | 创建公开课项目 |
| 项目管理 | 项目列表与切换 | 必须 | 左侧项目栏 |
| 对话 | 自然语言输入 | 必须 | 中间主工作区 |
| 对话 | 需求追问 | 必须 | 分组追问缺失字段 |
| 对话 | 任务卡与确认 | 必须 | 不只是聊天气泡 |
| 节点 | 节点串展示 | 必须 | 右侧压缩线性节点 |
| 节点 | 悬浮预览 | 必须 | 关键摘要和动作 |
| 节点 | 详情侧栏 | 必须 | Markdown 和详情查看 |
| 节点 | 复制关键内容 | 必须 | 复制到剪贴板 |
| 节点 | 作为下一步输入 | 必须 | 插入输入框 |
| 节点 | 确认和重做 | 必须 | 线性推进核心 |
| 需求 | 需求规格说明书 | 必须 | 第一个正式产物 |
| 教材 | 教材证据包 | 必须 | 基于粘贴或上传教材 |
| 教案 | 公开课教案 | 必须 | Markdown 预览 |
| PPT | PPT 大纲和逐页脚本 | 必须 | PPTX 后续增强 |
| 视频 | 导入视频方案和分镜 | 必须 | 成片后续增强 |
| 交付 | 最终交付清单 | 必须 | 汇总所有已完成产物 |
| 文件 | Word/PDF/PPTX/视频下载 | 可选 | 可后续逐步接入 |
| 教材 | 自动检索教材 | 可选 | 不进入 V1 必做 |
| 审查 | 多维自动评分 | 可选 | 后续增强 |

## 12. 数据与保存需求

V1 需要真实保存：

- 项目信息。
- 对话消息。
- 节点状态。
- 节点产物。
- 产物版本。
- 用户确认状态。
- 重做记录。

刷新页面后应恢复：

- 当前项目。
- 对话历史。
- 当前节点。
- 右侧节点状态。
- 已确认产物。

V1 至少支持 1-2 人真实试用，并预留 5 人小规模试运行。

## 13. 生成产物格式

### 13.1 Markdown 优先

V1 所有文本类产物优先使用 Markdown：

- 需求规格说明书。
- 教材证据包。
- 教案。
- PPT 大纲。
- 视频方案。
- 分镜提示词。
- 最终交付清单。

原因：

- 易预览。
- 易复制。
- 易下载。
- 易作为下游输入。
- 易做版本记录。

### 13.2 文件产物降级

文件类产物可逐步接入：

- 教案 Word。
- 教案 PDF。
- PPTX。
- PPT PDF。
- 图片文件。
- 视频文件。

未接入时，界面必须明确标记，不得展示成已完成。

## 14. 关键验收标准

### 14.1 产品验收

V1 完成时必须满足：

- 教师可以新建项目。
- 教师可以输入一句自然语言需求。
- 系统可以识别已知字段和缺失字段。
- 系统可以追问并生成配置清单。
- 用户确认后生成需求规格说明书。
- 系统可以生成教材证据包。
- 系统可以生成教案。
- 系统可以生成 PPT 大纲和逐页脚本。
- 系统可以生成导入视频方案和分镜提示词。
- 系统可以生成最终交付清单。
- 右侧节点能查看、复制、作为输入、确认、重做。
- 上游变更后下游能标记需重审。
- 失败时给教师可理解恢复态。
- 刷新后状态不丢。

### 14.2 前端验收

必须满足：

- 页面整体纯白、简约、低噪声。
- 中间对话是主视觉。
- 左侧项目栏清晰可扫描。
- 右侧节点默认压缩。
- 节点悬浮预览不与详情侧栏冲突。
- 点击节点可平滑打开和关闭详情侧栏。
- AI 回复复制按钮默认隐藏，悬浮时显示。
- Enter 可以发送。
- 输入框和按钮可用。
- 普通教师主界面没有工程词。
- 字号层级不超过 3 种。
- 最大字号不超过 35px。

### 14.3 真实可用验收

必须满足：

- 对话和节点产物真实保存。
- 刷新后恢复。
- 两个项目之间不串数据。
- 演示数据和真实生成有明确区分。
- 没有把未完成文件产物伪装为完成。

## 15. V1 范围边界

### 15.1 V1 必做

- 对话式需求澄清。
- 需求规格说明书。
- 教材证据包。
- 教案。
- PPT 大纲和逐页脚本。
- 导入视频方案。
- 图片和分镜提示词。
- 最终交付清单。
- 项目、对话、节点、产物保存。
- 节点查看、复制、作为输入、确认、重做。
- 纯白极简前端体验。

### 15.2 V1 可做但不阻塞

- 教案 Word/PDF 导出。
- PPTX 初步生成。
- 图片生成。
- 视频首帧图。
- 视频片段测试。
- 教材检索。

### 15.3 V1 不做

- 完整自动剪辑成片。
- 高质量商业级 PPTX 自动生成闭环。
- 全教材库自动匹配。
- 企业级权限。
- 支付计费。
- 大规模并发。
- 复杂自动审查和三轮自动重试。

## 16. 产品决策结论

### 16.1 接受：当前前端形态继续作为主界面

理由：

- 已经符合“左项目、中对话、右节点”的核心产品形态。
- 用户已经认可当前原型方向。
- 后续重点应转向真实状态和真实生成，而不是重写 UI。

### 16.2 接受：V1 先做真实文本闭环

理由：

- 需求、教材、教案、PPT 大纲、视频方案是公开课生产的核心骨架。
- 文本链路跑通后，PPTX、图片、视频可以作为外部能力逐步接入。
- 这样能最快验证产品价值。

### 16.3 调整：多智能体先收敛为任务模式

理由：

- 原始多智能体分工正确，但第一版不宜过早复杂化。
- 任务模式能保留边界，又能快速交付。

### 16.4 调整：教材检索后置

理由：

- 教材检索涉及版权、版本、准确性和来源可信度。
- V1 先支持用户上传或粘贴教材，更可控。

### 16.5 调整：审查先轻量，后续增强

理由：

- 质量审查重要，但评分阈值需要样本校准。
- V1 先做自检清单、用户确认和重做。
- 后续再做独立审查智能体和自动重试。

### 16.6 调整：视频先做方案，不急于成片

理由：

- 视频生成成本高、耗时长、质量波动大。
- 导入视频的产品关键是创意方向、课程锚点、脚本和分镜。
- 成片能力可以在方案稳定后接入。

## 17. 下一阶段开发入口

下一阶段不是继续写架构文档，而是围绕 V1 需求做真实闭环开发。

建议第一条可交付链路：

```text
新建项目
-> 输入一句话需求
-> 系统追问缺失字段
-> 用户确认配置清单
-> 生成需求规格说明书
-> 保存为右侧节点
-> 节点可查看、复制、作为输入
-> 刷新后恢复
```

这条链路跑通后，再依次接：

```text
教材证据
-> 教案
-> PPT 大纲
-> 导入视频方案
-> 最终交付清单
```

## 18. V1 成功定义

V1 成功不是“生成了一个看起来很炫的 PPT 或视频”，而是：

- 教师从一句话开始能走完一个可理解、可确认、可回看的备课链路。
- 每一步产物都能沉淀为节点。
- 上一步确认内容能进入下一步。
- 教师能随时复制、查看、重做和继续。
- 刷新后状态不丢。
- 未完成的高成本文件能力不会被伪装为完成。
- 整体界面像成熟 To B 产品，而不是 demo 或工程后台。
