# 真实 API 实验总结

实验时间：2026-07-11。所有密钥仅在进程内读取，未写入实验文件。

## 1. Agent Brain：PPT 工作流 A/B

### 方法

- 相同模型：`gpt-5.5`。
- 相同课题与三页范围：《1～5的认识》。
- 方案 A：现行“四层设计稿”提示，要求底图、元素、文字、排版和教学动作。
- 方案 B：A+ 提示，先定义传播任务与累计叙事，再输出 PageSpec、主视觉、文字预算、本地数学层和页面验收。
- 第三次调用作为独立 Critic，只看到“方案 A/B”，不知道版本身份。

### Provider 可用性证据

- 主 Agent Brain 的 `/models` 正常，能列出 `gpt-5.5`。
- 主 `/responses` 在默认负载和 `low reasoning + max_output_tokens` 两种负载下连续返回 504，因此停止原路重试。
- 独立 fallback 的 `/models` 与 `/responses` 正常，三次实验调用全部完成。

这说明运行时应把“models 可见”和“实际生成可用”分开探测，并对 504 进入 Provider failover，而不是继续同路重试。

### 盲评结果

| 方案 | 总分 | Critic 选择 |
|---|---:|---|
| A：四层提示 | 87.5/100 | 否 |
| B：A+ PageSpec | 95/100 | 是 |

方案 B 的主要提升：

- 学习叙事 4/4。
- 主视觉解释力 4/4。
- 投影可读性 4/4。
- 页面推进 4/4。
- 可执行性 4/4。

Critic 给出的具体证据包括：单一展台近景、不分栏、屏幕文字不超过两行；用“物体 -> 点子 -> 数字标签”形成视觉路线；生成图与本地可编辑数学层分离。

### 发现的 A+ 缺口

方案 B 为降低单页密度，只完整示例了部分数字，可能遗漏 `1～5` 的完整覆盖。已据此修订：

- `ppt_page_specs` 增加 `learning_objective_coverage`。
- 质量门要求每个已批准目标/概念映射到至少一页。
- PPT Director Prompt 必须先输出目标到 page_id 的覆盖矩阵。

## 2. imagegen-free：视觉构图 A/B

### 方法

- 同一 Provider profile：`aiyunrouter-cnapi`。
- 同一模型：`gpt-image-2`。
- 相同画幅：16:9、1K 语义尺寸。
- 实际输出均为 1672×941。

### 基线

Prompt 直接要求“五个展柜横向整齐排列、每个有空白标签”。结果准确落入等宽五列卡片式构图，数学任务易懂，但整体像课件面板，缺少叙事世界、景深和视线运动。

证据：`images/baseline.png`。

### A+

Prompt 指定沉浸式全幅场景、儿童视角、对角纵深、主次焦点、标题安全区、下方本地标签叠加区，并明确禁止卡片网格、文字、数字和精确可数教学对象。

结果形成明确的左上文字安全区、左下儿童、右侧发光目标、中央飞纸视线轨迹，视觉更适合作为封面/情境页；数学标签可由本地确定性层叠加。

证据：`images/optimized.png` 与 `images/comparison-side-by-side.png`。

### 结论边界

这是构图与素材职责实验，不证明 A+ 图片自动满足全部教材或课堂要求。精确数量、中文和数字仍必须由本地层完成。

## 3. Evolink 视频：A+ ShotSpec 实验

### 请求

- 模型：`grok-imagine-text-to-video-beta`。
- 目标：6 秒、16:9、480p。
- Prompt 明确 subject、action、context、camera、start/end state、motion、style 和 negative constraints。
- Provider 实际消耗：4.8 credits。

### 技术结果

| 项目 | 实际值 |
|---|---|
| 时长 | 6.041667 秒 |
| 主视频 | H.264 High，752×416，24fps |
| 音频 | AAC，44.1kHz，双声道 |
| 文件大小 | 1,452,330 bytes |
| 完整解码 | 通过 |
| blackdetect | 无命中 |
| freezedetect，0.8 秒阈值 | 无命中 |
| 综合响度 | -32.3 LUFS（FFmpeg EBU R128） |
| 响度范围 / True Peak | 1.1 LU / -19.6 dBFS |

### 视觉结果

通过项：

- 从宽景到向前推进的连续镜头成立。
- 儿童、飞纸、发光门洞形成清晰动作和悬念终点。
- 没有卡片墙和平均多列构图。
- 四个抽帧中角色与场景基本连续。

失败/风险：

- 实际分辨率为 752×416，不等于预期常见 854×480，也不适合作为最终 1080p 交付。
- 背景中出现类似展签的模糊生成文字，违反 `no signage / no text` 负面约束。
- 输出包含额外 attached-picture 视频流；最终装配前应规范化为明确的一路主视频和一路音频。
- 音轨实测为 -32.3 LUFS，明显偏低；最终混音必须按交付 profile 做响度归一化并重新测量，不能以“存在音轨”代替可听性通过。
- 本实验只验证一个 A+ ShotSpec，没有生成同 Provider 的视频基线，因此不能把它表述为严格视频 A/B 胜出。

### 已回写的 Contract

- `video_final_review` 增加分辨率、宽高比、帧率、codec、音频采样率和流数量门禁。
- 增加抽帧中的生成文字、展签、Logo 和水印检查。
- 最终交付需统一转码到目标 profile，再进行时间轴组装。

## 4. 总结

本次实验支持两个有明确边界的结论：

1. 在这一个课题、一次生成、一次同模型盲评的文本设计稿实验中，传播任务、累计叙事、PageSpec 和本地数学层获得了更高的可执行性与可读性评分；这不证明真实 PPTX 或跨课题总体效果已经提高。
2. 对视觉与视频 Provider，结构化构图/镜头描述能改善画面世界和动势，但负面约束不会自动可靠执行，仍必须依赖渲染后审查和定向返工。

因此最终产品不能只优化 Prompt；必须同时优化 Contract、资产绑定、真实渲染审查、Provider failover 和局部返工路由。
