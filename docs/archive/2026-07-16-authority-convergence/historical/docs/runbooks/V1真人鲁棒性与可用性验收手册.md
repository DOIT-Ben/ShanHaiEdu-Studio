# V1 真人鲁棒性与可用性验收手册

更新时间：2026-07-13

状态：`场景库与代码预审版`

代码审计基线：`main@0ead3c2`

## 1. 目标

本手册用于验证 ShanHaiEdu 是否真能作为一个可打断、可改道、可恢复、可返修的教师助手工作，而不是只在标准路径下可用的线性流水线。

验收分为两层：

1. 真人场景验收：教师只通过产品界面和自然语言操作，产品 Main Agent 自主规划、调用 Tool、触发 Critic、等待 HumanGate、执行 Replan 和保存产物。
2. 代码证据审查：每个场景必须找到对应源码、自动化测试或真实浏览器证据；只有需求文档或 Prompt 约束，不算已经实现。

外部 Codex 只记录现象、审查最终交付包和定位责任层，不替 Main Agent 选 PPT 样张、视频创意、课程锚点或返修范围。

## 2. 状态定义

| 状态 | 含义 |
|---|---|
| `A 已实现` | 有服务端实现和针对性自动化测试，仍需抽样做真人验收 |
| `B 待真人验证` | 主要合同和代码存在，但尚无当前 V1 的真人完整证据 |
| `C 部分实现` | 只覆盖部分时点、措辞或状态，存在明确缺口 |
| `D 未实现` | 当前代码没有可靠执行路径，不能对用户承诺 |

## 4. 代码预审结论

### 4.1 已有可靠基础

- 自然语言待确认改道、修订、暂停、恢复、取消和多分支消歧已经进入 `ConversationControlResolver`，并有 ConversationTurnService 持久化测试。
- 旧 actionId 在顺序执行场景下会失效；暂停不提升 IntentEpoch，取消和实质修改会提升 IntentEpoch。
- PPT 已有逐页设计、样张 Critic、教师批准、全量审查和 `pageIds` 局部返修合同。
- 视频已有课程锚点六硬门、成片九项门、`shotIds` 单镜头生成、Provider taskId 恢复和最终包证据门。
- ConversationTurnJob 采用项目 FIFO，GenerationJob 具备幂等键、inputHash、taskId 恢复和 `submission_unknown` 停止重提。
- 两个不同项目在单 Node / 单 Prisma V1 拓扑下具备隔离与受控并发测试。

### 4.2 P1：上游回改后，stale 产物仍可能被当作可信输入

`repository.ts` 只把直接下游 WorkflowNode 标记为 stale，并保留旧 Artifact。现有测试还明确断言旧下游 Artifact 保持 `isApproved=true`，且更深层下游暂时继续是 approved。

与此同时：

- `capability-availability.ts` 只按 Artifact 的 `status=approved` 与 `isApproved=true` 判断前置是否成立。
- `agent-world-state.ts` 也只按 Artifact 状态和批准标记加入 `trustedInputs`，没有排除其 WorkflowNode 已 stale 的情况。
- `getApprovedInputs()` 读取已批准 Artifact，没有把对应 WorkflowNode 的 stale 状态作为硬门。

因此 R13、R14、R15、R20、R21 不能判为可靠完成。需要把“当前节点版本/批准指针/节点非 stale/IntentEpoch/上游 digest”统一纳入可信输入选择，并补传递性失效或基于依赖图的影响计算。

主要证据：

- `src/server/workbench/repository.ts:257-283`
- `src/server/workbench/__tests__/stage4-stale-propagation.test.ts:81-116`
- `src/server/workbench/__tests__/stage4-stale-propagation.test.ts:119-163`
- `src/server/capabilities/capability-availability.ts:95-101`
- `src/server/conversation/agent-world-state.ts:132-155`

### 4.3 P1：M68 完整动作状态机尚未落地

正式需求要求 `PendingAction + ActionOffer + ConversationControlDecision + ActionExecution` 使用同一服务端事实源，并通过 CAS、唯一活动动作和执行 checkpoint 保证跨标签页、崩溃恢复与真实 Provider 一次执行。

当前代码只有 message metadata 中的 `pendingDeliveryPlan`：

- actionId 在创建 assistant message 后再次更新 metadata 才补入，不是一次原子事务。
- supersede、IntentEpoch 更新、教师消息影响记录和新计划创建由多次独立写组成。
- `HumanGate` 主要验证 actionId 字符串相等，没有 PendingAction 版本 CAS。
- Prisma 中没有 `PendingAction` 或 `ActionExecution` 模型，也没有统一执行协调器。

因此 R29、R30 只能判部分实现，真实 Provider 运行中的取消和对账场景 R27 当前不能承诺。

主要证据：

- `src/server/conversation/conversation-turn-service.ts:263-318`
- `src/server/conversation/conversation-turn-service.ts:1506-1561`
- `src/server/guards/human-gate.ts:1-10`
- `docs/stages/local-real-mvp-m68-conversation-control-plan.md`

### 4.4 P1：最终包 Runtime 来源门仍是可写标签

V1-9G 会检查四类语义 Artifact 的 `generationMode/providerStatus/runtimeKind`，但通用 Artifact API 允许客户端提交整个 `structuredContent`，Artifact 也没有绑定权威 AgentRun。

因此 R32 目前只能证明“不正确标签会被拒绝”，还不能证明“正确标签一定来自真实 Runtime”。应改为服务端不可写来源记录，绑定 artifactId、agentRunId、inputHash 和 outputDigest，并禁止普通 Artifact 写入口修改来源证据。

主要证据：

- `src/server/package/final-package-input-contract.ts:136-144`
- `src/app/api/workbench/projects/[projectId]/artifacts/route.ts:47-62`
- `prisma/schema.prisma:172-207`

### 4.5 P2：运行中的长任务不能被自然语言即时打断

ConversationTurnJob 在同一项目内按 FIFO 串行执行；新消息只有在当前 running TurnJob 结束后才能开始。代码中没有 GenerationJob/provider cancel 状态或取消调用。

所以当前“暂停、取消、改道”主要只覆盖 pending/paused 计划，不覆盖已经开始的真实长任务。R09、R10、R27 必须先补任务级 cancel/reconciliation 合同，再做真人验收。

主要证据：

- `src/server/conversation/conversation-turn-queue.ts:68-117`
- `src/server/workbench/repository.ts:634-727`
- `src/server/conversation/conversation-control-resolver.ts:45-149`

### 4.6 P2：完整视频直达缺少稳定 fallback 路径

`CapabilityPlanner` 只专门识别 PPT、教案、需求、完整材料包和 PPT 单页返修，没有独立的完整视频规划分支。`ConversationControlResolver` 对视频直达使用较窄的“生成视频/视频片段”匹配，未覆盖常见的“我要做一个视频”“帮我做导入视频”。

OpenAI Main Agent 可能通过模型语义理解成功，但模型失败后的 deterministic 路径和针对性回归测试不足，因此 R03 不能判稳定支持。

主要证据：

- `src/server/capabilities/capability-planner.ts:38-100`
- `src/server/conversation/conversation-control-resolver.ts:314-327`
- `tests/conversation-control-resolver.test.ts:14-68`

### 4.7 P2：字幕和音轨只能定位，尚无专用局部执行能力

视频 Critic 已能把 finding 定位到 track、timeline、shot 或 frame range，但 V1-7 收尾文档明确记录，当前业务 Tool 主要完成镜头级生成，专用后期执行能力不能视为已实现。因此 R23 只能做识别与规划验收，不能宣称已经完成真实局部返修。

## 5. 真人验收执行轮次

建议不要一次跑完 36 个场景，也不要每个场景都烧真实媒体。

| 轮次 | 场景 | 验收方式 |
|---|---|---|
| 第一轮：基础对话 | R02-R08、R11-R12、R24、R26 | 确定性夹具、真实浏览器、不得调用媒体 Provider |
| 第二轮：回改与影响 | R13-R23 | 构造已有 Artifact 的项目，验证 stale、版本、digest 和局部返修范围 |
| 第三轮：中断与并发 | R09-R10、R25、R27-R31 | 故障注入、双标签页、双账号、队列和 lease 证据 |
| 第四轮：质量负例 | R32-R35 | 伪造页数、短视频、坏锚点、版本错配，验证 fail-closed |
| 第五轮：真实 E2E | R01、R22、R34、R36 | 只执行一次真实完整交付，由真人教师与外部黑盒验收共同收尾 |

## 7. V1 鲁棒性最低通过标准

在邀请制 V1 正式开放前，至少满足：

1. R01 完整 E2E 由产品 Main Agent 独立完成，外部编排为 0。
2. R02、R03、R07、R08、R12、R13、R17、R20、R24-R31 全部有真人或故障注入证据。
3. P1 的 stale 可信输入、统一动作状态机和 Runtime 来源证据缺口关闭。
4. 运行中无法取消时，UI 必须明确告诉教师“已提交，正在安全收敛”，不能假装已经停止；正式支持取消前不得对外承诺即时取消。
5. 所有确定性/降级草稿、失败 Critic、过期 action、旧版本 Artifact 和错版本最终包都稳定 fail-closed。
6. 真人发现的问题必须能通过反馈入口绑定项目、页面、阶段和产物版本提交。
