# M52 半自动对话触发门与选项式澄清阶段规划

日期：2026-07-08

## 目标

修复用户截图指出的对话产品逻辑：普通聊天由主智能体自然回复，不展示需求规划卡或产物卡；只有用户明确提出备课业务需求后，才进入需求澄清和后续生成。进入业务链路后，前端优先给出可点击选项和推荐项，让用户通过选择缩小范围，再确认发送。

## 范围

- 后端对话判断：继续保留 `ConversationOrchestrator` 的 chat / clarify / start_requirement 分层，但收紧确定性触发规则，避免泛化词误触发。
- 前端对话卡：`ChatTranscript` 只有在真实 artifact 存在且 assistant 明确说明已生成时，才展示 inline 产物卡。
- 选项式回复：在澄清/普通问候类 assistant 回复下展示低噪声 quick reply chips，点击后填入输入框等待用户确认发送。
- 左侧栏降噪：公开课栏目可折叠，移除或弱化无实际价值的假功能入口；保留新建、搜索、项目选择这些真实可用入口。

## 成功标准

- 输入“你好”后只出现普通 assistant 回复和选项，不出现“生成内容已进入产物链”。
- 输入明确备课需求后，才出现需求规格/产物链。
- 选项 chip 包含一个推荐项，点击后只填充输入框，不自动生成。
- 左侧“公开课备课”栏目可折叠；回收站等未接能力不占主要视觉。
- 浏览器验收覆盖普通聊天、选项点击、业务需求生成、侧栏折叠。

## 风险与回退

- 本阶段不引入新的 agent 编排表或多智能体后台，只在现有对话与前端承接层实现半自动入口。
- 选项内容先以前端规则生成，后续可以由真实需求智能体返回结构化选项替换。
- 如果真实模型返回“已生成”但没有 artifact，前端仍不展示产物卡，以后端保存结果为准。
