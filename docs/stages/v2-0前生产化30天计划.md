# ShanHaiEdu V2.0前生产化30天计划

更新时间：2026-07-13

状态：`planned / awaiting implementation approval`

## 1. 总目标

30个自然日内关闭`RQ-035`，达到50名不同用户在线、5名真实教师同时使用的最低生产容量。当前V1真实E2E、域名切流和教师签收仍是第一个恢复点；生产化开发不得把未关闭的V1门禁伪装成已通过。

## 2. 阶段顺序

| 时间 | 阶段 | 核心交付 | 阶段门 |
|---|---|---|---|
| 第1-3天 | P0 基线冻结 | 关闭或明确V1剩余门；冻结容量口径、数据清单、Provider额度和回退点 | 当前数据有可验证备份；目标与范围签字 |
| 第4-9天 | P1 数据底座 | PostgreSQL、正式迁移历史、SQLite迁移工具与演练 | 表数量、记录数、关联和digest一致；新旧切换可回退 |
| 第10-15天 | P2 执行底座 | Redis/BullMQ、独立Worker、三类工作池、重试/取消/死信/补偿扫描 | 杀死Web或Worker不丢任务；重复提交不重复计费 |
| 第16-20天 | P3 存储与成本 | 对象存储、积分账本、Provider配额与熔断、运营任务入口 | 文件权限与digest通过；预留/结算/释放对账为0差异 |
| 第21-24天 | P4 安全与观测 | 分布式限流、链路日志、错误监控、容量与成本告警、安全复核 | 故障可定位；告警演练到达；无高危未处理项 |
| 第25-27天 | P5 容量与恢复 | 50会话负载、5并发任务、故障注入、备份恢复和回滚演练 | 全部硬指标通过，P0为0 |
| 第28-30天 | P6 分批放量 | 内部账号、10人、30人、50人在线窗口逐级开放 | 每档观察后再扩大；5名真实教师并发验收通过 |

## 3. 依赖关系

```text
V1基线与备份
  -> PostgreSQL与迁移版本
  -> 可靠任务记账
  -> Redis/BullMQ与独立Worker
  -> 对象存储、积分和Provider治理
  -> 监控、安全与故障演练
  -> 50在线/5并发
  -> V2.0发布候选
```

数据库迁移必须先于多应用实例和多Worker；监控必须先于公开放量；真实放量必须后于自动化容量与恢复测试。

## 4. 每阶段固定完成标准

- 有批准的阶段plan和test-plan，新增需求先回写需求总账。
- 有向后兼容或明确停机窗口，有备份和回退方式。
- 新任务具备幂等、权限、超时、重试上限、成本上限和可观测性。
- 专项测试、全量测试、TypeScript、生产构建和数据库迁移演练通过。
- 涉及生产环境时保存脱敏的健康、负载、告警、备份恢复和回滚证据。
- Closeout必须区分工程通过、生产通过和真实用户签收，不混用结论。

## 5. 首发资源基线

| 资源 | 建议起步 | 扩容依据 |
|---|---|---|
| Web/API | 4核8GB，1个实例起步 | API p95、CPU、内存和错误率 |
| Worker | 8核16GB，独立进程或主机 | 队列等待、FFmpeg/LibreOffice占用 |
| PostgreSQL | 云数据库2核4GB起步 | 连接数、慢查询、CPU和存储 |
| Redis | 云Redis 1GB起步 | 队列长度、内存和命令延迟 |
| 对象存储 | 按量 | 文件总量、下载流量和生命周期 |

正式购买前必须用P5负载测试校准。预算受限时Web与Worker可暂时同机，但数据库、Redis和对象存储仍保持独立；同机方案视为单点，不冒充高可用。

## 6. 放量与停止条件

放量顺序固定为内部账号、10人、30人、50人在线窗口。以下任一出现立即停止扩大并保留现场：跨用户数据、重复Provider付费、重复积分扣减、数据库损坏、队列停止消费、用户错误率达到2%、API p95超过1600ms、RTO/RPO失败或新增高危安全问题。

## 7. 延期项

微服务、Kubernetes、多地域、自动伸缩、复杂组织租户、SSO和V2.0后的持续多轮审查不进入本30天主线。真实数据证明现有边界不足时再独立立项。
