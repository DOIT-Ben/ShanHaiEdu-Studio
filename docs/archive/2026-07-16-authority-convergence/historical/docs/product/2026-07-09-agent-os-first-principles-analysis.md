# ShanHaiEdu Agent OS 第一性原理问题分析

日期：2026-07-09

状态：新主线口径草案，待确认后替换或修订 `docs\product\current-requirements-baseline.md`

## 1. 结论

当前 ShanHaiEdu 的核心问题不是单个节点、单条提示词或某个 PPTX 接口失败，而是底层工作方式仍偏向“固定流程 + 模型填空”，尚未形成真正的“模型主控 + 结构化记忆 + 动态工具 + 渐进门禁”的 Agent OS。

后续主线应从以下原则出发：

1. 先释放大模型理解、规划、改写、追问、重试和换路能力。
2. 再基于真实运行失败，逐个节点建立必要约束。
3. 门禁只在真实交付、安全和不可造假边界上保持强约束。
4. 节点状态、工具状态和用户确认语义必须结构化进入模型上下文。
5. 不再把模型当成“内容填空器”，而是让模型成为可观察工具结果、可调整计划的主控 Agent。

## 2. 第一性原理

ShanHaiEdu 不是传统固定流程软件。传统软件可以写死：用户点击 A，系统执行 A，输出固定结果。

ShanHaiEdu 的真实工作流是：

```text
教师自然表达
-> 模型理解目标
-> 模型规划下一步
-> 系统调用工具
-> 产物改变项目状态
-> 教师继续追问、确认、修改或回退
-> 模型基于最新状态继续推进
```

因此，节点不是稳定的静态开关。每个节点会在草稿、待确认、隐式确认、显式确认、失败、可重试、可回退、可作为下游输入、过期等状态之间变化。

如果一开始就把所有节点都锁死，会导致：

- 模型知道有产物，但执行层因为状态字段不匹配而拒绝。
- 用户点击“继续下一步”，模型认为是在确认，后端却没有同步为可用输入。
- 工具调用失败后系统只报错，模型没有机会重新规划。
- 为阻止跑偏加入关键词硬拦截，逐渐把智能体退化成 if-else 流程。

## 3. 已暴露的底层问题

### 3.1 上下文记忆不足

当前主控模型看到的上下文过粗，通常只是最近若干条消息、项目基础字段、可用 artifact kind、能力列表和 pending plan。

不足之处：

- 最近消息不等于项目记忆。
- artifact kind 不等于 artifact 状态。
- 没有稳定表达“当前做到哪里、刚完成什么、失败在哪、下一步有哪些选项”。
- 没有把工具健康度、最近失败、fallback 选项喂给模型。
- 用户的“继续下一步”没有被解释成对上一轮产物的确认或带风险继续。

### 3.2 对话记忆、产物记忆和工具状态割裂

系统已有消息、产物、节点、生成任务、队列和工具注册表，但没有统一编译成模型可理解的 `AgentWorldState`。

结果是：

- 模型看到的世界与执行层判断的世界不一致。
- 模型说“可以继续”，执行层说“缺前置输入”。
- 用户感觉系统忘记上下文。

### 3.3 节点门禁过早、过硬

部分门禁本应用于防止真实交付造假，却被提前用来限制理解、规划和推进。

典型问题：

- 必须 `approved` 才能推进下一步，但用户点击“继续”没有自然映射为隐式确认。
- 小学范围用关键词硬拦截，而不是让模型基于产品范围自然纠偏。
- 节点顺序过度固定，模型难以跳步、回退、补齐或改路。

### 3.4 工具能力不稳定且缺少 fallback

当前工具注册更像静态能力表，模型看到“可以调用 PPTX”，但执行时可能出现：

- 外部服务配置不可用。
- 请求失败或超时。
- 下载链接失败。
- PPTX 结构无效。
- slideCount 不匹配。
- 没有本地替代生成路径。

工具失败后，系统直接给教师返回失败文案，模型没有读取工具 observation 并重新规划。

### 3.5 错误诊断信息分层不足

教师界面不能暴露工程细节是正确的，但内部必须记录足够清晰的失败类型。

当前问题是外部 PPTX 失败被折叠成笼统文案，难以区分：

- 配置缺失。
- 请求失败。
- 下载失败。
- 文件无效。
- 页数不匹配。
- 设计稿未通过质量校验。

## 4. “先解放模型，再逐步约束”的原则

后续主线应把约束分成三类。

### 4.1 需要先放开的部分

- 用户表达方式。
- 模型理解和追问方式。
- 节点推进顺序。
- 是否可跳步或回退。
- `needs_review` 是否能在用户点击继续后作为下游输入。
- 失败后是否重试、换工具、降级或追问。
- 非默认范围需求是否可继续作为低保障模式规划。

### 4.2 必须保留的硬门禁

- 不得把 mock、placeholder、deterministic draft 冒充真实交付。
- 不得把失败任务显示为成功。
- 不得保存无效 PPTX、伪图片、伪视频或不合格最终包。
- PPTX 必须真实校验 zip 结构和 slideCount。
- 不得泄露密钥、token、本地绝对路径、provider 内部字段或调试信息。

### 4.3 应改成软约束的部分

- 产品默认小学范围：应默认引导小学，不用关键词硬锁；用户明确越界时，提示“超出默认优化范围，可继续但质量未保证”。
- artifact 审核状态：允许“显式确认”和“点击继续的隐式确认”两种确认路径。
- 节点依赖：允许模型说明风险并继续尝试，失败后再回退或补齐。
- 工具不可用：先给模型 observation，让模型决定重试、fallback 或询问用户。

## 5. 目标架构：Agent OS

建议引入 ShanHaiEdu Agent OS 概念，至少包含：

1. `AgentWorldState`：每轮模型调用前的统一世界状态。
2. `ContextCompiler`：把消息、产物、节点、工具、任务和失败记录编译为模型上下文。
3. `ToolObservationLoop`：工具执行结果重新回到模型，而不是直接终止任务。
4. `DynamicCapabilityRuntime`：工具不只是静态注册表，而有健康度、输入就绪、失败类型、fallback。
5. `NodeStateModel`：节点表达可用性、版本、确认方式、失败和回退目标。
6. `FallbackPolicy`：关键交付工具失败后有可解释降级路径。

## 6. 推荐后续主线

下一阶段不应继续补表层文案，而应推进 `M61 Agent OS：上下文记忆、动态节点和工具 fallback 底座`。

成功标准：

- 模型每轮都知道当前目标、已完成节点、最新产物状态、失败原因和下一步选项。
- 工具失败后模型能拿到 observation 并选择重试、换路、降级或询问。
- 用户可以回退到任意关键节点并保留版本。
- 真实交付门禁仍然强，但不会卡死模型理解和规划。
