# ShanHaiEdu V2.0前生产化与容量要求

更新时间：2026-07-13

状态：`accepted / required before V2.0`

## 1. 目标

在一个月内，把当前可运行的单实例产品升级为可正式开放内测的生产系统。V2.0发布候选形成前，系统至少能够承载50名不同邀请用户在线，其中5名不同教师同时推进各自任务，并在失败、重启、Provider限流和版本升级时保持数据一致、任务可恢复和费用可追踪。

该目标是V2.0前硬门，不是后续优化建议。未通过本文容量、数据、安全和恢复门禁，不得宣称具备大批量开放条件。

## 2. 验收口径

### 2.1 50名用户在线

- 50个不同密码认证账号同时保持有效会话至少30分钟。
- 会话期间可以读取项目、消息、计划、任务状态和产物，不出现异常登出或跨账号数据。
- 工程预验收使用50个独立认证会话做自动化容量测试；正式放量阶段还要有50名真实邀请用户进入内测池并形成真实运行数据。

### 2.2 5名用户同时使用

- 5名真实教师使用5个不同账号和不同项目，在同一受控时间窗口内对话、提交并推进长任务。
- 五条任务主线可以因Provider并发额度进入队列，但入队响应不得等待整项生成结束，教师能够看到真实状态并继续使用其他非冲突功能。
- 五条任务最终必须成功、明确失败或进入人工处理，不能永久停在`queued`或`running`。
- 不得发生项目串线、产物串线、旧IntentEpoch提交、重复Provider任务或重复积分扣减。

### 2.3 最低服务指标

| 指标 | V2.0前最低门槛 |
|---|---|
| 普通API p95 | 不高于800ms，不含外部媒体生成时间 |
| 长任务入队 p95 | 不高于2秒 |
| 用户可见非预期错误率 | 低于1% |
| 同项目有效写者 | 始终不超过1个 |
| 任务恢复 | Worker重启后5分钟内恢复领取或给出明确状态 |
| 数据恢复点 RPO | 不高于15分钟 |
| 服务恢复时间 RTO | 不高于30分钟 |
| 内测月可用性目标 | 不低于99.5% |

## 3. 必做系统

### 3.1 PostgreSQL与数据库迁移

- PostgreSQL成为账号、项目、消息、计划、任务、产物元数据、质量报告、积分和审计日志的唯一事实源。
- 建立显式迁移版本和迁移历史；禁止继续只靠启动时`CREATE TABLE IF NOT EXISTS`或逐列补字段判断版本。
- SQLite迁移必须先在副本演练，再停写、备份、迁移、核对数量与关联、切换；失败不覆盖原数据。
- 迁移采用向后兼容的扩展优先策略，代码回滚前必须确认旧程序能读取当前数据合同。

### 3.2 Redis、BullMQ与独立Worker

- Web/API只负责验证请求、持久化任务和快速返回，不在请求生命周期内执行完整Agent、PPT、视频或FFmpeg长任务。
- 至少拆分快速Agent任务、外部媒体任务和本地渲染任务三个工作池，分别配置并发、超时、重试和暂停规则。
- Redis只保存排队和短期协调状态；任务业务状态、幂等键、Provider taskId和最终结果必须回写PostgreSQL。
- 采用“先在数据库记账，再可靠入队”的模式，并设置补偿扫描，避免数据库已有任务但队列消息丢失。
- 支持指数退避、最大尝试次数、死信任务、人工重试、暂停、取消和进程重启恢复。

### 3.3 对象存储与产物一致性

- PPTX、PDF、图片、视频、反馈附件和最终包进入对象存储；应用本地磁盘只保存可清理的临时文件。
- 数据库只记录对象键、版本、大小、MIME、SHA-256、血缘和可见权限，不保存公开永久URL。
- 上传、下载、删除、最终包生成和备份恢复必须校验账号权限、对象存在性和digest。

### 3.4 Provider、积分与成本控制

- 每个Provider有独立并发上限、速率上限、超时、重试预算、熔断状态和紧急关闭开关。
- 建立追加式积分账本：任务开始前预留、成功后结算、失败或取消后释放；所有变动带幂等键和责任任务。
- 管理端可以查看任务成本、失败率、队列积压和用户额度，不在教师界面泄露Provider、模型或内部价格。
- 超出用户额度、系统预算或Provider容量时必须排队、降级建议或明确阻塞，不得无界重试和重复付费提交。

### 3.5 安全与隐私

- 继续保持邀请制、公开注册关闭、服务端owner/membership授权、CSRF和会话撤销。
- 登录、反馈、任务提交和下载限流迁移到共享限流存储，多实例规则一致。
- 复核安全响应头、HTTPS、可信代理、Cookie、依赖漏洞、容器权限、上传文件、下载URL、服务端外链请求和日志脱敏。
- LLM输出和用户上传均视为不可信；权限由服务端代码强制，不依赖Prompt。
- 建立数据导出、账号停用、数据保留和安全事件处理口径；正式公网开放前另行完成适用的合规核查。

### 3.6 可观测性与运营后台

- 所有请求和任务绑定requestId、projectId、runId、jobId和脱敏用户标识，能够贯穿API、Worker、Provider和产物提交。
- 监控登录错误率、API延迟、在线会话、队列长度、最老任务等待时间、Worker存活、Provider成功率、重复提交、积分消耗、数据库连接和对象存储错误。
- P0错误、跨用户访问、队列停止消费、Provider异常、成本突增、磁盘或存储异常必须触发告警。
- 运营后台至少提供任务检索、失败归因、受控重试、用户额度、Provider开关、反馈查看和版本追踪。

### 3.7 发布、备份与恢复

- 镜像在本地或CI构建一次并推送镜像仓库，服务器按不可变digest拉取运行，不在正式服务器临时修改源码。
- 程序、数据库、Redis和对象存储分别定义健康、就绪、备份和恢复路径。
- 发布顺序为备份、迁移、启动新版本、内部验证、分批放量；出现数据异常、安全问题或错误率越界时停止放量或回滚。
- 每月至少演练一次程序回滚和数据恢复；恢复结果必须通过账号、项目、产物、反馈和任务抽检。

## 4. 版本关系

- V1.1继续完成反馈闭环、当前计划、长任务活动流、错误现场反馈和搜索。
- V1.2继续完成每阶段独立QA与一次定点返修复验。
- 本文生产化主线与V1.1、V1.2按依赖交叉推进，但不得削弱二者既有验收。
- 上述产品需求和本文生产化门禁全部关闭后，才能形成V2.0发布候选。
- 持续多轮深度审查、付费高强度审校、多Critic循环仍按RQ-033在V2.0发布后评估。

## 5. 明确不做

- 不全面重写Main Agent或现有PPT、视频业务合同。
- 不把LangGraph、其他Agent框架迁移作为V2.0前置。
- 不拆成大量微服务，不引入Kubernetes，不建设多地域或多活架构。
- 不用加大服务器、前端禁用或无界全局锁掩盖数据与并发问题。
- 不在容量门通过前一次性向全部用户开放。

## 6. 关键假设

- 50在线指50个不同有效账号保持会话；5人同时使用指5条不同项目任务主线在同一窗口推进。
- 昂贵媒体Provider可以按额度限制实际执行槽位，但系统必须可靠接收、排队、恢复并向用户解释状态。
- 一个月目标是受控公开内测，不等同于上千人同时生成媒体的无限容量服务。
