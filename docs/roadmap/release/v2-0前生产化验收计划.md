# ShanHaiEdu V2.0前生产化验收计划

更新时间：2026-07-13

状态：`planned / release blocking`

## 1. 验收原则

本计划是V2.0发布候选的阻塞门。单元测试、单进程夹具或模拟Provider不能单独证明生产容量；必须同时具备自动化容量证据、故障恢复证据和真实教师并发证据。

## 2. 数据与迁移

- 在生产副本上完整执行SQLite到PostgreSQL迁移演练。
- 核对每张表记录数、主外键关系、owner/membership、任务状态、产物版本、digest、反馈附件和审计记录。
- 同一数据库连续应用迁移两次，第二次无重复数据和破坏性变化。
- 迁移中断后能够从原SQLite重新开始或继续，不留下对外可用的半迁移状态。
- 上线前备份可验证，恢复到全新目标后账号、项目、产物和任务抽检通过。

## 3. 队列与Worker

- API持久化任务后在2秒内返回，不等待Agent、PPT、视频或渲染完成。
- Web进程、Worker进程分别在任务执行中被终止；重启后任务恢复、重试或明确失败。
- Redis短暂不可用时任务事实仍在PostgreSQL，恢复后补偿扫描重新入队。
- 相同幂等键、快速重复点击、跨标签页和重复回调不会创建重复Provider任务或重复积分变动。
- 达到最大尝试次数后进入死信或人工处理，不无限循环。
- 同项目始终只有一个有效写者，不同项目可以并发推进。

## 4. 50在线与5人同时使用

### 4.1 自动化容量门

- 建立50个不同认证账号和50个独立会话，持续30分钟。
- 50个会话持续执行项目读取、消息读取、计划/任务状态查询和少量受控写入。
- 其中5个不同账号在60秒内分别提交一条Main Agent任务并启动不同项目的长任务链。
- 五条任务全部有明确最终状态；允许媒体执行槽位受Provider额度控制，但队列状态和恢复必须正确。
- 普通API p95不高于800ms，长任务入队p95不高于2秒，用户可见非预期错误率低于1%。

### 4.2 真实用户门

- 50名真实邀请用户进入受控内测池，并安排一次50账号在线观察窗口。
- 至少5名真实教师在同一时间段使用不同账号和项目，完成对话、任务提交、查看进度和至少一个产物操作。
- 记录每位教师的任务结果、等待时间、错误、反馈和是否需要人工救援；不得由外部Codex代替产品Main Agent完成包内编排。

## 5. 安全

- 未认证访问返回401，公开注册保持403，普通教师不能访问管理员接口。
- 账号乙读取账号甲项目、消息、任务、产物、对象和反馈均失败且不泄露存在性。
- 所有写请求验证CSRF；登录、反馈、任务提交和下载的共享限流在多实例下保持一致。
- 上传覆盖伪造MIME、超大文件、损坏文件、路径逃逸和压缩炸弹；服务端外链下载覆盖私网地址和非允许协议。
- 日志、错误、监控、导出和下载URL不包含密钥、完整会话token、内部绝对路径或跨用户内容。
- 依赖和镜像扫描无未处理的高危、严重漏洞。

## 6. 成本与Provider

- 积分预留、成功结算、失败释放、取消释放和人工调整均可对账，余额不为负且重复事件不重复入账。
- 每个Provider并发上限生效；超限任务排队，不通过无界重试放大成本。
- Provider超时、限流、返回未知状态和已受理后断网分别验证；已存在taskId时恢复只轮询，不重复提交。
- 熔断和紧急关闭开关经过演练，关闭后新任务不再进入真实付费调用。

## 7. 监控、发布与恢复

- 健康检查分别覆盖应用、数据库、Redis、Worker和对象存储；Provider状态作为诊断指标，不把暂时波动伪装成应用健康。
- 队列停止消费、最老任务等待超限、错误率突增、Provider失败率突增、积分成本异常和存储失败能够触发告警。
- 新镜像按不可变digest部署；代码回滚继续连接兼容数据时通过健康、登录、项目和产物抽检。
- 数据恢复演练达到RPO不高于15分钟、RTO不高于30分钟。

## 8. 发布判定

只有以下条件全部成立才允许形成V2.0发布候选：

1. RQ-035各系统完成并形成closeout。
2. 自动化50在线/5并发门通过。
3. 5名真实教师同时使用门通过，P0为0。
4. 跨用户泄露、重复计费和数据损坏均为0。
5. 备份恢复、代码回滚、监控告警和分批放量演练通过。
6. V1.1与V1.2既定产品门禁没有因生产化改造发生回归。

任一硬门失败时保留V1系列内测，不得通过加大服务器、放宽断言或人工改库宣布V2.0可发布。
