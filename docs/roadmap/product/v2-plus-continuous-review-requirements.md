# V2.0之后持续多轮审查需求

日期：2026-07-13

状态：deferred；V2.0发布后重新评估

关联需求：RQ-033

## 1. 记录目的

记录“完成交付后自动建立全面审查计划，由多个独立质量智能体持续发现问题、反馈给Main Agent、定点修复并多轮复验”的未来能力。该能力当前不进入V1.1或V1.2开发和发布门。

## 2. 候选能力

- 多个独立Critic从教学、视觉、叙事、技术和跨产物一致性角度审查。
- finding -> 影响分析 -> 定点RepairPlan -> 复验的多轮有界循环。
- 高风险阶段第二Reviewer、交叉复核或异议仲裁。
- 最终包跨产物全面审查和未关闭finding复核。
- 标准/深度或类似审校套餐，向教师明确提示额外耗时与积分。
- ReviewPolicySnapshot冻结审查轮次、返修轮次、Provider调用、积分和时间预算。
- 长任务活动流持续展示审查待办、当前检查、finding、修复、复验和最终收尾。

## 3. 不变边界

- 不在教师输入框伪装教师自动发送审查提示词。
- Critic只返回结构化报告，不能直接执行有副作用Tool或批准自己的产物。
- Main Agent负责Replan、影响分析和RepairPlan；HumanGate继续约束范围与预算扩张。
- 审校深度与模型生成强度相互独立，不能静默升级模型。
- 不允许无限循环；达到任何预算必须暂停。
- 上游问题只重建真实受影响的最小依赖切片，不默认重跑整包。

## 4. 启动条件

只有以下事实具备后才重新规划：

1. V2.0已经正式发布。
2. V1.2阶段QA稳定运行，并积累足够的阶段问题率、返修收益和失败数据。
3. 积分与Provider成本能够可靠计量和预测。
4. Artifact lineage、stale传播和跨产物影响分析覆盖教案、PPT、视频和最终包。
5. 多Reviewer独立性、质量收益、双用户隔离和恢复能力有实验数据证明。

## 5. 后续验证问题

- 多Reviewer是否显著提高首次可授课率，而不是制造重复finding。
- 多轮返修的边际收益在哪一轮开始明显下降。
- 哪些课程和交付类型值得建议深度审校。
- 如何在开始前给出可信的耗时、Provider调用和积分上限。
- 标准与深度模式怎样定价，且不把基础质量门变成付费功能。

在上述问题没有真实数据前，本需求只保留为路线记录，不写实施阶段、不接代码、不进入当前版本验收。
