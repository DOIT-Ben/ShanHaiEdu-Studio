## 2. V1 产品切片

### 2.1 一句话目标

让受邀小学数学教师输入一节公开课需求后，能够通过自然语言控制和局部返修，真实获得结构化教案、逐页设计且可编辑的 PPTX、课堂视觉图、完整导入视频和可校验最终材料包。

### 2.2 默认交付档

V1 冻结一个 `PublicLessonDeliveryProfile v1`：

| 角色 | V1 要求 |
|---|---|
| 教案 | 有教材依据、教学流程、师生活动、板书、评价和风险处理的结构化产物 |
| PPT | 默认走 Quality 路径；Fast Coze 只允许作为 `preview_only` |
| 课堂视觉图 | 有 assetId、用途、来源、hash 和实际嵌入关系 |
| 导入视频 | 默认走 Full Intro 逐镜头路径；Short Preview 只允许预览 |
| 最终材料包 | 只收录当前有效、真实可打开、质量门通过且版本一致的产物 |

10 页 PPT 和 3 镜头视频是 V1 合同与测试夹具，不是永久产品上限。

### 2.3 V1 必须支持的教师控制

- 用自然语言继续、暂停、取消、修改、改道和局部返修。
- 在样张计划、PPT 逐页设计、视频分镜或单镜头结果处打断。
- 修改上游叙事大纲后，旧下游产物保留历史版本但退出当前有效集。
- 质量失败时优先返修 pageId、assetId、shotId 或 track，不默认整链重跑。
- 缺少关键输入时只问最小必要问题；达到时间、费用或步骤预算时暂停。

该能力采用当前 Harness 内的有界循环：

```text
Observe -> Plan -> Guard -> Act -> Observe -> Replan
```

V1 不迁移 LangGraph，也不把 Resolver 扩成第二个 Planner。Resolver 只处理当前动作的确认、取消、修改和改道控制语义；Main Agent 根据 observation 做下一步规划。

这里后置的是“通用 Agent checkpoint”，不是 Provider 付费任务恢复。`providerTaskId + inputHash + idempotencyKey` 的最小恢复能力是 V1 必需项。

### 4.1 阶段 2 的三个 Agent Tools

V1 不把每个节点都封装成 Tool。职责按四层拆分：

| 类型 | 放什么 | V1 例子 |
|---|---|---|
| Skill | 方法、判断启发式、阶段知识切片 | `edu-ppt-production`、`edu-intro-video-production` |
| Contract / Rubric | 必须满足的结构、硬门和评分标准 | PageSpec、ShotSpec、PPT/视频质量量表 |
| Tool | 可执行且可观察的能力 | Provider、PPT composer、render、ffprobe、FFmpeg、Validator |
| Agent Tool | 需要专业推理但自身只读的角色能力 | `ppt_director.plan_or_repair`、`video_director.plan_or_repair`、`delivery_critic.review` |

Main Agent 负责路由、Replan 和教师对话；业务状态由 Artifact、Job、Report、Decision 和 Lease 持久化。Agent Tool 不自动批准产物，也不能修改 Validator 结论。

### 4.2 V1 数据流与状态推进

```text
TeacherBrief(version) + 当前有效 Artifact + Teacher IntentEpoch
  -> Main Agent 生成 ActionInvocation
  -> PlanGuard / HumanGate / DataRightsGuard
  -> 获取 ProjectExecutionLease + fencingToken
  -> Tool 或 Provider 产生 immutable staging result
  -> ContractPostValidator 生成 ValidationReport
  -> Critic 读取真实 render/media 生成 CriticReport
  -> QualityDecisionEngine 确定性聚合
  -> 提交时复核 briefVersion、inputHash、IntentEpoch、依赖版本和 fence
     -> 一致：promote 为当前 ArtifactRevision
     -> 不一致：quarantine，保留审计但不进入 WorldState
  -> FinalDeliveryGate 只读取 effectiveValidity=true 且 final_eligible 的版本
```

关键状态必须分开：

| 状态 | 含义 | 能否进入最终包 |
|---|---|---|
| `staging` | Tool/Provider 已产出，尚未通过校验和提交栅栏 | 否 |
| `preview_only` | 可供教师预览，但生产路径能力上限不足 | 否 |
| `final_candidate` | 已完成生产与审查，等待确定性决策或教师批准 | 否 |
| `final_eligible` | 硬门通过、版本当前有效、批准范围匹配 | 是 |
| `stale` | 上游或教师意图已变化，历史版本仍保留 | 否 |
| `quarantined` | 迟到、失去租约、输入不一致或身份失效的结果 | 否 |
| `partial/degraded` | 失败后可导出的中间成果或降级成果 | 否，除非后续重新走完整门禁 |

### 4.3 失败与恢复策略

| 失败场景 | V1 处理 | 禁止行为 |
|---|---|---|
| Provider 已提交但 taskId 未可靠保存 | 标记 `PROVIDER_SUBMISSION_UNKNOWN`，按 Provider 幂等键/查询接口对账，无法确认则暂停人工处理 | 自动重复付费提交 |
| 轮询中进程退出 | 读取已持久化 providerTaskId 继续 poll | 新建同输入任务 |
| 租约过期或 fence 失效 | 停止写回，结果进入 quarantine；新 worker 从当前 WorldState 重新规划 | 旧 worker 覆盖新结果 |
| 教师在生成中修改大纲 | 递增 IntentEpoch，计算影响范围，旧结果保留但不可提升 | 静默接受旧结果为当前版本 |
| Validator 失败 | 返回类型化 locator，定点返修 page/asset/shot/track | 用 Critic 高分或人工点击绕过硬门 |
| Critic 不确定或成本超预算 | 暂停并请求教师选择，或转人工语义审查 | 无限自动重试 |
| 某 Provider 不可用 | 按 ProviderProfile 选择兼容替代，或交付 `partial/degraded` 中间包 | 把降级产物冒充最终完成 |
| 文件写入成功但数据库提交失败 | 文件保持 staging/orphan 身份，恢复任务对账后再提升或清理 | 仅凭文件存在显示成功 |

### 6.2 V1 试点集

1. 低年级数概念：检验精确数量、简单叙事和投影可读性。
2. 中年级几何/测量：检验多表征、过程动画和视觉关系。
3. 高年级百分数/分数：检验教材证据、关系图、生活迁移和导入视频连续性。

三个任务均固定输入、目标页数和视频时长，对现有流程和 V1 流程做盲评；记录首次可授课率、人工修改时间、返修轮数、成本和 Provider 失败率。

## 7. 提交与并行计划

### 7.0 Skill 使用边界

本计划不绑定任何开发方法类 Skill，也不把特定开发 Skill 作为实施前置、阶段门禁或完成条件。尤其不得要求调用任何 `superpowers:*` 系列 Skill。开发过程以本项目 `AGENTS.md`、阶段 plan/test-plan、实际代码、测试和验收证据为准。

允许按任务需要使用与 ShanHaiEdu 业务逻辑和交付物直接相关的功能型 Skill，例如教材驱动 PPT 工作流、PPTX 制作与检验、图片生成与编辑、视频制作与媒体检验、浏览器流程验证，以及真实 Provider/API 调用支持。这些 Skill 只提供专项能力，不接管 Main Agent 编排，不改变 Node Contract、Guard、Quality Gate、Artifact/Job 状态和教师确认边界，也不构成对某个 Skill 实现的强制依赖。

具体是否调用某个功能型 Skill，应在对应阶段根据真实任务、当前可用能力和验收目标决定；规划文档只规定业务结果、合同和质量门，不固定开发代理的工作方法。

### 7.1 推荐提交组

1. `test: 冻结V1节点与质量验收`
2. `feat: 增加执行幂等和任务恢复底座`
3. `feat: 接入节点合同与可执行质量报告`
4. `feat: 闭环PPT质量路径`
5. `feat: 闭环视频逐镜头路径`
6. `feat: 增加最终交付有效性门禁`
7. `test: 完成真实Provider与教师任务验收`

实际提交信息应继续遵守项目 `AGENTS.md` 的中文格式和日期要求。

### 7.2 并行边界

```text
原开发线完成 Phase 0
  -> 共享合同与状态底座冻结
     -> PPT 工作树 -----\
     -> 视频工作树 ------> 单一集成人串接 -> QA/Release
     -> Validator 工作树 /
```

PPT 与视频可以并行，但必须在共享 Contract、ProductionIdentity、版本语义和 locator 冻结之后。前端以 `fffdfb3` 为候选基线，完成 M72-M78 浏览器封板前不得继续扩展新视觉需求。

### 10.1 外部前置条件
