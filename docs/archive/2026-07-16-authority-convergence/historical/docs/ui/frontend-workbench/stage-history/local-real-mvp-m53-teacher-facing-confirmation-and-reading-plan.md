# M53 教师视角备课确认与成果阅读阶段规划

日期：2026-07-08

## 目标

把 M52 的半自动对话触发门继续收口成教师可理解的备课工作流：

```text
普通聊天
-> 明确备课需求
-> 先展示备课任务确认
-> 教师确认开始
-> 生成需求规格
-> 在对话和右侧面板阅读成果正文
-> 复制、作为输入、确认使用或调整后重做
```

本阶段的核心不是增加后台能力，而是修正用户态信息架构：识别需求不等于立刻生成；生成成果不等于暴露产物链、字段名、Markdown key、上游来源等工程信息。

## 用户旅程

### 1. 普通聊天

教师只是在问候、闲聊或咨询方向时，主智能体自然回复，并给出低噪声可选回复。此时不展示需求规划卡、产物卡或后台字段。

### 2. 备课需求识别

教师明确说出“我要做某年级某课题的公开课、课件、教案、PPT 大纲或导入视频”等业务需求后，系统先回一张可读的备课任务确认：

- 我理解的课题。
- 已确认信息。
- 还缺的信息。
- 推荐先生成的成果。
- 明确下一步：确认开始、补充信息、先聊创意。

### 3. 确认后生成

只有当教师输入或选择“确认开始、开始生成、按默认生成、直接生成”等确认信号后，后端才调用运行时生成 `requirement_spec` artifact。

### 4. 成果阅读

生成后的对话内嵌卡和右侧成果面板只呈现教师关心的内容：

- 标题。
- 摘要。
- 正文节选或完整正文。
- 操作：展开查看、复制、作为输入、确认使用、调整后重做。

普通界面禁止出现：`Markdown`、`状态`、`关键字段`、`上游来源`、`...`、`产物链`、`产物预览`、`生成来源`、`页面脚本`、`提示词`，以及 schema、manifest、provider、node_id、storage、API、debug、local path 等工程词。

## 竞品参考

- Eduaide：先让教师输入目标、课题或标准，再生成可编辑草稿。
- MagicSchool：用工具入口和模板入口降低空白聊天成本。
- Diffit：先确认材料、年级、语言、标准和阅读水平，再输出课堂资源。
- Brisk：把创建、反馈和改写嵌在教师已有工作环境中。

共同启发：面向教师的主界面应先确认任务和约束，再输出可编辑、可复制、可继续加工的正文，不展示后台结构。

## 范围

- 后端消息路由增加确认门：明确备课需求但未确认时，只保存 assistant 确认消息，不保存 artifact。
- 对话内嵌成果卡改为教师成果卡：去掉产物链、字段网格、上游来源和字段 key。
- 右侧成果阅读面板改文案和信息层级：去掉产物预览、关键字段、正文预览、上游来源等标题。
- `workbench-mappers` 不再把正文放到可见 `Markdown` key，不再用 `状态` 作为 fallback label。
- 更新 M50/M52/Stage7/API 相关测试预期，保持既有能力但调整用户态表达。

## 非目标

- 不新增真实 PPTX、图片、视频 provider。
- 不重写工作台三栏布局。
- 不引入新的长期队列或多智能体编排表。
- 不处理无关桌面端启动文件改动。

## 成功标准

- 输入“我要做苏教版小学六年级的百分数这个知识点的备课课件。”时，返回备课任务确认，不生成 artifact。
- 输入“确认开始”或等价确认后，才生成需求规格 artifact。
- 普通教师可见主界面不出现 `Markdown`、`状态`、`关键字段`、`上游来源`、`...`、`产物链`、`产物预览`、`生成来源`、`页面脚本`、`提示词`。
- Markdown 正文仍按标题、列表、段落渲染，不能退化成一坨纯文本。
- M52 的 quick reply 行为保持：点击选项只填入输入框，不自动发送。

## 风险与回退

- 确认信号过严会多问一步；本阶段优先避免误生成，后续再做更细的结构化选项。
- 老测试中直接生成需求规格的断言需要升级为“先确认、再生成”。
- 如真实模型 runtime 返回内容异常，仍以服务端保存 artifact 成功作为前端展示门槛。
