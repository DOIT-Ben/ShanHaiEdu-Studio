# 第一性原理产品架构遗漏审计

## 1. 审计问题

产品承诺已经明确：ShanHaiEdu 要成为一个“百依百顺、万能、真正把公开课课件做完的小助手”。

从第一性原理看，委托成功不是某个 Agent 或文件单独成功，而是：

```text
委托成功
= 意图忠实度
× 过程可控性
× 交付完整性
× 课堂可用性
× 信任
```

任何一项接近零，其他层再完善也不能形成用户价值。

## 2. 已经覆盖较好的部分

现有设计方向没有错误，以下方面已经达到较高完整度：

- Main Agent、ToolRouter、Provider Adapter 和业务事实层的职责边界。
- Artifact、版本、血缘、stale、ValidationReport、CriticReport 和 QualityDecision。
- PPT 逐页设计、样张、资产、渲染、页级返修。
- 视频课程锚点、Beat、ShotSpec、资产、片段、时间线和镜头级返修。
- HumanGate、预算、ProjectExecutionLease、FinalDeliveryGate 的候选设计。
- LangGraph、OpenAI Agents SDK、Vercel AI SDK 的框架适配边界。

主要遗漏不是再增加一个 Agent，而是没有把“教师委托关系、真实课堂环境和运行时权威”提升为一等架构。

## 3. P0：设计冻结前必须补齐

### 3.1 版本化教师委托合同

当前 `AgentProjectContext` 只有年级、学科、课题、课时、教材版本、教师目标和期望输出；`requirement_spec` 主要要求项目概述、用户目标、教材、交付范围和质量约束。桌面 `DeliveryStrategy` 补充了受众、课堂任务、课程锚点和非目标，但仍没有统一表达：

- 哪些是教师硬要求、软偏好和临时想法。
- 哪些是系统默认、模型推断和未确认假设。
- 截止时间、预算、目标设备、自主度和保护项。
- 本次委托的教师可见完成定义。

后果：所有技术质量门都可能通过，但做出的仍不是教师要的东西。

需要新增 `TeacherBrief / DelegationContract`：

```text
briefId + version
goal + audience + lesson context
must / should / may / must_not
deliverables + acceptance criteria
deadline + budget range + environment
autonomy policy + approval scope
protected decisions + open questions + assumptions
```

所有 Plan、Run、Artifact、HumanDecision 和 FinalDeliveryGate 必须绑定 `briefVersion`。

证据：

- `main/src/server/agent-runtime/types.ts:21-29`
- `main/src/server/agent-runtime/task-guidance.ts:10-13`
- `ShanHaiEdu-交付效果工作流优化-2026-07-11/contracts/delivery-strategy.schema.json`

### 3.2 最新教师意图缺少提交栅栏

现有设计提到了 ChangeSet、ImpactResolver、stale 和迟到结果隔离，但尚未形成一个不可绕过的内容版本栅栏。当前运行读取项目状态后执行模型或 Provider，完成时可以继续保存 Artifact；项目只有生命周期版本，没有统一 `IntentEpoch / ProjectStateRevision`。

如果教师在视频生成期间修改叙事大纲，旧任务仍可能把结果写回当前项目。LangGraph Checkpoint 不会自动解决这个问题，节点重放反而可能扩大风险。

必须增加：

```text
IntentEpoch
ProjectStateRevision
RunInputSnapshot
commit-time CAS
late-result quarantine
```

任何结果 promotion 前必须证明：输入 brief、上游 Artifact、意图 epoch、租约 fencing token 和批准范围仍然有效。

证据：

- `main/src/server/conversation/conversation-turn-service.ts:140`
- `main/src/server/conversation/conversation-turn-service.ts:520`
- `main/prisma/schema.prisma:15-44`

### 3.3 Artifact stale 还不是完整有效性真相

当前代码在上游批准变更后只查找直接下游已批准节点并标记 stale；旧 Artifact 本身仍可能保持 `isApproved=true`。WorldState 和 Provider 输入选择仍可能把旧 Artifact 当作可信来源。

需要把“批准过”与“当前有效”分开：

```text
immutable ArtifactRevision
ArtifactDependencyEdge
dependency version snapshot
transitive invalidation closure
effectiveValidity query
```

证据：

- `main/src/server/workbench/repository.ts:213-235`
- `main/src/server/conversation/agent-world-state.ts:103`
- `main/src/server/conversation/conversation-turn-service.ts:730`

### 3.4 后台 Agent 恢复缺少强制执行身份

当前 `AgentRuntimeInput` 和 `ToolRouterInput` 不包含不可伪造的 actor / tenant / grant；授权函数在 actor 缺失时默认允许。HTTP 请求内可能有用户身份，但 LangGraph 或其他后台恢复通常脱离原请求上下文。

这会造成：用户已停用、成员关系已撤销或项目已转移后，旧后台 Run 仍可能继续生成和写入。

需要增加：

```text
ExecutionPrincipal
scoped ExecutionGrant
tenant/user/project scope
grant expiry and revocation revision
side-effect-time authorization recheck
```

框架 `thread_id`、Checkpoint、Trace 和 Artifact 查询都必须带租户及项目作用域，不能只使用可猜测 projectId。

证据：

- `main/src/server/agent-runtime/types.ts:38-46`
- `main/src/server/tools/tool-router.ts:13-25`
- `main/src/server/auth/authorization.ts:9-40`

### 3.5 输入侧数据、版权和提示词注入治理缺失

现有设计把隐私、版权、品牌和儿童安全放进视频 Rubric，也要求资产记录许可/来源，但这发生得太晚。教师上传的教材、学生照片、录音、校本材料或外部网页在进入模型/Provider 前，尚没有统一的：

- 数据分级和 PII 检测。
- 学生肖像、声音和未成年人同意记录。
- 教材与第三方素材的用途、传播和署名许可。
- Provider 数据保留策略匹配。
- 不可信教材/OCR/网页内容与系统指令隔离。
- 下载 URL 的 SSRF、私网、重定向和大小限制。

需要 `DataRightsGuard + UsageRightsManifest + TrustLabel`，并在 Tool 调用前执行，而不是只在最终 Critic 中评分。

证据：

- `main/src/server/agent-runtime/openai-runtime.ts:150`
- `main/src/server/image-generation/image-generation-run.ts:189`
- `main/src/server/video-generation/video-generation-run.ts:313`

### 3.6 完整公开课缺少统一课堂运行规格

现有 PPT 和视频分别有较完整的质量工艺，FinalDeliveryGate 也计划核对角色、版本、血缘和质量报告。但“分别合格”不等于“整堂公开课可用”。

仍可能出现：

- 教案、PPT、视频内容重复或顺序冲突。
- 视频提前泄露 PPT 后续要探究的答案。
- 总课堂时间超限。
- 教师不知道何时播放、何时停顿、何时切板书。
- 在 WPS、目标 PowerPoint、投影或断网环境下无法运行。

需要 `PublicLessonDeliveryProfile + ClassroomRunSpec`：逐分钟课时、页/视频/板书/互动/评价衔接、答案揭示点、教师讲稿、设备操作和离线备用；并增加 CrossArtifactCritic 与整堂课 Rehearsal Gate。

## 4. P1：产品闭环前必须补齐

### 4.1 “万能”的能力边界没有产品化

当前运行时明确限制小学一至六年级，多个核心 Prompt、PPT 和视频实现仍偏小学数学。万能不能由模型自称，需要 Capability Coverage Matrix：

```text
native
tool_assisted
import_required
manual_handoff
unsupported
```

同时按学科建立 Skill Pack 和确定性 Validator。数学的数量准确性不能直接复用于语文朗读、英语发音、科学实验安全或历史时间线。

证据：

- `main/src/server/conversation/model-main-conversation-agent.ts:172-173`
- `main/src/server/agent-runtime/openai-runtime.ts:140`
- `main/src/server/coze-ppt/coze-ppt-run.ts:388-424`

### 4.2 缺少已有材料导入和外部编辑往返

当前产品和设计以从零生成 Artifact 为主。尚未形成项目教材/PDF/PPTX/DOCX/图片/视频的统一导入、解析、信任标记和 provenance；也没有教师在 PowerPoint/WPS 手改后重新导入、组件级 diff、保护人工修改和合并冲突的协议。

需要：

```text
SourceMaterial / ImportArtifact
ExternalEditIngest
component-level diff
teacher-owned region / preservation lock
merge-or-overwrite decision
```

当前产品基线把 Studio 编辑、资源库和共享资产池放在内测后的第二档，但架构现在必须保留可往返边界，否则后续会被生成器数据模型锁死。

证据：

- `main/docs/product/current-requirements-baseline.md:118`
- `main/prisma/schema.prisma:15-44`
- `ShanHaiEdu-智能体与交付工艺架构设计-20260711/03-PPT生产工艺与质量架构.md:65`

### 4.3 教师自主度、问题数量、审批和进度没有统一体验政策

已有 RunBudget 和 HumanGate 主要是技术约束，没有定义教师可见的自主度：

- 先看方案再执行。
- 平衡模式。
- 尽量自动，只在付费或不可逆动作前询问。

如果所有节点都确认，教师会重新成为项目经理；如果全自动，教师又会失去控制。需要 `AutonomyPolicy + ExecutionOffer`，展示质量档、预计时间/成本区间、不可逆动作和审批范围，并设置问题预算、合并审批和里程碑更新。

### 4.4 Teacher Memory 只有概念，没有完整生命周期

权威架构已经定义 TeacherProfileMemory、ProjectMemory 和 SessionMemory 的边界，但当前代码没有相应模型和运行链。仍需明确显式/推断偏好、来源、作用域、置信度、有效期、同意、修改和忘记；当前指令必须永远高于记忆。

证据：

- `main/docs/architecture/2026-07-09-山海智教智能体-核心设计串联.md:148-162`
- 当前 `prisma/schema.prisma` 和 `src/` 未出现 TeacherProfileMemory / ProjectMemory 实现。

### 4.5 质量标准还没有真实教师行为真值

现有 Rubric、Schema、图片/文本实验和单条视频证据质量较高，但实验仍是单课题、小样本；真实 PPTX A/B、跨学科回归和课堂效果没有证明。上线门只要求至少一名教师，无法校准不同评审者的一致性。

需要建立多学科、多年级黄金任务集，并记录：

- 首次可授课率。
- 教师人工修改分钟数和最终 diff。
- 放弃率与返修轮数。
- 课堂播放事故。
- 教材错误和修正率。
- 完整交付时间、成本和 Provider 失败率。

证据：

- `ShanHaiEdu-智能体与交付工艺架构设计-20260711/09-独立审查报告.md:124`
- `ShanHaiEdu-交付效果工作流优化-2026-07-11/07-落地优先级与产品路线.md:73-117`
- `ShanHaiEdu-交付效果工作流优化-2026-07-11/08-验证与交付边界.md:16-20`

### 4.6 技术恢复没有完整的降级交付与人工接管

真正“做完”不等于永不失败，而是失败后仍能给教师可用成果和清楚选择。需要明确：换 Provider、降低质量档、先交可用部分、导出中间材料、人工续做、稍后自动恢复。部分交付必须保持 `partial/degraded` 身份，不能冒充最终完成。

## 5. 运行时审计新增的阻塞不变量

框架接入不会自动解决以下问题：

1. 当前设计中的 `ProjectExecutionLease` 尚未在代码形成真正的 fencing lease；对话队列只有锁字段和进程内执行集合。
2. `GenerationJob` 没有 `inputHash`、幂等键和 `providerTaskId`，Provider 提交后崩溃可能重复计费。
3. HumanGate actionId 当前缺少随机 nonce、消费、过期、actor 和 CAS 状态机。
4. 项目 Snapshot 由多次查询拼装，没有统一 revision，UI 重连可能看到互相矛盾的状态。
5. 文件与数据库不是原子提交，崩溃可能形成孤儿文件或“数据库成功但文件缺失”。
6. 当前预算主要是失败次数与上下文估算，尚未持久记录真实 Token、金额、墙钟时间、用户配额和 Provider 并发。

这些不能只留在“迁移风险”章节，必须升级为框架迁移前的数据与接口合同。

## 6. 应补充的三块架构

```text
Teacher Delegation & Trust Layer
  TeacherBrief / DelegationContract
  AutonomyPolicy / ExecutionOffer
  ChangeSet / ImpactPreview / Undo
  TeacherMemory

Runtime Authority & Effect Layer
  IntentEpoch / ProjectStateRevision / commit fence
  ExecutionPrincipal / ExecutionGrant
  Artifact effective validity
  ProjectExecutionLease / ExternalActionLedger / EventLog

Classroom Delivery & Safety Layer
  PublicLessonDeliveryProfile / ClassroomRunSpec
  ClassroomEnvironmentProfile / Offline Rescue Package
  DataRightsGuard / UsageRightsManifest / TrustLabel
  Teacher Evaluation / Rubric Calibration
```

## 7. 审查门结论

```text
Stage: products-plan-eng-review
Recommendation: no-go for direct design freeze
Gate: revise
```

现有 Agent Runtime、PPT/视频工艺、Artifact Truth 和 Quality 架构应保留。最小修正不是立即接 LangGraph，而是先冻结以下六个跨层合同：

1. `TeacherBrief / DelegationContract`
2. `TeacherChangeRequest / ChangeSet / ImpactPreview`
3. `IntentEpoch / RunInputSnapshot / commit fence`
4. `ExecutionPrincipal / ProjectExecutionLease / ExternalActionLedger`
5. `PublicLessonDeliveryProfile / ClassroomRunSpec`
6. `DataRightsGuard / UsageRightsManifest / TeacherEvaluationPlan`

完成后再进行框架 Spike，否则系统可能成为一个技术上复杂、质量门很多、但仍需要教师亲自担任项目经理的生产后台。
