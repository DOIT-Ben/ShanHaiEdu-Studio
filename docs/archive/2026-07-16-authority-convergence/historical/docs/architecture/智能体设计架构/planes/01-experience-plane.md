# Experience Plane 体验平面

## 1. 定义

体验平面是用户、管理员和运营人员直接接触智能工作台的界面集合。它负责让复杂的 Agent 工作流被业务用户理解、操作、确认和接收。

## 2. 负责什么

- 项目入口：创建、切换、恢复业务项目。
- 主对话：承接自然语言需求、追问、确认、修改。
- 进度表达：展示节点状态、失败状态、待确认动作。
- 产物阅读：展示文档、图表、文件、报告、素材等 Artifact。
- 人工确认：对高风险动作、真实外部调用、最终交付提供确认入口。
- 下载和交付：提供真实文件、版本和交付包入口。
- 管理入口：契约、记忆、权限、质量报告等后台配置入口。

## 3. 不负责什么

- 不直接调用模型或外部 Provider。
- 不自行推断节点成功或交付完成。
- 不把本地路径、provider、schema、node_id、token 等工程词暴露给普通用户。
- 不把用户点击行为直接当作高风险授权，必须绑定结构化 actionId。

## 4. 设计做法

体验平面应围绕“业务项目”组织，而不是围绕“聊天记录”组织。

```text
左侧：项目/历史/任务
中间：主对话/协作流
右侧：节点进度/产物/证据/待确认动作
底部或侧栏：下载、版本、质量状态
```

普通用户看到的是业务语言：已生成草稿、等待确认、文件正在生成、需要补充来源。管理员才看到 contract、provider、schema、audit 等工程控制面。

## 5. 参考机制

- Linear / Notion：低噪声任务状态和清晰对象导航。
- OpenCode TUI：对话、工具、文件上下文的分层展示。
- 企业 SaaS 审批流：高风险动作需要明确确认和可追溯记录。

## 6. 验收问题

- 用户能否在 10 秒内知道当前项目做到哪一步？
- 用户能否区分草稿、真实文件、已交付文件？
- 失败时是否给出业务可理解的下一步，而不是工程堆栈？
- 高风险动作是否有明确、可追踪的确认动作？
