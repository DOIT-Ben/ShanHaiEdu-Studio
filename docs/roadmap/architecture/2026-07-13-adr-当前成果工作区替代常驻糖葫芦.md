# ADR：当前成果工作区替代常驻糖葫芦

日期：2026-07-13

状态：Accepted

关联需求：RQ-015、RQ-016、RQ-028、RQ-032、RQ-036、RQ-037

## 1. 背景

ShanHaiEdu 早期使用右侧常驻产物 Rail 展示教案、PPT、图片、视频和最终交付。该设计最初解决了成果不可发现的问题，但长期呈现为节点串后，会让教师和开发者潜意识地把产品理解为固定线性流水线。

当前产品已经明确：Main Agent 支持 Observe、Plan、Guard、Act、Observe、Replan，教师可以从任意可用能力切入、改道和局部返修。需求分析、教案、PPT 大纲、逐页设计和视频脚本属于教师要阅读和修改的 Artifact，不是必须逐项执行的 Plan。

2026-07-13 对 MagicSchool Raina 保存页面的浏览器审查显示，其桌面体验将对话与当前可编辑文档并列：对话承担交流和摘要，右侧大面积可调工作区承载完整文档。该模式比常驻节点 Rail 更符合 ShanHaiEdu 的智能体定位，但其移动端裁切、重复追问和能力不足不作为复制目标。

## 2. 决策

V1.5 采用以下信息架构：

```text
对话控制台 + 当前成果工作区 + 全部成果抽屉
```

具体决策：

1. 下线桌面常驻糖葫芦式 Artifact Rail。
2. 保留 Artifact、版本、依赖、状态、权限、下载和质量证据。
3. 保留并复用 RQ-016 的成果分组抽屉，作为全部成果入口。
4. 将现有只读窄侧栏演进为可调宽度、按成果类型渲染的当前成果工作区。
5. Main Agent 通过真实 artifact 引用和呈现指令决定何时打开、聚焦或更新成果，前端不从文本猜测。
6. 当前制作 Plan 独立展示，不与 Artifact 导航合并。
7. 普通草稿和局部编辑保持低摩擦；真实费用、外部副作用、不可逆动作和正式批准继续执行门禁。

## 3. 不采用的方案

### 3.1 保留常驻 Rail，只继续美化

不采用。即使把逐节点改成五类聚合，常驻纵向入口仍持续强化“按能力串行推进”的心智模型，也占用固定宽度。

### 3.2 完全删除成果模型，只保留聊天消息

不采用。聊天消息不能可靠承担版本、依赖、质量证据、局部返修、下载和最终交付，且长文阅读与编辑体验较差。

### 3.3 所有完整成果都内嵌在对话中

不采用。会造成消息极长、重复内容、版本混淆和编辑困难，也让执行说明与交付正文混在一起。

### 3.4 V1 收尾一次完成全部改造

不采用。完整编辑器、版本冲突、局部 Patch 和 QA 定位涉及前端、Agent 合同和持久化，必须分阶段迁移并在 V1.5 总验收。

## 4. 后果与取舍

正向结果：

- 教师以“正在共同修改一份成果”理解产品，不需要理解节点链。
- Main Agent 的非线性改道和局部返修在界面上得到一致表达。
- 长文阅读、编辑和审查获得足够空间。
- 对话更简洁，只保留解释、摘要、决策和下一步。

代价与风险：

- 必须补齐 artifact/version 与 panel state 的持久化合同。
- 直接编辑和自然语言编辑需要统一版本事实源。
- 多用户编辑需要冲突检测，不能静默覆盖。
- 自动打开策略如果过度会抢夺焦点，因此必须由 Main Agent 显式指定并遵守低优先级不打扰规则。
- 移动端不能照搬桌面双栏，需要单面板切换。

## 5. 验证方式

- 产品验收以 `docs\roadmap\product\v1-5-artifact-workspace-requirements.md` 为准。
- 验证普通聊天、主要成果生成、局部修改、HumanGate、QA返修、版本冲突、刷新恢复和两用户隔离。
- 桌面至少覆盖 1366x768 与 1440x900，移动覆盖 390x844。
- 检查常驻 Rail 已移除，但全部成果、下载、确认、版本和质量状态仍可访问。
- 检查 Main Agent 呈现指令绑定真实 artifact/version，前端没有关键词猜测和伪造状态。

## 6. 回退方式

若新成果工作区在阶段验收中出现阻塞，可暂时回退到“顶部成果入口 + 原成果抽屉 + 原只读详情”，但不得恢复逐节点糖葫芦作为长期目标，也不得删除新版本数据。回退只影响呈现层，Artifact、版本、权限和质量证据保持不变。
