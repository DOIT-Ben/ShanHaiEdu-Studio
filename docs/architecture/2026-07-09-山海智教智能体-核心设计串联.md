## 4. 5 平面如何落到山海智教

| 平面 | 山海智教里的含义 | 第一优先落地点 |
|---|---|---|
| Experience Plane | 教师工作台、主对话、节点进度、产物阅读、下载确认 | 保持中间对话 + 右侧节点/产物的低噪声工作台 |
| Agent Control Plane | 主对话智能体、节点契约、上下文编排、PlanGuard、HumanGate | ConversationContextBuilder、NodeContractRegistry、PromptCompiler |
| Runtime Plane | OpenAI-compatible Runtime、Coze PPT、图片、视频、文档解析 Provider | Provider Adapter 统一封装和真实文件校验 |
| Data & Memory Plane | 项目、对话、Artifact、Evidence、ProjectMemory、SessionContextSnapshot | Conversation Log、Artifact、ProjectMemory、SessionContextSnapshot |
| Quality & Governance Plane | PPTX 校验、视频门禁、教材证据、摘要校验、隐私安全、最终包门禁 | ContractValidator、SummaryValidator、FinalDeliveryGate |

## 5. 12 系统如何落到山海智教

| 系统 | 山海智教第一阶段关注点 | 不应先做的事 |
|---|---|---|
| Workbench UX | 恢复项目、展示节点、展示真实 Artifact、确认高风险动作 | 大改营销页和炫技 UI |
| Project & Actor | 项目作为中心对象，记录确认、产物、失败和审计主体 | 复杂组织权限体系 |
| Main Conversation Agent | 模型主导理解和计划，代码负责边界和验收 | 继续堆正则 planner 当主脑 |
| Node Contract Control Plane | 先文件化关键节点契约 | 一开始做复杂可视化后台 |
| Capability Registry | 让能力目录可被模型读取和选择 | 把 capability 和 provider 混成一层 |
| Workflow Orchestration | 节点状态、失败恢复、局部重试 | 一次性重写整个 workflow 引擎 |
| Agent Runtime | 统一模型调用、结构化输出和 fallback 标记 | 在 UI 或业务服务里散落模型 SDK |
| Provider Adapter | Coze PPT、图片、视频等真实产物适配 | 伪造外部成功或只返回文本 |
| Artifact & Asset | 草稿、设计稿、真实 PPTX/图片/视频/最终包区分 | 把下载链接当成完成状态 |
| Memory | ProjectMemory、SessionContextSnapshot、MemoryWriteProposal | 静默写长期用户画像 |
| Knowledge & Evidence | 教材、课标、页码、OCR、引用置信度 | 无证据冒充教材结论 |
| Quality / Guard / Governance | PlanGuard、HumanGate、ContractValidator、SummaryValidator | 让模型自评“已完成” |

## 6. 三个横切机制必须先立住

### 6.1 对话上下文与无感压缩

来源：`docs\architecture\智能体设计架构\patterns\01-conversation-context-compaction.md`

山海智教落地点：

```text
Conversation Log 完整保留
SessionContextSnapshot 后台压缩旧历史
ContextPackage 每轮作为模型输入边界
SummaryValidator 防止摘要伪造完成状态或丢失关键教学约束
```

原因：教师回到历史项目时，必须能继续上次备课上下文；模型不能因为长对话而丢掉教材、课时、PPT 页数、视频边界和已确认事项。

### 6.2 节点契约控制

来源：`docs\architecture\智能体设计架构\patterns\02-node-contract-control.md`

山海智教落地点：

```text
ppt_design
coze_ppt
video_storyboard
video_segment_plan
final_package
```

原因：山海智教不是聊天式“随便生成”，而是多节点教育内容生产。每个节点必须知道输入、输出、禁止项、质量门禁和失败策略。

### 6.3 记忆、事实、契约边界

来源：`docs\architecture\智能体设计架构\patterns\03-memory-boundaries.md`

山海智教落地点：

```text
TeacherProfileMemory：教师长期偏好，需审批或可删除。
ProjectMemory：当前项目已确认目标、教材、风格、决策。
SessionMemory：当前会话摘要和临时状态。
NodeContract：系统规则，不被用户偏好覆盖。
Artifact/Evidence：真实产物和可信来源，不由摘要决定。
```

原因：不能把“这次用赛博风”写成长期偏好；不能让用户偏好覆盖节点门禁；不能让摘要把未完成 PPTX 写成已完成。

## 6.4 Agent Harness 受控循环

来源：`docs\architecture\2026-07-09-山海智教智能体-统一口径与工作准则.md` 的 5.9-5.12。

山海智教落地点：

```text
AgentWorldState：让模型看到可信世界状态。
CapabilityAvailability：让模型知道哪些能力当前可执行、缺什么、为什么不能执行。
PlanGuard / HumanGate：把模型计划变成可审查、可确认、可回退的动作。
ToolObservation：把工具失败、provider 不可用、质量门禁失败回写到真实状态源。
Workflow State Machine：确定流程由契约和状态机推进，不由模型每一步重新猜。
预算与熔断：按耗时、成本、重试、重复动作和连续失败控制循环。
```

原因：山海智教的 PPT、图片、视频和最终包都有真实副作用、真实成本和真实交付门禁。模型可以做判断和计划，但不能绕过契约、门禁、状态持久化和质量验证直接执行。

统一判断：

```text
不确定任务，两阶段；确定流程，工作流。
上下文是模型可见状态，不是系统真实状态源。
```

### 6.5 Tool Registry 与 ToolRouter 工具层

来源：`docs\architecture\2026-07-09-山海智教智能体-统一口径与工作准则.md` 的 5.13-5.14。

山海智教落地点：

```text
Tool Registry：把现有 capability、provider、未来 MCP 工具注册成稳定 ToolDefinition。
ToolRouter：把 GPT tool_call 或内部 toolPlan 路由到 Internal、Provider 或 MCP adapter。
Internal Capability Adapter：复用 CapabilityRunner 执行内部文本节点。
Provider Adapter：复用 Coze PPT、图片、视频等真实 provider，并统一 Artifact Truth Gate。
MCP Client Adapter：未来承接浏览器、文件、数据库、搜索、资料库等 MCP 工具，M64 只保留接口边界，不做真实 MCP 接入。
```

统一判断：

```text
OpenAI SDK 管模型，MCP 管工具，中间必须有 ToolRouter。
工具注册先于 MCP 接入。
模型只提出 tool_call 意图，不执行真实工具。
```

原因：当前 `ConversationTurnService` 已承载过多职责。继续把每个 capability、provider 和未来 MCP 分支写进主链路，会把 Agent Harness 变成业务 if/else 层。M64 必须先把工具定义、可用性、门禁、执行、结果和 observation 显式注册化、路由化，再谈 GPT 原生 tool_call 闭环和 MCP 接入。

## 7. 第一阶段建议切片

后续第一阶段不建议做完整 12 系统一口气落地，而应做一个能稳定后续所有能力的骨架切片：

```text
MVP1：上下文、契约与门禁骨架
```

包含：

```text
ConversationContextBuilder
ContextBudgetManager
SessionContextSnapshot
SessionCompactor
SummaryValidator
NodeContractRegistry
PromptCompiler
ContractValidator
PlanGuard
HumanGate actionId
```

不包含：

```text
复杂 Admin 可视化后台
完整组织权限系统
全量视频生产链路重写
所有节点一次性契约化
长期用户画像自动写入
```

## 8. MVP1 成功标准

1. 用户完整 Conversation Log 不因压缩丢失。
2. 长对话能生成 SessionContextSnapshot，并用 ContextPackage 继续运行。
3. 关键节点至少有文件化 Node Contract。
4. 模型输出计划必须经过 PlanGuard。
5. 高风险真实 Provider 调用必须绑定 HumanGate actionId。
6. Artifact 真实完成状态只能来自质量门禁。
7. 摘要不能把未完成产物写成完成，不能把临时偏好写成长期记忆。
8. 现有真实 PPTX 链路、图片/视频链路的成功状态不得被伪造。
