# V1 智能体引导式回复与阶段成就摘要需求

日期：2026-07-13

状态：accepted for V1 closeout

关联需求：RQ-028、RQ-032、RQ-036

## 1. 需求目标

V1 收尾阶段需要补齐智能体的教师可见表达。回复不仅要正确，还要让教师快速看懂：系统理解了什么、准备怎么做、已经完成什么、当前质量如何、下一步需要谁行动。

本需求借鉴 MagicSchool Raina 的两类呈现：

1. 长任务阶段结束时，用紧凑概要呈现真实成果、完成数量、质量状态和下一步，让教师获得进展感与成就感。
2. 复杂问答或材料上传后，用清晰标题、加粗重点、分隔线、列表、表格、检查项、示例和逐步引导组织完整回复，让教师感到智能体可靠、主动且容易协作。

本需求纳入 V1 收尾体验门，最迟不得晚于 V1.5。它不改变 Main Agent 的业务能力、工具权限或工作流，只规范教师可见的表达与证据投影。

版本边界：V1先关闭裸Markdown、重点未渲染和回复层级问题；V1.1不再继续扩展独立Response Presenter UI Runtime，而由`docs\product\v1-1-assistant-ui-conversation-runtime-requirements.md`统一实现MessagePart、安全Renderer、流式活动和交互组件。本文件继续作为回复内容与证据约束，不重复规定传输和Runtime。

## 2. 竞品证据与适配结论

参考入口：`https://app.magicschool.ai/raina?share=8140810f-0441-4624-98bd-130ab1263dff`

调研时分享页要求登录，未授权读取登录后的完整会话正文。因此，已证实结论来自用户提供的两张完整截图；登录页只用于确认产品与分享入口，不把无法读取的内容写成已证实事实。

### 2.1 阶段成就摘要

竞品的有效做法：

- 先用一句话说明当前目标和已经推进到的位置。
- 每个阶段用独立、可折叠的行呈现，完成状态一眼可见。
- 行内给出数量、状态和简短成果，不要求用户阅读完整过程。
- 全部完成后明确说明“已完成”，随后立即给出下一步验证动作。

ShanHaiEdu 不复制工程型信息。不得显示“探索文件、执行命令、子智能体、Task ID、Token”等内容，应转换为教师业务成果：

- 已分析教材 18 页。
- 已形成 12 页逐页设计。
- 已完成 3 张关键样张。
- 已审查 6 个镜头。
- 已返修第 3、7 页。

### 2.2 引导式完整回复

竞品的有效做法：

- 开头直接回应教师的问题，不先讲系统能力。
- 用一个明确总标题建立当前主题。
- 按“已识别信息 -> 将如何处理 -> 会产出什么 -> 教师如何审核 -> 下一步”组织内容。
- 每个阶段使用短标题和短列表，复杂对比才使用表格。
- 教师审核项使用明确检查状态，避免只写抽象形容词。
- 给出教师可以直接说的反馈示例，降低组织修改意见的成本。
- 信息分阶段出现：材料刚上传时说明已识别与下一步；形成中间结果后补充变化；完成后再给成果、验证和修改入口。

真正值得借鉴的是信息层级和协作引导，不是把每条回复都拉成长文，也不是照搬装饰符号。

## 3. 四种回复模式

| 模式 | 适用场景 | 默认长度 | 核心内容 |
|---|---|---|---|
| `brief` | 问候、简单追问、单一事实确认 | 1-3 段 | 直接回答，必要时给一个下一步 |
| `guided` | 上传材料、复杂需求澄清、工作流解释、修改方案 | 3-7 个结构块 | 已知信息、处理方式、预期产出、教师审核、下一步 |
| `milestone` | 长任务阶段完成、等待 HumanGate、局部返修后 | 紧凑摘要 + 可展开详情 | 真实成果数量、质量状态、发现、下一动作 |
| `completion` | PPT、视频或最终包收尾 | 结果摘要 + 验证 + 入口 | 已完成成果、版本、质量结论、剩余事项、查看/下载/反馈 |

模式由 Main Agent 根据当前意图、计划阶段和真实事件选择。前端不能根据字数猜测模式，也不能把所有回答强制套进同一模板。

## 4. 回复内容规则

### 4.1 开头

- 第一段必须直接回应当前教师输入，并说明系统此刻的理解或结果。
- 不使用空泛欢迎语，不重复介绍完整生产链。
- 材料上传后先确认真实读取到的材料类型、范围和可识别状态；无法读取时明确说缺口。

### 4.2 主体

- 一个回复只保留一个主标题，复杂内容最多两级标题。
- 每个列表项只表达一个动作、发现或结果。
- 加粗用于关键结论、风险、教师决策和下一步，不整段加粗。
- 分隔线只用于两个以上明显阶段之间，不作为装饰。
- 表格只用于三项以上的稳定映射或对比；流式输出时应在完整表格块到达后一次渲染。
- 允许少量语义图标或 Emoji 帮助区分“完成、进行中、待确认、风险”，但必须同时有文字；不得连续堆叠、卡通化或替代可访问标签。

### 4.3 结尾

- 必须给出一个与当前状态匹配的下一步：系统自动继续、等待教师确认、需要教师补充，或成果已可查看。
- 只有真实 `ActionOffer` 或可执行动作存在时，才能承诺“回复一句即可继续”。
- 教师可修改时，给出 2-3 个贴近当前内容的自然语言示例，不使用固定万能话术。

## 5. 阶段成就摘要规则

阶段摘要不是模型自述，而是真实执行事件的教师可见投影。

每条摘要至少包含：

1. 阶段名称。
2. 状态：已完成、进行中、等待确认、需返修、受阻。
3. 一个可核验成果或发现。
4. 真实数量或定位信息；没有可靠数据时不显示数字。
5. 下一步动作及执行方。

建议示例：

```text
PPT 逐页设计已完成

- 已完成：12 页逐页设计，均包含底图、元素、文字、排版和教学动作
- 已检查：发现第 7 页信息偏多，已拆分为两层呈现
- 当前状态：等待您选择关键样张方向
- 下一步：确认样张后，我会生成完整 PPT 并逐页检查
```

禁止示例：

```text
我已经深入思考并调用多个工具，完成了大量工作。
```

该表述无法核验，也没有给教师提供可行动信息。

## 6. 呈现层合同

建议在现有消息正文之外增加可选的结构化呈现信息：

```text
mode
title
opening
sections[]
evidenceSummary[]
reviewChecklist[]
examples[]
nextAction
expandableDetails[]
```

职责边界：

```text
Main Agent：决定意图、内容、业务动作和是否继续
Response Presenter：把已决定内容组织为教师可读结构
Evidence Binder：从计划、Tool、Artifact、Critic、HumanGate 和 QualityDecision 绑定真实事实
Safe Renderer：安全渲染标题、列表、分隔线、表格、检查项和折叠详情
```

Presenter 只约束教师可见表达，不得改变计划、补造事实、替 Main Agent 批准门禁或限制 Main Agent 可调用的能力。

## 7. 流式与渐进式呈现

- 普通文本按完整段落或完整列表项提交，避免半句话反复跳动。
- 标题、表格、检查清单等待完整结构块后渲染，避免流式 Markdown 造成布局闪烁。
- 长任务过程优先追加真实活动事件，不在同一条消息中无限重写旧内容。
- 当前阶段详细展示；已完成阶段折叠为一行摘要；教师可以展开查看证据摘要。
- 刷新或重新进入项目后，应从持久化事件恢复同样的摘要与顺序，不能只依赖前端内存。

## 8. 证据、安全与边界

- 所有完成数量、页码、镜头数、版本、耗时和质量状态必须来自真实数据源。
- 不展示模型思维链、隐藏 Prompt、原始工具参数、命令、Provider、API、MCP、Skill、路径、Token 或调试日志。
- 不把“模型认为不错”显示为质量通过；质量状态必须来自 Validator、CriticReport 或 QualityDecision。
- 不把目标页数、文件名或计划项数量当作真实产物数量。
- Markdown 必须经过安全白名单渲染；用户内容不能注入脚本、样式或伪造系统状态。
- 两个教师的活动、数量、成果和下一动作必须按项目与权限隔离。

## 9. 响应式与可访问性

- 1366x768 与 390x844 下，标题、表格、列表和折叠行不得横向溢出或遮挡输入框。
- 窄屏表格优先转换为键值列表；无法转换时放入独立可滚动区域。
- 状态不能只靠颜色区分，必须配合文字、图标和 ARIA 标签。
- 折叠详情支持键盘操作、焦点可见、展开状态可读。
- 文本层级控制在 2-3 种字号，正文保持适合连续阅读。

## 10. 验收场景

1. 教师只说“你好”时，使用 `brief`，不输出完整生产流程。
2. 教师上传教材后，使用 `guided`，准确说明已识别信息、缺口、拟处理步骤和下一步。
3. PPT 逐页设计完成后，使用 `milestone`，显示真实页数、检查结果和样张确认动作。
4. 视频课程锚点未通过时，显示可理解的 finding 和返修范围，不显示 Critic 内部字段。
5. 等待 HumanGate 时，摘要明确“等待谁确认、确认什么、确认后做什么”。
6. 局部返修后，只汇报受影响页或镜头及复验结论，不重复整套流程。
7. 最终包完成后，使用 `completion`，列出真实版本、校验结果、成果入口和未关闭事项。
8. 页面刷新和重新进入项目后，摘要顺序、数量和状态与服务端事实一致。
9. 两名教师并行使用时，任何摘要、成果数量和下一动作均不串线。
10. 无可靠数量时不显示虚构数字，无真实质量报告时不显示“已通过”。
11. 1366x768 与 390x844 下，长回复可扫描、可折叠、不遮挡输入。
12. 用户输入 Markdown 或 HTML 时，不能伪造系统标题、完成状态或执行按钮。
13. 所有状态在无颜色、键盘和屏幕阅读器场景下仍可理解。
14. 回复中不出现工程词、思维链、密钥、路径或内部工具细节。

## 11. 完成定义

本需求只有同时满足以下条件才算完成：

- 四种回复模式有明确触发规则并接入真实 Main Agent 输出路径。
- 阶段摘要绑定真实事件和产物证据，不由前端或模型虚构。
- 普通问答仍保持自然简短，复杂任务能够逐步引导。
- 桌面与窄屏完成真实浏览器验收。
- 至少覆盖教材上传、PPT阶段完成、视频返修、HumanGate和最终收尾五类真实会话。
- 不泄露思维链、工程词和跨用户数据。
